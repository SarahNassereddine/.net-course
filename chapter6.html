Chapter 6

Mapping URLs to Endpoints Using Routing

How ASP.NET Core maps incoming HTTP requests to endpoint handlers — from route templates and parameters to constraints, catch-alls, and URL generation.

1

What is Routing?

Routing is the process of mapping an incoming HTTP request to a method that will handle it. You can use routing to control the URLs you expose in your application, map multiple URLs to the same handler, and automatically extract data from a request's URL.

Routing in ASP.NET Core — who handles what
1
Minimal APIs
The handler is the endpoint handler lambda/method associated with a route registered via MapGet(), MapPost(), etc.
2
Razor Pages
The handler is a page handler method (OnGet, OnPost…) inside a Razor Page file.
3
MVC
The handler is an action method on a controller class.
📋
Why not just use file-based routing?
If you used a purely file layout-based routing system (one file = one URL), you'd face a problem with dynamic URLs like /products/42. You'd need either a separate handler per product (unfeasible) or everything in the query string (/products?id=42). Route templates solve this cleanly — /products/{id} matches any product ID while keeping URLs readable and clean.

Note: The query string is not used by the routing infrastructure to select an endpoint — it is only extracted later by model binding.
🧠 A URL like /products?id=42 uses a query string. What does the routing infrastructure use the query string for?
2

Endpoint Routing Middleware

Endpoint routing is implemented by two separate middleware components. Splitting the work into two steps is intentional — it allows other middleware (like authorization) to run between selection and execution and inspect which endpoint was chosen.

MiddlewareIts jobWhere it sits
RoutingMiddleware
UseRouting()		 Looks at the URL and HTTP method, finds the matching endpoint, and saves that decision in the request context. Does not run your handler yet. Start of the pipeline
EndpointMiddleware
UseEndpoints()		 Reads the decision that RoutingMiddleware stored and actually calls the selected handler. This is where your lambda runs and the response is sent. End of the pipeline

All the endpoints you register with app.MapGet() are stored in a shared dictionary inside WebApplication. RoutingMiddleware reads it to find a match; EndpointMiddleware reads it to execute the match.

💡
WebApplication adds both automatically
WebApplication automatically inserts RoutingMiddleware at the start and EndpointMiddleware at the end of the pipeline. You can override their position by calling UseRouting() or UseEndpoints() explicitly — but in most apps you never need to.
Request pipeline — where each middleware must live
→ Incoming request
StaticFileMiddleware Serves .css, .js, images directly — no routing needed
BEFORE routing ✓
RoutingMiddleware Selects an endpoint, stores the result in HttpContext
DECISION POINT
AuthorizationMiddleware Checks permissions — works because the endpoint is already known
AFTER routing ✓
Your Custom Middleware Can also inspect the selected endpoint if placed here
AFTER routing ✓
EndpointMiddleware Executes the selected handler — sends the response
EXECUTION
← Response goes back up
C#Registering multiple endpoints — the shared route dictionary
var builder = WebApplication.CreateBuilder(args); var app = builder.Build(); // Each MapXxx() adds a route template + handler to the dictionary app.MapGet("/products", () => "All products"); app.MapGet("/products/{id}", (int id) => $"Product {id}"); app.MapPost("/products", () => "Created product"); app.MapDelete("/products/{id}", (int id) => $"Deleted {id}"); app.Run(); // RoutingMiddleware auto-added at pipeline start // EndpointMiddleware auto-added at pipeline end

3

Route Templates

A route template is a URL pattern used to match against incoming request URLs. Templates contain literal segments (fixed text) and route parameter placeholders (variable values in curly braces {}).

Template anatomy
/ currency / {from} / convert / {to}
Literal segment — must match exactly (case-insensitive)
Route parameter — captures any URL segment as a route value
✅  /currency/USD/convert/GBP  → from=USD, to=GBP
❌  /currency/USD  (missing segments)    ❌  /fx/USD/convert/GBP  (wrong literal)
C#Route template shapes
// ① Literal only — exact match app.MapGet("/healthcheck", () => "OK"); // ② Single parameter app.MapGet("/products/{id}", (string id) => $"Product {id}"); // ③ Multiple parameters app.MapGet("/orders/{year}/{month}", (int year, int month) => $"Orders {year}/{month}"); // ④ Parameter separators don't have to be "/" // Matches: /archive/2024-03 → year=2024, month=03 app.MapGet("/archive/{year}-{month}", (int year, int month) => $"{year}-{month}"); // ⑤ Real-world example from the slides app.MapGet("/currency/{from}/convert/{to}", (string from, string to) => $"Rate: {from} → {to}");
4

Parameters & Literal Segments

Each segment is separated by / and is either a literal (must match exactly) or a route parameter (captures any value). Captured values are stored as route values and fed into model binding.

TypeSyntaxBehaviourExample match
LiteralproductsMust match exactly (case-insensitive)/products
Route parameter{id}Captures any non-empty segment/products/42 → id="42"
Optional parameter{id?}Segment may be omitted; value is null if absent/products and /products/42
Default value{page=1}Uses default when segment is absent/blog → page=1
Constrained{id:int}Must satisfy constraint or route doesn't match/products/5 ✅  /products/abc ❌
Catch-all{**rest}Captures all remaining segments including slashes/files/a/b/c → rest="a/b/c"
📋
Route values drive model binding
When a template matches, captured parameter values are stored in a route values dictionary on the request. Model binding then reads from this dictionary to populate your handler's C# parameters — the parameter name in your handler must match the placeholder name in the template.
5

Optional Parameters & Default Values

A single route template can match multiple URL shapes using optional parameters ({param?}) or default values ({param=value}).

Optional — {param?}
// Segment may be absent; value is null app.MapGet("/stock/{id?}", (int? id) => id is null ? "All stock" : $"Stock for {id}"); // /stock → id = null // /stock/123 → id = 123
Default value — {param=value}
// Segment may be absent; uses default app.MapGet("/products/{page=1}", (int page) => $"Page {page}"); // /products → page = 1 (default) // /products/3 → page = 3
⚠️
Optional and default parameters must come last in the template
Placing an optional/default parameter before a required one creates an ambiguous template the router can't resolve. /products/{id?}/details is invalid — the router can't tell whether details is the literal segment or the value of id.
🧠 Route: /blog/{year=2024}/{month=1}. Request: GET /blog. What are the values of year and month?
6

Route Constraints

Constraints narrow down which URLs a template matches by adding type or value restrictions to a parameter using the colon syntax: {id:int}. If the segment doesn't satisfy the constraint, the route doesn't match — the next registered route is tried instead.

🚫
Don't use constraints for general input validation
Constraints produce 404 Not Found when they fail — not 400 Bad Request. Using them to validate emails, business rules, etc. will confuse users. Use constraints only to disambiguate overlapping routes. Put actual validation logic inside your handler.
ConstraintAcceptsExample templateMatch / No-match
:int32-bit integer/products/{id:int}/products/5 ✅   /products/abc ❌
:long64-bit integer/data/{id:long}/data/9999999999 ✅
:guidGUID / UUID/users/{id:guid}/users/3fa85f64-... ✅
:booltrue / false/flags/{active:bool}/flags/true ✅
:datetimeDateTime value/archive/{date:datetime}/archive/2024-01-15 ✅
:alphaA–Z / a–z only/lang/{code:alpha}/lang/en ✅   /lang/en2 ❌
:min(n)Integer ≥ n/page/{n:min(1)}/page/1 ✅   /page/0 ❌
:max(n)Integer ≤ n/rating/{r:max(5)}/rating/3 ✅   /rating/6 ❌
:range(m,n)Integer between m and n/month/{m:range(1,12)}/month/6 ✅   /month/13 ❌
:minlength(n)String length ≥ n/search/{q:minlength(3)}/search/abc ✅   /search/ab ❌
:maxlength(n)String length ≤ n/slug/{s:maxlength(50)}any ≤ 50 chars ✅
:regex(expr)Matches regex/{slug:regex(^[a-z-]+$)}/hello-world ✅
C#Disambiguating overlapping routes with constraints
// Without constraints these two are ambiguous // With constraints, routing picks the right one automatically app.MapGet("/products/{id:int}", (int id) => $"Product by ID: {id}"); app.MapGet("/products/{slug:alpha}", (string slug) => $"Product by slug: {slug}"); // GET /products/42 → first route (id=42) // GET /products/shoes → second route (slug=shoes) // GET /products/42abc → no match → 404 // Multiple constraints chained with colons: app.MapGet("/calendar/{year:int:min(2000)}/{month:int:range(1,12)}", (int year, int month) => $"{year}-{month:D2}");

Catch-All Parameters

A normal route parameter matches exactly one segment — the text between two slashes. It can't contain a slash itself. So /product/{name} matches /product/apple but not /product/summer/apple.

Sometimes the number of segments is genuinely unknown. The currency converter from the slides is a clear example:

/USD/convert/GBP
/USD/convert/GBP/EUR
/USD/convert/GBP/EUR/CAD

A catch-all parameter — written with * or ** — captures everything remaining in the URL, including slashes:

C#Catch-all captures the entire URL tail as one string
app.MapGet("/USD/convert/{**others}", (string? others) => { // /USD/convert/GBP → others = "GBP" // /USD/convert/GBP/EUR → others = "GBP/EUR" // /USD/convert/GBP/EUR/CAD → others = "GBP/EUR/CAD" // /USD/convert → others = "" ← empty string, still matches var targets = others?.Split('/') ?? []; return $"Converting USD to: {string.Join(", ", targets)}"; }); // Catch-all must always be the LAST segment in the template.
{*others} — single star
// URL-decodes the captured value. // /convert/a%2Fb → others = "a/b" // (%2F decoded into an actual /) // Can create ambiguity with real // path separators — avoid.
{**others} — double star (use this)
// Keeps the raw URL value as-is. // /convert/a%2Fb → others = "a%2Fb" // (%2F stays encoded — no confusion) // What ASP.NET Core uses internally. // Prefer ** in all cases.
💡
Catch-all also matches an empty string
Unlike normal parameters, a catch-all matches even when there's nothing left after the prefix. GET /USD/convert matches the template above with others = "". Handle the empty case in your code.
🧠 Template: /files/{**path}. Request: GET /files/docs/2024/report.pdf. What is path?

Generating URLs — Why Hard-Coding is Bad

Hard-coding URL strings like "/products/42" is fragile. If you rename or restructure a route, every hard-coded string is silently wrong. ASP.NET Core's routing infrastructure can generate URLs dynamically from a route name and a set of values.

❌ Hard-coded (fragile)
// If route ever changes, // this string is silently wrong var url = $"/products/{newId}"; return Results.Redirect(url);
✅ Generated (robust)
// Framework generates the correct // URL from the named endpoint var url = linkGen.GetPathByName( "GetProduct", new { id = newId }); return Results.Redirect(url!);
How route values are placed in the generated URL
📍
Value matches a template parameter
Inserted into the URL path. Template /product/{name} + name=big-widget → /product/big-widget
🔗
Value has no matching template parameter
Appended as a query string. Template /product + name=big-widget → /product?name=big-widget
9

The LinkGenerator Service

LinkGenerator is an injectable ASP.NET Core service available anywhere in your application. You name an endpoint with .WithName() then reference that name from any handler or service to generate a URL.

⚠️
Endpoint names are case-sensitive and must be globally unique
Unlike route templates (case-insensitive), names in WithName() are case-sensitive. Duplicate endpoint names cause a runtime exception at startup. Use clear, descriptive names: "GetProductById", "CreateOrder".
C#LinkGenerator — full example
// ❶ Name your endpoint app.MapGet("/product/{name}", (string name) => $"Product: {name}") .WithName("GetProduct"); // ❷ Inject LinkGenerator anywhere and generate a path app.MapPost("/product", (LinkGenerator linkGen) => { // {name} is in template → inserted in path var path = linkGen.GetPathByName( "GetProduct", new { name = "big-widget" }); // path = "/product/big-widget" // {color} has no template parameter → becomes query string var path2 = linkGen.GetPathByName( "GetProduct", new { name = "big-widget", color = "blue" }); // path2 = "/product/big-widget?color=blue" return Results.Ok(new { path, path2 }); }); // ❸ GetUriByName takes HttpContext → builds a full absolute URL // (extracts scheme and host from the incoming request) app.MapGet("/test", (LinkGenerator linkGen, HttpContext ctx) => { // Returns: https://yourapp.com/product/shoes var url = linkGen.GetUriByName(ctx, "GetProduct", new { name = "shoes" }); return url; });

GetPathByName vs GetUriByName

GetPathByName() — relative path
// Returns: "/product/big-widget" // No scheme, no host. // Use for: Location headers, // internal redirects, href values. linkGen.GetPathByName("GetProduct", new { name = "big-widget" }); // → "/product/big-widget"
GetUriByName() — full URL
// Returns: "https://myapp.com/product/big-widget" // Includes scheme + host. // Use for: emails, webhooks, // external redirects. linkGen.GetUriByName(httpContext, "GetProduct", new { name = "big-widget" }); // → "https://myapp.com/product/big-widget"
🧠 You call linkGen.GetPathByName("GetProduct", new { name="shoes", size=42 }) for template /product/{name}. What URL is generated?

Redirects with IResults

From inside a handler, use Results.RedirectToRoute() to redirect to another named endpoint in your app, or Results.Redirect() for any arbitrary URL (including external sites).

C#Redirect examples
// Named target endpoint app.MapGet("/new-product/{name}", (string name) => $"New: {name}") .WithName("NewProduct"); // Redirect old URL → new URL (302 Found by default) // RedirectToRoute() generates the Location header automatically app.MapGet("/old-product/{name}", (string name) => Results.RedirectToRoute("NewProduct", new { name })); // Redirect to an external URL app.MapGet("/go-external", () => Results.Redirect("https://example.com")); // Permanent redirect (301 Moved Permanently) app.MapGet("/legacy", () => Results.Redirect("/new-home", permanent: true));
MethodStatusUse case
Results.RedirectToRoute(name)302 FoundRedirect to a named endpoint in your own app
Results.Redirect(url)302 FoundRedirect to any arbitrary URL including external sites
Results.Redirect(url, permanent: true)301 MovedPermanent redirect — tells browsers and search engines to update

Controlling Generated URLs with RouteOptions

RouteOptions sets global defaults for all URL generation in your app — lowercase paths, trailing slashes, and query string casing. These are applied automatically whenever LinkGenerator or RedirectToRoute generates a URL.

C#Configuring RouteOptions in Program.cs
var builder = WebApplication.CreateBuilder(args); builder.Services.Configure<RouteOptions>(options => { options.LowercaseUrls = true; // /HealthCheck → /healthcheck options.AppendTrailingSlash = true; // /products → /products/ options.LowercaseQueryStrings = false; // query string casing unchanged }); var app = builder.Build(); app.MapGet("/HealthCheck", () => "OK").WithName("HealthCheck"); // GetPathByName("HealthCheck") now returns "/healthcheck/" // instead of "/HealthCheck"
📋
Per-call overrides with LinkOptions
You can override RouteOptions defaults for a single URL generation call by passing an optional LinkOptions parameter to LinkGenerator methods. Useful for legacy APIs that can't adopt lowercase globally but need it in specific places.
