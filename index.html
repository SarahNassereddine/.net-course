Chapter 1

Getting Started with ASP.NET Core

An introduction to the ASP.NET Core framework — what it is, what you can build with it, and how HTTP requests flow through it.

1

What is ASP.NET Core?

ASP.NET Core is a modern, cross-platform, open-source web framework for building internet-connected applications. Rather than writing low-level networking code yourself, ASP.NET Core gives you structure, helper functions, and a complete framework so you can focus on business logic.

🏗️ Framework Benefits
• Offers structure, helper functions, and application scaffolding
• Reduces the need for manual coding, enhancing efficiency
• Manages the underlying networking structure for you
• Your handlers are called by the framework
🧠 Business Logic
• The core of your application
• Independent of ASP.NET Core — no framework dependencies
• Interacts with databases, remote APIs, and other services
• You write this; the framework calls it
🔄
Application Flow — Three Layers
A typical ASP.NET Core application has three layers:
(1) ASP.NET Core framework code — handles requests from the client, dealing with complex networking.
(2) Handlers (Razor Pages, Web API controllers) — you write these using framework primitives.
(3) Domain/business logic — plain C# classes with no ASP.NET Core dependencies.

What Types of Applications Can You Build?

ASP.NET Core supports multiple application paradigms — you can even combine several within a single application. All paradigms share the same foundational building blocks: configuration, logging, and dependency injection.

API TypeDescriptionUse When
Minimal APIs Simple HTTP APIs — low ceremony, no controllers Mobile backends, SPAs, microservices
Web APIs More structured, feature-rich, controller-based Complex enterprise APIs needing full conventions
gRPC APIs Efficient binary protocol for server-to-server High-performance internal microservice communication
TypeDescription
Razor Pages Page-based server-rendered applications. Each page has its own handler. Simplest model for form-driven apps.
MVC Controllers Similar to Razor Pages but uses the Model-View-Controller pattern. More flexible routing, separates concerns more explicitly.
Blazor WebAssembly
Runs in the browser using the WebAssembly standard. You write C# instead of JavaScript. Comparable to Angular, React, and Vue — a true SPA framework.
Blazor Server
Runs on the server, maintains a stateful connection (SignalR) to the browser. Combines ease of server rendering with client-side interactivity.
🧭
Factors for Choosing a Paradigm
API Requirements — What kind of data contract do clients expect?
Integration — How does this fit with existing applications?
Customer Environment — Browser support and OS constraints?
Scalability — What are the uptime and performance requirements?
🔗
Shared Building Blocks
All ASP.NET Core paradigms leverage the same foundational elements — configuration, logging, dependency injection, and the middleware pipeline. You learn once, apply everywhere.

Choosing ASP.NET Core

Compared with other cross-platform web frameworks, ASP.NET Core has several compelling selling points:

⚡
High Performance
One of the fastest web frameworks available. Benchmarks consistently rank it among the top performers across all platforms.
🌐
Cross-Platform
Build and run on Windows, macOS, or Linux. Deploy to any cloud provider or on-premise server.
🔓
Open Source
Fully open-source on GitHub. Community contributions, transparent development, and no licensing fees.
📐
Familiar Patterns
Uses well-known design patterns like MVC, dependency injection, and middleware. C# is expressive, type-safe, and modern (VB.NET and F# also supported).

Kestrel and the Request Pipeline

ASP.NET Core uses a self-hosted web server called Kestrel to receive and process HTTP requests. Kestrel is built into your application — you don't need a separate server process.

📋
Note — How a request is processed
An ASP.NET Core application runs a self-hosted web server. The web server processes the request and passes it to the body of the application, which generates a response and returns it to the web server. The web server sends this response to the browser.

Kestrel is responsible for receiving raw requests and constructing an internal representation of the data — an HttpContext object — which can be used by the rest of the application.

Kestrel vs Reverse Proxy — Separation of Concerns
Kestrel is concerned with generating HTTP responses — it is tightly integrated with your application code.
The reverse proxy (IIS, Nginx, Apache) is concerned with handling the raw connection to the internet — TLS termination, load balancing, and security.

This separation means your ASP.NET Core code is the same whether deployed behind IIS on Windows, Nginx on Linux, or directly in a container.
