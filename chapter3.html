A Brief Overview of an ASP.NET Core Application

📋
Note — The HttpContext Object
The HttpContext constructed by the ASP.NET Core web server is used by the application as a sort of storage box for a single request. Anything that's specific to this particular request and the subsequent response can be associated with it and stored in it, such as: properties of the request, request-specific services, data that's been loaded, or errors that have occurred.

The web server fills the initial HttpContext with details of the original HTTP request and other configuration details and then passes it on to the rest of the application.

Understanding the Project Layout

When you create a new ASP.NET Core web application in Visual Studio (or with dotnet new), you get a specific folder and file structure. Here's what each part means:

📋
Note — Solution Explorer vs Disk
Solution Explorer also displays the Connected Services and Dependencies nodes, which list NuGet and other dependencies, though the folders themselves don't exist on disk.

Visual Studio doesn't show the .csproj file explicitly, but you can edit it if you double-click the project name in Solution Explorer or right-click and choose Properties.

Your project folder contains a subfolder called Properties, which contains a single file: launchSettings.json. This file controls how Visual Studio will run and debug the application.

Program.cs — Defining Your Application

Every ASP.NET Core application begins as a .NET Console application. In .NET 6+, the startup code is written directly in Program.cs using top-level statements — no class, no static void Main().

💡
Top-Level Statements
In .NET 6+, you can write the body of the Main method directly in a file. The compiler generates the Main method for you. This reduces boilerplate and makes Program.cs clean and readable.
C#Program.cs — minimal ASP.NET Core app
// ❶ Create the builder — configure app before building var builder = WebApplication.CreateBuilder(args); // WebApplicationBuilder configures these by default: // Configuration — loads appsettings.json + environment variables // Logging — extensible logging system for debugging // Services — DI container (add your services here) // Hosting — Kestrel web server configured here // ❷ Register services with the DI container builder.Services.AddRazorPages(); // example service // ❸ Build — finalize configuration and create the WebApplication var app = builder.Build(); // ❹ Configure the middleware pipeline app.UseHttpsRedirection(); app.UseStaticFiles(); app.UseRouting(); app.UseAuthorization(); // ❺ Map endpoints (define responses to specific URLs) app.MapRazorPages(); app.MapGet("/hello", () => "Hello, World!"); // ❻ Run the app — starts Kestrel and begins listening app.Run();
📋
Note — The Builder Pattern
You'll find this pattern of using a builder object to configure a complex object repeated throughout the ASP.NET Core framework. This technique is useful for allowing users to configure an object, delaying its creation until all configuration has finished.

In this simple application we don't make any changes to WebApplicationBuilder before calling Build(), but it configures a lot of things by default.

After configuring the builder you call Build() to create a WebApplication instance. The WebApplication instance is where you define how your application handles and responds to requests, using two building blocks: Middleware and Endpoints.

Adding Functionality to Your Application

The example below shows two added features: HTTP request logging (only in Development), and a /person endpoint that returns JSON.

C#Program.cs — adding logging + a JSON endpoint
var builder = WebApplication.CreateBuilder(args); // ❶ Customize features by registering services builder.Services.AddHttpLogging(o => { }); var app = builder.Build(); // ❷ Ensures logs added by HTTP logging middleware are visible app.UseHttpLogging(); // ❸ Add middleware conditionally based on the runtime environment if (app.Environment.IsDevelopment()) { // ❹ HTTP logging middleware — logs each request in development app.UseHttpLogging(); } // ❺ New endpoint at /person — returns C# object as JSON app.MapGet("/person", () => { return new Person("Al", "Pacino"); // Output: {"firstName":"Al","lastName":"Pacino"} }); app.Run(); // ❻ Record type — immutable data carrier, one line record Person(string FirstName, string LastName);
📋
Note — What the application does
The application configures two features:
(1) When running in the Development environment, details about each request are logged using HttpLoggingMiddleware.
(2) A new endpoint at /person creates an instance of the C# record Person and serializes it in the response as JSON.

When you run the application and send requests via a web browser, you see details about the request displayed in the console.

Adding and Configuring Services

ASP.NET Core uses small, modular components for distinct features. Each component can evolve independently with loose coupling. These components are exposed as services within the application.

💡
What is a Service?
Within the context of ASP.NET Core, a service refers to any class that provides functionality to an application. Services could be classes exposed by a library or code you've written for your application. Services are registered in the DI container and injected wherever they are needed.
C#Registering services with the DI container
var builder = WebApplication.CreateBuilder(args); // Framework services — add capabilities to your app builder.Services.AddRazorPages(); // Razor Pages engine builder.Services.AddControllers(); // MVC controllers builder.Services.AddDbContext<AppDbContext>(o => o.UseSqlServer(builder.Configuration.GetConnectionString("Db"))); // Your own services builder.Services.AddScoped<IEmailService, SmtpEmailService>(); builder.Services.AddSingleton<ICacheService, MemoryCacheService>(); var app = builder.Build();

Middleware and Endpoints

A WebApplication instance lets you do three things: add middleware to the pipeline, map endpoints, and run the application.

ConceptWhat it isHow to add
Middleware Small components that execute sequentially on each HTTP request. Handles logging, auth, static files, error handling. app.Use*() extension methods
Endpoints Define the response for a specific URL pattern. Executed after routing selects the right one. app.MapGet(), app.MapPost(), etc.
Routing Middleware Inspects the request URL and selects which endpoint to invoke. Added automatically by WebApplication
Endpoint Middleware Executes the selected endpoint and generates the final response. Added automatically by WebApplication
📋
Note — Routing + Endpoint Middleware
WebApplication automatically adds more middleware, including two of the most important: the routing middleware and the endpoint middleware.

Together, this pair is responsible for interpreting the request to determine which endpoint to invoke, reading parameters from the request, and generating the final response. For each request, the routing middleware uses the request's URL to determine which endpoint to invoke. Then the rest of the middleware pipeline executes until the request reaches the endpoint middleware, which executes the endpoint to generate the final response.
