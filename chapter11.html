Chapter 11

Documenting APIs with OpenAPI

Generate rich, machine-readable API descriptions, explore them with Swagger UI, and auto-generate strongly-typed C# clients using NSwag.

1

What is OpenAPI?

OpenAPI (previously called Swagger) is a language-agnostic specification for describing RESTful APIs. At its core, OpenAPI defines the schema of a JSON document that describes the URLs available in your application, how to invoke them, and the data types they return.

💡
OpenAPI vs Swagger — What's the difference?
OpenAPI is the name of the specification. Swagger is the name of the tooling ecosystem built around it (Swagger UI, Swashbuckle, etc.). Think of OpenAPI as the standard, and Swagger as the brand of tools that implement it.

Providing an OpenAPI document for your application unlocks a range of automation possibilities:

🔍
Swagger UI
Browse and test endpoints visually
⚡
Client Generation
Auto-generate typed C# clients
☁️
API Gateways
Integrate with Azure API Management
📝
WSDL Analogy
If you're familiar with SOAP from old ASP.NET, think of OpenAPI as the HTTP/REST equivalent of WSDL. Just as a .wsdl file described your XML SOAP services, an OpenAPI document describes your REST API.

2

Adding an OpenAPI Description

ASP.NET Core supports OpenAPI document generation through third-party libraries. The two most popular are NSwag and Swashbuckle. In this chapter we use Swashbuckle.

Install via the .NET CLI:

CLITerminal
dotnet add package Swashbuckle.AspNetCore

Then configure your Program.cs to enable OpenAPI generation and expose it via middleware:

C#Program.cs — Adding Swashbuckle
using System.Collections.Concurrent; WebApplicationBuilder builder = WebApplication.CreateBuilder(args); builder.Services.AddEndpointsApiExplorer(); // ❶ Endpoint discovery builder.Services.AddSwaggerGen(); // ❷ OpenAPI document generator WebApplication app = builder.Build(); var _fruit = new ConcurrentDictionary<string, Fruit>(); app.UseSwagger(); // ❸ Exposes /swagger/v1/swagger.json app.UseSwaggerUI(); // ❹ Serves the Swagger UI at /swagger app.MapGet("/fruit/{id}", (string id) => _fruit.TryGetValue(id, out var fruit) ? TypedResults.Ok(fruit) : Results.Problem(statusCode: 404)); app.MapPost("/fruit/{id}", (string id, Fruit fruit) => _fruit.TryAdd(id, fruit) ? TypedResults.Created($"/fruit/{id}", fruit) : Results.ValidationProblem(new Dictionary<string, string[]> { { "id", new[] { "A fruit with this id already exists" } } })); app.Run(); record Fruit(string Name, int Stock);
HOW IT WORKS — SWASHBUCKLE PIPELINE
1
ASP.NET Core Metadata Service reads all endpoint metadata (route, verb, parameters, types)
2
Swashbuckle OpenAPI Services process the metadata and generate a JSON OpenAPI document
3
Swagger Middleware serves the OpenAPI document at /swagger/v1/swagger.json
4
Swagger UI Middleware serves an interactive web UI at /swagger built from the document

You can also customize the document title and description by passing options to AddSwaggerGen():

C#Customizing the OpenAPI document info
builder.Services.AddSwaggerGen(x => x.SwaggerDoc("v1", new OpenApiInfo() { Title = "Fruitify", Description = "An API for interacting with fruit stock", Version = "1.0" }));

Testing Your APIs with Swagger UI

Swagger UI is an open-source web UI that visualizes and lets you test your OpenAPI endpoints. Think of it as a lightweight, built-in alternative to Postman for local development.

After calling app.UseSwaggerUI(), navigate to /swagger in your running app. You'll see:

Feature Description
Endpoint listAll routes with their HTTP verb and route template
Parameter detailsRequired/optional, source (path/query/body), and type
Response schemasExpected response types and status codes
"Try it out"Fill in parameters and execute real HTTP requests to your running app
curl equivalentSwagger shows the equivalent curl command after every request
✨
Swagger UI vs Postman
Swagger UI is great for quick local testing — no setup needed. Postman shines when you need collections, team sharing, environments, and scripted test flows. For individual endpoint testing during development, Swagger UI is perfect.

Adding Metadata to Your Minimal APIs

Metadata is information about an API that doesn't change how the endpoint executes — it enriches the OpenAPI description. Without it, Swagger UI may show incomplete response information.

There are three broad categories of metadata you can add:

Routing
e.g. WithName() — globally unique name for URL generation
Middleware
e.g. authorization, hostname filtering, output caching per-endpoint
OpenAPI
Drives Swashbuckle's document generation — tags, response types, etc.

Here's how to add tags and explicitly declare response types so that Swagger UI shows accurate information:

C#Adding OpenAPI metadata to endpoints
app.MapGet("/fruit/{id}", (string id) => _fruit.TryGetValue(id, out var fruit) ? TypedResults.Ok(fruit) : Results.Problem(statusCode: 404)) .WithTags("fruit") // ❶ Groups endpoint in Swagger UI .Produces<Fruit>() // ❷ Documents the 200 OK response type .ProducesProblem(404); // ❸ Documents the 404 Problem Details response app.MapPost("/fruit/{id}", (string id, Fruit fruit) => _fruit.TryAdd(id, fruit) ? TypedResults.Created($"/fruit/{id}", fruit) : Results.ValidationProblem(new Dictionary<string, string[]> { { "id", new[] { "A fruit with this id already exists" } } })) .WithTags("fruit") .Produces<Fruit>(201) // ❹ Returns 201 Created, not 200 OK .ProducesValidationProblem(); // ❺ Documents the 400 validation error response
✨
Use Route Groups for Shared Metadata
You can use route groups (from Chapter 5) to apply tags and other metadata to multiple endpoints at once, avoiding repetition. For example: var fruit = app.MapGroup("/fruit").WithTags("fruit");

Generating Strongly-Typed Clients with NSwag

One of the most powerful features of OpenAPI is automatic client generation. Using NSwag, you can generate a C# class that wraps all the HTTP calls to your API — no manual serialization, no string interpolation for paths, no error-code checking boilerplate.

🔍
How Code Generation Works
1. Visual Studio or the .NET tool downloads your OpenAPI JSON locally.
2. The NuGet package hooks into your build, reads the JSON, and generates a Client.cs file in your obj folder.
3. Any time the JSON changes, the client regenerates automatically on the next build.

Using Visual Studio

Right-click your client project → Add > Service Reference → select OpenAPI → paste the URL to your swagger.json → set a namespace and class name → click Finish.

Using the .NET Global Tool

CLITerminal — Installing and using the OpenAPI tool
# Install once globally dotnet tool install -g Microsoft.dotnet-openapi # Add the OpenAPI reference from your project folder dotnet openapi add url http://localhost:5062/swagger/v1/swagger.json \ --output-file OpenAPIs\fruit.json # Install required packages dotnet add package NSwag.ApiDescription.Client dotnet add package Microsoft.Extensions.ApiDescription.Client dotnet add package Newtonsoft.Json

After either approach, your .csproj will contain an <OpenApiReference> element. Update the packages to their latest versions and add ClassName/Namespace attributes:

XML.csproj — Updated OpenAPI reference
<OpenApiReference Include="OpenAPIs\fruit.json" SourceUrl="http://localhost:5062/swagger/v1/swagger.json" Namespace="Fruit" ClassName="FruitClient" /> <!-- Keep these at latest versions: --> <PackageReference Include="Microsoft.Extensions.ApiDescription.Client" Version="7.0.0" /> <PackageReference Include="NSwag.ApiDescription.Client" Version="13.18.2" /> <PackageReference Include="Newtonsoft.Json" Version="13.0.1" />
⚠️
Visual Studio adds out-of-date packages
After using Visual Studio's Add Service Reference wizard, you must manually update NSwag.ApiDescription.Client and Microsoft.Extensions.ApiDescription.Client to their latest versions — the wizard installs old ones that may not build.

6

Using the Generated Client

After building, NSwag generates a FruitClient class (or whatever class name you specified) in your obj folder. Using it is straightforward:

C#Program.cs — Using the generated FruitClient
using Fruit; // ❶ Generated namespace var client = new FruitClient( // ❷ Instantiate the generated client "https://localhost:7186", // ❸ Base address of the API new HttpClient()); // ❹ HttpClient used for requests // ❺ Call the MapPost endpoint — auto-serializes the body Fruit.Fruit created = await client.FruitPOSTAsync("123", new Fruit.Fruit { Name = "Banana", Stock = 100 }); Console.WriteLine($"Created {created.Name}"); // ❻ Call the MapGet endpoint — auto-deserializes the response Fruit.Fruit fetched = await client.FruitGETAsync("123"); Console.WriteLine($"Fetched {fetched.Name}");
✨
Improve Auto-Generated Method Names
NSwag uses the endpoint's OperationID for method names. Without it, you get ugly names like FruitPOSTAsync. Add .WithName("CreateFruit") to your endpoint, refresh the OpenAPI document, and NSwag will generate a clean CreateFruitAsync() method instead.
7

Customizing the Generated Client

NSwag provides several options to configure the generated code. Add them as an <Options> element inside <OpenApiReference> in your .csproj:

XML.csproj — NSwag customization options
<OpenApiReference Include="OpenAPIs\fruit.json" SourceUrl="http://localhost:5062/swagger/v1/swagger.json" Namespace="Fruit" ClassName="FruitClient"> <Options>/UseBaseUrl:false /GenerateClientInterfaces:true /JsonLibrary:SystemTextJson</Options> </OpenApiReference> // ← Don't forget to close the outer element!
OptionDefaultWhat it does
/UseBaseUrl:false true Removes the baseUrl constructor arg; relies on HttpClient.BaseAddress instead — friendlier for DI
/GenerateClientInterfaces:true false Generates an IFruitClient interface — lets you mock the client in unit tests
/JsonLibrary:SystemTextJson Newtonsoft.Json Uses the built-in System.Text.Json instead of Newtonsoft — fewer dependencies

After applying these options, update your consuming code:

C#Using the updated client with interface and HttpClient.BaseAddress
IFruitClient client = new FruitClient( // ❶ Now uses the interface new HttpClient() { BaseAddress = // ❷ Base address on HttpClient new Uri("https://localhost:7186") }); Fruit.Fruit created = await client.FruitPOSTAsync("123", new Fruit.Fruit { Name = "Banana", Stock = 100 });
⚠️
Options Don't Auto-Regenerate
Changing <Options> in your .csproj does not trigger regeneration on its own. If the client doesn't update, delete your obj folder and rebuild. Alternatively, make a tiny change in the JSON file, build, then revert it.

Adding Summaries & Descriptions

To add human-readable summaries and descriptions to your endpoints, install the Microsoft.AspNetCore.OpenApi package and call WithOpenApi() on your endpoint:

CLITerminal
dotnet add package Microsoft.AspNetCore.OpenApi
C#Adding summary and description via fluent methods
app.MapGet("/fruit/{id}", (string id) => _fruit.TryGetValue(id, out var fruit) ? TypedResults.Ok(fruit) : Results.Problem(statusCode: 404)) .WithName("GetFruit") .WithTags("fruit") .Produces<Fruit>() .ProducesProblem(404) .WithSummary("Fetches a fruit") // ❶ Short summary .WithDescription("Fetches a fruit by id, or returns 404" + " if no fruit with the ID exists") // ❷ Longer description .WithOpenApi(); // ❸ Required to expose summary/desc to Swashbuckle

To also add a description for individual parameters, use the WithOpenApi() overload that accepts a lambda:

C#Describing parameters inline
.WithOpenApi(o => { o.Parameters[0].Description = "The id of the fruit to fetch"; o.Summary = "Fetches a fruit"; return o; });
💡
WithOpenApi() is the key
WithSummary() and WithDescription() add metadata to the endpoint, but that metadata is only picked up by Swashbuckle when you also call WithOpenApi(). Without it, the summary and description are silently ignored in the OpenAPI document.
9

Attributes vs Fluent Methods

Almost every fluent metadata method has an equivalent attribute you can apply directly to the handler lambda or method. Both produce the same OpenAPI output — it's a matter of style preference.

Fluent Methods
app.MapGet("/fruit/{id}", handler) .WithName("GetFruit") .WithTags("fruit") .WithSummary("Fetches a fruit") .Produces<Fruit>() .ProducesProblem(404) .WithOpenApi();
Attributes
[EndpointName("GetFruit")] [Tags("fruit")] [EndpointSummary("Fetches a fruit")] [ProducesResponseType(typeof(Fruit), 200)] [ProducesResponseType(..., 404)] (string id) => ...
⚠️
WithOpenApi() cannot be replaced
Even when using attributes, you must still call .WithOpenApi() on the endpoint for Swashbuckle to read the attribute metadata. It is the only call with no attribute equivalent.

XML Documentation Comments

For a more "C# native" feel, you can use XML documentation comments (///) on your handler methods as the source of OpenAPI summaries and parameter descriptions.

⚠️
Limitation: No lambda support in .NET 7
XML comments on endpoint handlers only work with static or instance methods, not lambda methods or local functions. Lambda support is tracked at dotnet/aspnetcore#39927.

Enable in three steps:

Step 1: Enable documentation generation in your .csproj:

XML.csproj
<PropertyGroup> <GenerateDocumentationFile>true</GenerateDocumentationFile> </PropertyGroup>

Step 2: Configure Swashbuckle to read the generated XML file:

C#Program.cs
builder.Services.AddSwaggerGen(opts => { var file = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml"; opts.IncludeXmlComments(Path.Combine(AppContext.BaseDirectory, file)); });

Step 3: Use a class method handler with XML comments:

C#FruitHandler.cs — XML comments as OpenAPI docs
internal class FruitHandler { /// <summary> /// Fetches a fruit by id, or returns 404 if it does not exist /// </summary> /// <param name="id">The ID of the fruit to fetch</param> /// <response code="200">Returns the fruit if it exists</response> /// <response code="404">If the fruit doesn't exist</response> [ProducesResponseType(typeof(Fruit), 200)] [ProducesResponseType(404)] [Tags("fruit")] public IResult GetFruit(string id) => _fruit.TryGetValue(id, out var fruit) ? TypedResults.Ok(fruit) : Results.Problem(statusCode: 404); } // Register in Program.cs: var handler = new FruitHandler(_fruit); app.MapGet("/fruit/{id}", handler.GetFruit).WithName("GetFruit");
✨
Development-Only Swagger
If you only want Swagger UI for local testing, register and activate it conditionally so it never runs in production:
if (app.Environment.IsDevelopment()) { app.UseSwagger(); app.UseSwaggerUI(); }
11

Knowing the Limitations of OpenAPI

OpenAPI is powerful, but it's not a silver bullet. The specification itself acknowledges that not all services can be described by OpenAPI. Here are the key rough edges to know:

❌ Some APIs can't be described
Dynamic API styles like HATEOAS include response links that change per request and per user — OpenAPI expects static, up-front knowledge of responses. It also struggles with complex multi-document API gateways.
❌ Generated code is opinionated
NSwag generates code you may not like — exceptions for error responses, two separately generated Product types that won't interchange between client classes. Sometimes writing your own client is simpler.
❌ Tooling lags the spec
OpenAPI spec, Swashbuckle, NSwag, and Swagger UI evolve independently. A new spec feature may not yet be supported by Swashbuckle, meaning NSwag can't generate client code for it.
✨
Consider gRPC for Code-Gen Heavy Workflows
If you love code generation but prefer an RPC-style approach, gRPC offers first-class code generation support across multiple languages with excellent .NET tooling. See learn.microsoft.com/aspnet/core/grpc.
