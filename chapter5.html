Chapter 5

Creating a JSON API with Minimal APIs

The complete reference for building HTTP JSON APIs with ASP.NET Core 8 Minimal APIs. Every concept, every code example, every slide note — in one place.

1

What is an HTTP API?

Traditional web applications built with Razor Pages return a complete HTML page on every request. The server renders the full page and sends it. This is called server-side rendering.

An HTTP API works differently — instead of returning HTML, it returns raw data, almost always as JSON. The caller is responsible for deciding what to do with that data.

HTTP APIs are also called web APIs, JSON APIs, or REST APIs depending on the design specifics. In ASP.NET Core, "web API" refers to a specific older technology using controllers — Minimal APIs are the modern, low-ceremony replacement.

📱
When to use an HTTP API
• Your frontend is a React / Vue / Angular / Blazor WebAssembly SPA
• You are building a mobile app backend (Android, iOS)
• Other backend services need to call your server (microservices)
• You want one backend to serve many different clients simultaneously
• Example: a web application that sends emails can expose an HTTP API so other services can trigger it
Traditional App (Razor Pages)
// Browser → GET /products // Server builds and returns: <html> <body> <h1>Products</h1> <ul><li>Apple</li></ul> </body> </html> // HTML is ready to display — server did all work
HTTP API (Minimal API)
// Browser → GET /products // Server returns: [{"id":"apple","name":"Apple","stock":10}] // Just data — no HTML // The client decides how to display it // Works for browsers, mobile apps, other services

2

SPAs and the API Model

A Single Page Application (SPA) is a web app that loads once in the browser and never does a full page reload again. Frameworks like Angular, React, and Vue build SPAs. Blazor WebAssembly is a newer SPA framework from Microsoft that lets you write C# instead of JavaScript — it runs in the browser using the WebAssembly web standard.

FlowComplete SPA ↔ API lifecycle
// Step 1 — User visits the site for the first time Browser → GET https://myapp.com Server ← index.html + the full JavaScript bundle (your React/Vue/Angular app) // Step 2 — JavaScript runs in browser, SPA initializes // No more full page reloads after this. The SPA controls the UI. // Step 3 — SPA needs data, calls the ASP.NET Core API Browser → GET https://myapp.com/api/fruit Server ← [{"id":"apple","name":"Apple","stock":10}] ← JSON only, no HTML // Step 4 — SPA parses the JSON and generates HTML dynamically in the browser // The server never touched the HTML rendering. // Step 5 — User clicks "Delete Apple" Browser → DELETE https://myapp.com/api/fruit/apple Server ← 204 No Content ← success, nothing to return // SPA removes the item from its local state — no page reload
💡
Key mental model
The server is data-only. It does not know or care what the UI looks like. The same API endpoint serves a React web app, an Android app, and an iOS app simultaneously. All three clients get the same JSON — each renders it differently.

Defining Minimal API Endpoints

A Minimal API endpoint maps a URL (route) to a C# function (handler). No controller classes, no attributes — just a URL pattern and a function. ASP.NET Core automatically serializes your C# return value to JSON.

C#Route parameters — extracted automatically from the URL
// The {name} in the route template is a route parameter app.MapGet("/person/{name}", (string name) => { // GET /person/Andrew → name = "Andrew" // GET /person/Al → name = "Al" return new Person(name, "Pacino"); // Automatically serialized to JSON: // [{"firstName":"Al","lastName":"Pacino"}] }); // By default, minimal APIs serialize C# objects to JSON automatically. // You never call JsonSerializer.Serialize() yourself.
💡
C# Records — data carriers for your API
A record is a reference type introduced in C# 9.0 for representing immutable data. All properties are read-only by default — once created, the data cannot change. Records give you value-based equality: two records with the same data are considered equal. They also auto-generate a constructor, Equals(), GetHashCode(), ToString(), and a deconstructor.
Class — 15 lines of boilerplate
public class Fruit { public string Name { get; init; } public int Stock { get; init; } public Fruit(string name, int stock) { Name = name; Stock = stock; } // Plus manual Equals(), GetHashCode(), // ToString() if you want them }
Record — 1 line, same result
record Fruit(string Name, int Stock); // That's it. You get: // ✓ Constructor: new Fruit("Apple", 10) // ✓ Read-only properties // ✓ Value equality // ✓ ToString(), Equals(), GetHashCode() // ✓ Deconstructor

HTTP Verbs — Mapping the Right Verb to Each Action

Every HTTP request carries a verb (method) that describes the intent of the request. Using the right verb is not optional for an API — it is a semantic contract. Minimal APIs use different Map*() functions for each verb.

VerbMeaningMinimal API methodNotes
GETRetrieve data. Never modify anything.app.MapGet()Safe + Idempotent
POSTCreate a new resource.app.MapPost()Body contains the new data
PUTReplace an existing resource completely.app.MapPut()Idempotent
DELETERemove a resource.app.MapDelete()Returns 204 on success
PATCHPartially update a resource.app.MapPatch()Only send changed fields
Any verbHandle multiple verbs on one endpoint.app.MapMethods()Avoid unless necessary
All verbsMatch any verb (no specific one).app.Map()Avoid unless necessary
FallbackMatches if no other endpoint matched.app.MapFallback()Useful for SPA client-side routing
🚫
405 Method Not Allowed
If your endpoint is registered with MapGet but a client sends a POST request, ASP.NET Core returns 405 Method Not Allowed automatically — your handler code never runs. Always match the verb in your request to the verb in your Map*() call. Even if the URL is correct, a wrong verb triggers this response.
⚠️
Why not just use GET and POST for everything?
Razor Pages and HTML forms only support GET and POST, so many Razor apps use only those two. This is acceptable for server-rendered apps. But for an API, you should use HTTP verbs with the correct semantics — it makes the API self-documenting and allows intermediaries (caches, proxies) to behave correctly.

Route Handlers — Five Ways to Define the Same Handler

The function you pass to MapGet() (or any Map*()) is called a route handler. C# gives you five syntactically different ways to write it — they all compile to the same behavior. The slides mark each one with a callout number:

C#❶ Inline lambda — simplest, least descriptive
app.MapGet("/fruit", () => Fruit.All); // ✓ Concise, good for very short endpoints // ✗ The function has no name — harder to debug in stack traces
C#❷ Named lambda variable — adds a name to the lambda
var getFruit = () => Fruit.All; app.MapGet("/fruit", getFruit); // ✓ Same as ❶ but now has the name 'getFruit' // ✓ Easier to read when Program.cs gets long
C#❸ Static method — recommended for large projects
app.MapGet("/fruit", FruitHandlers.GetAll); // --- In a separate file: FruitHandlers.cs --- public static class FruitHandlers { public static IEnumerable<Fruit> GetAll() => Fruit.All.Values; } // ✓ Keeps Program.cs clean // ✓ Each endpoint module lives in its own file
C#❹ Instance method — for handlers needing DI
var handlers = new FruitHandlers(); app.MapGet("/fruit", handlers.GetAll); public class FruitHandlers { public IEnumerable<Fruit> GetAll() => Fruit.All.Values; } // ✓ Use when the handler class needs injected services in its constructor
C#❺ Local function — C# 7.0+, defined in the same scope
app.MapGet("/fruit", GetFruit); IEnumerable<Fruit> GetFruit() => Fruit.All.Values; // Defined below the MapGet call in the same scope (Program.cs top-level) // ✓ Has a name, still inline — middle ground between ❶ and ❸
ℹ️
Which to use?
Minimal APIs are flexible by design — you can organize endpoints any way you choose. For production, extract endpoints into separate files (option ❸ or ❹) to keep your code modular and testable. The book recommends embracing this modularity — it's the way Minimal APIs were intended to be used.

Body Binding — Reading Complex Types from the Request

Route parameters like {id} are simple types extracted directly from the URL. But what about sending a full object to the server (e.g., a new fruit to add)? Minimal APIs automatically deserialize the JSON request body into any complex type parameter in your handler.

⚠️
Minimal API binding limitation — important for the exam
Unlike ASP.NET Core Web API controllers (which support multiple binding sources and multiple JSON libraries), Minimal APIs can only bind to JSON bodies and always use System.Text.Json for deserialization. They do not support XML bodies or other binding sources like form data natively.
C#Complete Fruit API — data model + all handlers
// The data model — a record with a static dictionary to store all fruits record Fruit(string Name, int Stock) { // Static dictionary shared by all requests (in-memory storage) public static Dictionary<string, Fruit> All { get; } = new Dictionary<string, Fruit>(); } // Handler class with static and instance methods class Handlers { // ❸ Static method — replaces an existing fruit by id public static void ReplaceFruit(string id, Fruit fruit) { // 'fruit' is deserialized FROM THE JSON REQUEST BODY automatically // Both id (from URL) and fruit (from body) are bound automatically Fruit.All[id] = fruit; } // ❹ Instance method public void AddFruit(string id, Fruit fruit) { Fruit.All.Add(id, fruit); } } // Wiring everything up in Program.cs: var handler = new Handlers(); app.MapPut("/fruit/{id}", Handlers.ReplaceFruit); // static ❸ app.MapPost("/fruit/{id}", handler.AddFruit); // instance ❹
💡
How binding works — the rule
Simple types (string, int, bool, etc.) → extracted from route parameters or query string.
Complex types (records, classes, objects) → deserialized from the JSON request body.
A handler can receive both simultaneously: (string id, Fruit fruit) — id from the URL, fruit from the body.
🧠 Your handler is app.MapPost("/orders/{customerId}", (string customerId, Order order) => ...). The client sends a POST request with a JSON body. Where does ASP.NET Core get customerId and where does it get order?

IResult — Controlling the HTTP Response

By default, returning a C# object gives a 200 OK with a JSON body. Real APIs need precise status codes. The IResult interface lets you specify the exact response — status code, body, and headers.

The endpoint middleware handles each return type according to a specific ruleset:

What you returnStatus codeBodyContent-Type
void or Task200 OKEmpty—
string or Task<string>200 OKThe string as-istext/plain
POCO / record / any object200 OKJSON of the objectapplication/json
IResult or Task<IResult>Depends on the implementationDepends on the implementationDepends
C#Using IResult with ConcurrentDictionary (thread-safe)
// ConcurrentDictionary is used instead of Dictionary // because multiple HTTP requests can arrive simultaneously. // Dictionary is NOT thread-safe — two simultaneous requests can corrupt it. // ConcurrentDictionary uses fine-grained locking internally — safe for concurrent access. var _fruit = new ConcurrentDictionary<string, Fruit>(); app.MapGet("/fruit/{id}", (string id) => { // TryGetValue: returns true if key exists, assigns the value to 'fruit' // 'out var fruit' declares and assigns in one step // 'out' = output parameter — the method writes into it return _fruit.TryGetValue(id, out var fruit) ? TypedResults.Ok(fruit) // found → 200 OK + JSON body : Results.NotFound(); // not found → 404 // ↑ Ternary operator: condition ? valueIfTrue : valueIfFalse });
ℹ️
Why ConcurrentDictionary?
A web server handles multiple requests simultaneously (multi-threaded). A regular Dictionary is not thread-safe — if two requests try to write at the same time, the dictionary can get corrupted. ConcurrentDictionary provides built-in support for concurrent access via fine-grained locking, preventing data corruption or race conditions.

Results vs TypedResults

Both Results and TypedResults are static helper types in Microsoft.AspNetCore.Http that generate common HTTP responses. They perform exactly the same function at runtime. The only difference is the compile-time return type.

Results — returns IResult (opaque)
// Return type: IResult // Compiler sees a black box return Results.Ok(fruit); // In unit tests — painful casting: var r = result as IValueHttpResult; var v = (Fruit)r!.Value!; // ugly, fragile // Swagger can't infer the response type
TypedResults — returns Ok<T> (concrete)
// Return type: Ok<Fruit> // Compiler knows exactly what's inside return TypedResults.Ok(fruit); // In unit tests — clean: var ok = result as Ok<Fruit>; var v = ok!.Value!.Name; // clear, type-safe // Swagger can generate the correct response schema
💡
Simple decision rule
Use TypedResults in production — better testability and Swagger docs.
Use Results for quick prototypes, or when a handler mixes many different return types that make TypedResults signatures awkward.
🧠 An endpoint returns either TypedResults.Ok(fruit) or Results.NotFound(). What is the return type of the handler function?

Writing the Response Manually with HttpResponse

For cases where Results / TypedResults don't give enough control, you can access the raw HttpResponse object directly. You write the status code, headers, and body yourself using streams.

C#Manual HttpResponse — for advanced control
app.MapGet("/manual", async (HttpResponse response) => { // Set the status code manually response.StatusCode = 200; // Set headers response.ContentType = "application/json"; // Write the body as a string (async) await response.WriteAsync("{\"message\":\"Hello!\"}"); // Return void — the response is already written // No return value needed when using HttpResponse directly }); // ASP.NET Core injects HttpResponse automatically — same as route params. // You can combine it with other params: app.MapGet("/manual/{id}", async (string id, HttpResponse response) => { response.StatusCode = 200; await response.WriteAsJsonAsync(new { Id = id }); });
⚠️
Use this sparingly
Writing to HttpResponse directly means you lose the type-safety and Swagger integration of TypedResults. Only use it when you need streaming, chunked transfer, or very fine-grained control over headers. For most endpoints, prefer Results / TypedResults.

Problem Details — Structured Error Responses

Problem Details is a web standard (RFC 7807) that defines a consistent JSON format for HTTP API errors. Instead of a bare 404 or a plain text message, you return a JSON object that any client can parse and understand programmatically.

Enable Problem Details globally in Program.cs:

var builder = WebApplication.CreateBuilder(args); // Register the IProblemDetailsService — required for automatic conversion builder.Services.AddProblemDetails(); var app = builder.Build(); // Convert unhandled exceptions → Problem Details (500) // When called without a path, it uses IProblemDetailsService automatically app.UseExceptionHandler(); // Convert raw 404/405/etc responses → Problem Details automatically app.UseStatusCodePages();

Endpoint Filters — Separating Cross-Cutting Concerns

An endpoint filter runs code before and/or after your handler — without cluttering the handler itself. It's analogous to middleware, but scoped to one specific endpoint (or group). The filter pipeline mirrors the middleware pipeline in structure.

❌ Without filter — cluttered handler
app.MapGet("/fruit/{id}", (string id) => { // Validation mixed into business logic if (string.IsNullOrEmpty(id) || !id.StartsWith('f')) return Results.ValidationProblem(...); return _fruit.TryGetValue(id, out var f) ? TypedResults.Ok(f) : Results.NotFound(); });
✅ With filter — clean separation
app.MapGet("/fruit/{id}", (string id) => { // Only business logic here return _fruit.TryGetValue(id, out var f) ? TypedResults.Ok(f) : Results.NotFound(); }) .AddEndpointFilter(async (ctx, next) => { var id = ctx.GetArgument<string>(0); if (string.IsNullOrEmpty(id) || !id.StartsWith('f')) return Results.ValidationProblem(...); return await next(ctx); });
ℹ️
Filter parameters explained
The filter function takes two parameters:
• context — provides details about the endpoint (arguments, metadata). Use context.GetArgument<T>(index) to read handler parameters by position.
• next — represents the rest of the filter pipeline. Calling await next(context) passes control to the next filter (or the handler if no more filters remain). If you do NOT call next, the pipeline is short-circuited and the handler never runs.

Multiple Filters — Building a Filter Pipeline

You can chain multiple AddEndpointFilter() calls on one endpoint. This builds an endpoint filter pipeline, which works exactly like the middleware pipeline — filters execute in order, and each can short-circuit the rest.

C#Two filters chained on one endpoint
app.MapGet("/fruit/{id}", (string id) => _fruit[id]) .AddEndpointFilter(async (ctx, next) => // Filter 1 — validates id format { Console.WriteLine("Filter 1: BEFORE"); var id = ctx.GetArgument<string>(0); if (string.IsNullOrEmpty(id) || !id.StartsWith('f')) return Results.ValidationProblem(...); // short-circuits here var result = await next(ctx); // passes to Filter 2 Console.WriteLine("Filter 1: AFTER"); return result; }) .AddEndpointFilter(async (ctx, next) => // Filter 2 — logs the request { Console.WriteLine("Filter 2: BEFORE"); var result = await next(ctx); // passes to handler Console.WriteLine("Filter 2: AFTER"); return result; });

The order of execution for a valid request (id passes validation):

Log outputValid request — all filters and handler run
// Request: GET /fruit/f-apple (passes filter 1) Filter 1: BEFORE Filter 2: BEFORE // Handler runs here Filter 2: AFTER Filter 1: AFTER
Log outputInvalid request — short-circuited at Filter 1
// Request: GET /fruit/INVALID (fails filter 1 — no 'f' prefix) Filter 1: BEFORE // Filter 1 returns ValidationProblem — neither Filter 2 nor handler runs // Response: 400 Bad Request (Problem Details)
⚠️
Keep it simple — filters have a cost
Moving code to filters reduces clutter in handlers, but it makes the application flow harder to follow. Only use filters when you find the same logic duplicated in multiple endpoints. Even then, prefer a filter over a plain method call only if it significantly simplifies the code.
🧠 You have two filters on an endpoint. Filter 1 runs first and calls next(). Filter 2 runs second and does NOT call next(), returning its own result instead. Does the endpoint handler ever execute?

IEndpointFilter — Reusable, Class-Based Filters

Writing filters as inline lambdas can be frustrating — the compiler requires a specific signature, and IDE autocompletion is limited. The IEndpointFilter interface lets you define a class with a single required method, giving you full IntelliSense and making the filter reusable across endpoints.

C#IEndpointFilter implementation
public class ValidateIdFilter : IEndpointFilter { // IEndpointFilter requires exactly this method signature public async ValueTask<object?> InvokeAsync( EndpointFilterInvocationContext context, // same as 'ctx' in lambda filters EndpointFilterDelegate next) // same as 'next' in lambda filters { var id = context.GetArgument<string>(0); // first param of the handler if (string.IsNullOrEmpty(id) || !id.StartsWith('f')) { return Results.ValidationProblem( new Dictionary<string, string[]> { ["id"] = new[] { "Id must start with the letter 'f'." } } ); } return await next(context); // pass to next filter or handler } } // Apply with the generic overload — clean one-liner: app.MapGet("/fruit/{id}", (string id) => _fruit[id]) .AddEndpointFilter<ValidateIdFilter>();
💡
Lambda vs IEndpointFilter — when to use each
Use lambda filters for quick, one-off checks on a single endpoint.
Use IEndpointFilter when the filter is reused across multiple endpoints or becomes complex enough to warrant its own class and unit tests.
Note: there is no IEndpointFilter equivalent for the filter-factory pattern — that only works with lambdas.

Filters vs Middleware — Which to Choose?

Both filters and middleware intercept requests. The key difference is where in the pipeline they run and what they have access to.

MiddlewareEndpoint Filters
Runs forEvery request in the appOnly the specific endpoint(s) it is attached to
Access to handler return valueNo — runs before/after the entire pipelineYes — can inspect or modify the IResult returned by the handler
Access to endpoint metadataLimitedFull — knows which handler will run, its parameters, attributes
Pipeline levelLower — sees raw HTTP request/responseHigher — tied directly to the endpoint handler execution
Best forLogging, authentication, CORS, compression — app-wide concernsEndpoint-specific validation, transformation, authorization checks
ℹ️
Decision rule
If the concern applies to all requests → use middleware.
If the concern is specific to one or a few endpoints → use endpoint filters.
Middleware sees only the raw HTTP request/response objects. Filters see the endpoint handler's arguments and return value — making them uniquely powerful for endpoint-level logic.

Route Groups — Organizing Your API

MapGroup() lets you collect endpoints that share a URL prefix and/or filters into a named group. The prefix and any filters are applied to all endpoints in the group automatically.

❌ Without groups — repetitive, error-prone
app.MapGet("/fruit", ...).AddEndpointFilter<ValidateIdFilter>(); app.MapGet("/fruit/{id}", ...).AddEndpointFilter<ValidateIdFilter>(); app.MapPost("/fruit/{id}", ...).AddEndpointFilter<ValidateIdFilter>(); app.MapPut("/fruit/{id}", ...).AddEndpointFilter<ValidateIdFilter>(); app.MapDelete("/fruit/{id}", ...).AddEndpointFilter<ValidateIdFilter>(); // Add a new endpoint? Easy to forget the filter.
✅ With MapGroup() — define once, apply everywhere
var fruit = app.MapGroup("/fruit") .AddEndpointFilter<ValidateIdFilter>(); fruit.MapGet("", ...); fruit.MapGet("/{id}", ...); fruit.MapPost("/{id}", ...); fruit.MapPut("/{id}", ...); fruit.MapDelete("/{id}", ...); // Filter automatically applies to all five above.
💡
Nested groups
Groups can be nested: app.MapGroup("/fruit").MapGroup("/citrus") creates a group with prefix /fruit/citrus. Prefixes are applied in order — the outer group's prefix comes first.

using System.Collections.Concurrent; var builder = WebApplication.CreateBuilder(args); builder.Services.AddProblemDetails(); // register IProblemDetailsService var app = builder.Build(); app.UseExceptionHandler(); // unhandled exceptions → Problem Details 500 app.UseStatusCodePages(); // raw 404/405 → Problem Details automatically // In-memory store — thread-safe for concurrent requests var _fruit = new ConcurrentDictionary<string, Fruit>(); // Route group: all endpoints share the "/fruit" prefix // and the ValidateIdFilter is applied to all of them automatically var api = app.MapGroup("/fruit").AddEndpointFilter<ValidateIdFilter>(); // GET /fruit — list all fruits api.MapGet("", () => _fruit); // GET /fruit/{id} — get one fruit by id api.MapGet("/{id}", (string id) => _fruit.TryGetValue(id, out var fruit) ? TypedResults.Ok(fruit) // 200 + JSON body : Results.NotFound() // 404 ); // POST /fruit/{id} — add a new fruit (Fruit deserialized from JSON body) api.MapPost("/{id}", (string id, Fruit fruit) => { if (!_fruit.TryAdd(id, fruit)) return Results.Conflict($"A fruit with id '{id}' already exists."); // 409 return TypedResults.Created($"/fruit/{id}", fruit); // 201 + Location header }); // PUT /fruit/{id} — replace a fruit completely (upsert semantics) api.MapPut("/{id}", (string id, Fruit fruit) => { _fruit[id] = fruit; // adds or replaces return TypedResults.Ok(fruit); // 200 + updated body }); // DELETE /fruit/{id} — remove a fruit api.MapDelete("/{id}", (string id) => { if (!_fruit.TryRemove(id, out _)) return Results.NotFound(); // 404 return Results.NoContent(); // 204 — success, no body }); app.Run(); // ── Data model ────────────────────────────────────────────────────── record Fruit(string Name, int Stock); // ── Reusable filter — validates id starts with 'f' ────────────────── public class ValidateIdFilter : IEndpointFilter { public async ValueTask<object?> InvokeAsync( EndpointFilterInvocationContext context, EndpointFilterDelegate next) { // GetArgument<string>(0) = the first string parameter of the handler ('id') var id = context.GetArgument<string>(0); if (string.IsNullOrEmpty(id) || !id.StartsWith('f')) return Results.ValidationProblem( new Dictionary<string, string[]> { ["id"] = new[] { "Id must start with the letter 'f'." } } ); // short-circuits — handler never runs return await next(context); // valid — continue to handler } }
