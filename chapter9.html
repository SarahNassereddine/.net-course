Chapter 9

Registering Services with Dependency Injection

How to tell the DI container which classes to create, how to create them, and how long to keep them alive — covering all three lifetimes and the captive dependency trap.

1

Registering Your Own Custom Services

From Chapter 8 you know that the DI container needs to be told which implementation to use for each interface. If you forget to register a service and then try to inject it, the minimal API framework gets confused — it sees a parameter it doesn't recognize as a model-binding source and tries to bind it from the JSON request body instead, causing a runtime exception.

🚨
What happens if you forget to register a service
When IEmailSender isn't registered, the minimal API infrastructure doesn't know it's a service — so it treats the parameter as a model-binding target and tries to deserialize it from the HTTP request body. You'll get an exception like: "Failed to bind parameter 'IEmailSender emailSender' from body."

The fix is always the same: register the service in builder.Services.

Configuring DI means making a series of statements like these:

When a service requires IEmailSender → use an instance of EmailSender
When a service requires NetworkClient → use an instance of NetworkClient
When a service requires MessageFactory → use an instance of MessageFactory

You make these statements by calling Add* methods on builder.Services.

2

The Add* Methods — Three Pieces of Information

Every registration call provides three things to the DI container:

1
Service Type (TService)
The type that will be requested as a dependency. Usually an interface like IEmailSender, but can be a concrete type like NetworkClient.
2
Implementation Type (TImplementation)
The concrete class the container will actually create. Must be a real class (not an interface). Can be the same as the service type if you're registering a concrete type directly.
3
Lifetime
Controls how long an instance lives. The method name encodes the lifetime: AddTransient, AddScoped, or AddSingleton. We'll cover these in detail in Section 7.
C#Registering the full EmailSender dependency graph
var builder = WebApplication.CreateBuilder(args); // ❶ Service type = IEmailSender, Implementation = EmailSender builder.Services.AddTransient<IEmailSender, EmailSender>(); // ❷ Service type = NetworkClient, Implementation = NetworkClient (same) builder.Services.AddTransient<NetworkClient>(); // ❸ Service type = MessageFactory, Implementation = MessageFactory (same) builder.Services.AddTransient<MessageFactory>(); var app = builder.Build(); // Now this just works — the container builds the whole graph app.MapPost("/register", (UserDto user, IEmailSender sender) => { sender.SendWelcomeEmail(user.Email); return Results.Ok(); });
📋
Important: EmailSender is only accessible via IEmailSender
Because you registered EmailSender under the service type IEmailSender, you can only resolve it by requesting IEmailSender — not by requesting EmailSender directly. Asking for the concrete type would throw an exception, because the container only knows the interface key.
🧠 You call builder.Services.AddTransient<IEmailSender, EmailSender>(). A handler asks for EmailSender directly as a parameter. What happens?
3

Registering with Objects & Lambdas

The generic Add*<TService, TImplementation>() methods work great for most classes, but the container makes assumptions that not every class can satisfy:

The container assumes the class:
① Is a concrete type (not an interface or abstract class)
② Has a single usable constructor
③ All constructor arguments are either registered services or have default values

Take EmailServerSettings — it requires a Host (string) and Port (int) in its constructor. These are plain values, not registered services. The generic method can't create it because the container doesn't know where to get those values.

📦 Register a pre-built object (Singleton)
// Create the object yourself, pass it to the container // The container will ALWAYS return this same instance // → This is always a singleton (one shared instance) builder.Services.AddSingleton( new EmailServerSettings("smtp.example.com", 587) );





🔧 Register a factory lambda (any lifetime)
// Provide a function; container calls it when it needs one // ❶ Not restricted to singleton — can use any lifetime // ❷ Given an IServiceProvider to resolve other deps // ❸ Called every time a new instance is needed builder.Services.AddTransient<EmailServerSettings>(provider => { var config = provider.GetRequiredService<IConfiguration>(); return new EmailServerSettings( config["Email:Host"], //or directly :Host: "smtp.example.com" int.Parse(config["Email:Port"]) //or directly put the Port: ... ); });
💡
When to use a lambda registration
Use a lambda when:
• The class constructor takes plain values (strings, ints) that must come from config
• You want to run initialization logic before returning the object
• You need to reach into the container (provider.GetRequiredService<T>()) to get other services to build this one

Avoid over-using it — if your class's constructor only depends on registered services, the generic method works fine and is cleaner.
4

The Extension Method Pattern — Keeping Program.cs Clean

As your app grows, Program.cs can become cluttered with dozens of registration calls. The standard convention is to group related registrations into a static extension method on IServiceCollection, then call it from Program.cs with a single Add*() call — exactly the same pattern used by ASP.NET Core framework libraries.

C#EmailServices extension method — keeping registrations organized
// In a separate file, e.g. EmailServiceExtensions.cs public static class EmailServiceExtensions { public static IServiceCollection AddEmailServices( this IServiceCollection services) { services.AddTransient<IEmailSender, EmailSender>(); services.AddTransient<NetworkClient>(); services.AddTransient<MessageFactory>(); services.AddSingleton(new EmailServerSettings("smtp.example.com", 587)); return services; // return services for chaining } } // In Program.cs — one clean call instead of four builder.Services.AddEmailServices();
💡
This is exactly how AddRazorPages(), AddAuthentication() etc. work
The framework libraries you call in Program.cs are all just extension methods on IServiceCollection. They group their internal registrations behind a single clean method. When you create your own feature modules, follow the same pattern: one AddMyFeature() method per cohesive group of services.
5

Multiple Implementations of the Same Interface

One of the advantages of coding to interfaces is that you can have multiple implementations. For example, you might want to notify users via email, SMS, or Facebook — all behind a single IMessageSender interface.

C#Registering multiple implementations of IMessageSender
public interface IMessageSender { void SendMessage(string message); } // Three different implementations public class EmailSender : IMessageSender { ... } public class SmsSender : IMessageSender { ... } public class FacebookSender : IMessageSender { ... } // Register all three under the same interface builder.Services.AddTransient<IMessageSender, EmailSender>(); builder.Services.AddTransient<IMessageSender, SmsSender>(); builder.Services.AddTransient<IMessageSender, FacebookSender>();

To inject all registered implementations, use IEnumerable<T> as the parameter type:

C#Injecting all implementations with IEnumerable<T>
// ❶ Use IEnumerable<IMessageSender> to get ALL three // ❷ Even though the container provides a T[] array, // you MUST declare the parameter as IEnumerable<T>, // not ICollection<T> or T[] app.MapPost("/notify", (NotifyRequest req, IEnumerable<IMessageSender> senders) => { foreach (var sender in senders) sender.SendMessage(req.Message); // sends via email, SMS, and Facebook return Results.Ok(); });
⚠️
When you request a single instance with multiple registrations
If you inject IMessageSender (not IEnumerable<IMessageSender>), the DI container gives you the last registered implementation — in the example above, that would be FacebookSender. The order of Add* calls matters for single-instance resolution.
6

Open Generics Registration

You cannot use the generic Add<T>() methods to register open generic types — types that contain an unresolved generic type parameter, such as IRepository<T>. For those, you must use a special overload that takes Type objects:

C#Registering open generic types
// ❌ This won't compile — you can't use open generics with the generic overload // builder.Services.AddScoped<IRepository<>, DbRepository<>>(); // ✅ Use the Type-based overload for open generics builder.Services.AddScoped( typeof(IRepository<>), typeof(DbRepository<>) ); // Now these all work automatically: // IRepository<Customer> → DbRepository<Customer> // IRepository<Order> → DbRepository<Order> // IRepository<Product> → DbRepository<Product>
💡
When to use open generics
This pattern is common for generic repository patterns and generic pipeline handlers (like MediatR). One registration covers all concrete versions of the generic type — you don't need a separate AddScoped call for each T.

7

TryAdd — Conditional Registration

TryAdd*() methods register a service only if it hasn't already been registered. This is the safe way to provide a default implementation that can be overridden by a consumer who registers their own version first.

⚠️ AddTransient (always adds)
// Adds a NEW registration every time, // even if IEmailSender was already registered. // Result: multiple registrations exist. builder.Services .AddTransient<IEmailSender, SmtpSender>(); builder.Services .AddTransient<IEmailSender, SendGridSender>();
✅ TryAddTransient (skips if exists)
// Registers SmtpSender as the default. // If someone already registered IEmailSender, // TryAdd does nothing — their version wins. builder.Services .TryAddTransient<IEmailSender, SmtpSender>();

💡
When is TryAdd useful?
Framework libraries use TryAdd*() internally when registering default services. This lets you override a default behavior by registering your own implementation before calling the framework's Add*() method. Your registration wins because TryAdd won't overwrite it.
8

Understanding Lifetimes — How Long Do Services Live?

Whenever the DI container is asked for a service, it has a choice: create a brand new instance, or reuse an existing one. The lifetime of a service controls this behavior.

ASP.NET Core has three lifetimes:

🆕
Transient
New instance every time one is requested
🔄
Scoped
Same instance within one request; new instance per request
♾️
Singleton
Same instance for the entire lifetime of the application
🧠 A service is registered as Scoped. Two different classes inside the same HTTP request both need this service. How many instances of the service are created for that request?
9

Transient — Everyone Gets Their Own

With transient, every request for a service creates a new instance — even within the same HTTP request and even if the same service is needed multiple times in the same dependency graph.

The example uses two classes that both depend on DataContext: a Repository that receives it via constructor injection, and a RowCounts handler that also receives it directly:

C# The classes used in this example
public class Repository { private readonly DataContext _dataContext; public Repository(DataContext dataContext) // ❶ { _dataContext = dataContext; // ❶ } public int RowCount => _dataContext.RowCount; // ❷ } // ❶ An instance of DataContext is provided using DI. // ❷ RowCount returns the same value as the current instance of DataContext. app.MapGet("/rowcounts", (Repository repo, DataContext ctx) => Results.Ok(new { repo.RowCount, ctx.RowCount })); // Both repo and ctx depend on DataContext — how many instances are created?
C# Transient registration
builder.Services.AddTransient<DataContext>(); builder.Services.AddTransient<Repository>();
Transient — what happens across two requests:
Request #1
Repository gets → DataContext A
RowCounts gets → DataContext B
Two different instances
Request #2
Repository gets → DataContext C
RowCounts gets → DataContext D
Two more new instances
⚠️
Transient = "new" every time — think about cost
Using transient is equivalent to calling new T() each time. If the service is cheap to create and stateless, transient is fine. If it's expensive (opens DB connections, reads config), use scoped or singleton instead to reuse instances.
10

Scoped — One Per Request

A scoped service creates one instance per scope. In ASP.NET Core, each HTTP request automatically gets its own scope. Within that request, every place that asks for the same scoped service gets the same object. A new request = a new scope = a new instance.

Using the same Repository and RowCounts handler from before — both depend on DataContext. With scoped, they share the same instance within one request:

C# The classes used in this example
public class Repository { private readonly DataContext _dataContext; public Repository(DataContext dataContext) // ❶ { _dataContext = dataContext; // ❶ } public int RowCount => _dataContext.RowCount; // ❷ } // ❶ An instance of DataContext is provided using DI. // ❷ RowCount returns the same value as the current instance of DataContext. app.MapGet("/rowcounts", (Repository repo, DataContext ctx) => Results.Ok(new { repo.RowCount, ctx.RowCount })); // Within one request: repo._dataContext and ctx are THE SAME instance.
C# Scoped registration — ideal for DbContext
builder.Services.AddScoped<DataContext>(); builder.Services.AddScoped<Repository>();
Scoped — one instance shared within a request:
Request #1 (one scope)
Repository gets → DataContext A
RowCounts gets → DataContext A
Same instance — shared
Request #2 (new scope)
Repository gets → DataContext B
RowCounts gets → DataContext B
New instance for the new scope
🗄️
Scoped is ideal for database contexts
Entity Framework Core's DbContext should be registered as scoped. This way, all operations within a single HTTP request share the same context — which means the same in-memory change tracker and the same transaction. A new request gets a fresh context, avoiding cross-request data leaks.
11

Singleton — There Can Be Only One

A singleton service is created once when first requested, and that same instance is returned for every subsequent request — across all scopes and all HTTP requests, for the entire application lifetime.

Same example: Repository and the RowCounts handler both ask for DataContext. With singleton, they always get the exact same object — every request, forever:

C# The classes used in this example
public class Repository { private readonly DataContext _dataContext; public Repository(DataContext dataContext) // ❶ { _dataContext = dataContext; // ❶ } public int RowCount => _dataContext.RowCount; // ❷ } // ❶ An instance of DataContext is provided using DI. // ❷ RowCount returns the same value as the current instance of DataContext. app.MapGet("/rowcounts", (Repository repo, DataContext ctx) => Results.Ok(new { repo.RowCount, ctx.RowCount })); // All requests: repo._dataContext and ctx are always DataContext #A — forever.
C# Singleton registration
builder.Services.AddSingleton<DataContext>(); builder.Services.AddSingleton<Repository>();
Singleton — one instance shared across ALL requests forever:
Request #1
Repository gets → DataContext #A
Handler ctx gets → DataContext #A
Same instance — shared
Request #2
Repository gets → DataContext #A
Handler ctx gets → DataContext #A
Still the exact same #A!
🔒
Singleton services must be thread-safe
In a web application, many requests run concurrently — potentially hundreds of threads using your singleton simultaneously. If your singleton holds mutable state without proper locking, you have a race condition. Use singletons only for stateless services or truly shared, thread-safe state (like configuration, caches with concurrent collections, etc.).
12

Captive Dependencies — The Classic DI Bug

A captive dependency happens when a service with a longer lifetime holds a reference to a service with a shorter lifetime. The shorter-lived service gets "captured" by the longer-lived one and never released or refreshed — causing subtle, hard-to-diagnose bugs.

🚨
The golden rule of lifetimes
A service should only depend on services that have a lifetime equal to or longer than its own.

Singleton → can only use Singleton
Scoped → can use Scoped or Singleton
Transient → can use Transient, Scoped, or Singleton

Here's the classic example: DataContext is registered as Scoped (a new one per request), but Repository is registered as Singleton (one for the entire app). The Singleton Repository captures the DataContext from the very first request, and all future requests reuse that stale, wrong DataContext.

❌ Captive Dependency Bug
// DataContext is scoped — new per request builder.Services.AddScoped<DataContext>(); // Repository is singleton — lives forever // BUT it depends on DataContext (scoped)! // → DataContext gets "captured" inside Repository // → All requests share the SAME stale DataContext builder.Services.AddSingleton<Repository>();
✅ Correct Fix
// DataContext is scoped builder.Services.AddScoped<DataContext>(); // Repository must also be scoped or transient // (same or shorter lifetime than its dependency) builder.Services.AddScoped<Repository>();


ASP.NET Core can detect captive dependencies for you. The ValidateScopes option throws an exception when a singleton tries to consume a scoped service:

C#Enabling scope validation to catch captive dependency bugs
// ValidateScopes: throws an exception at runtime if a singleton // depends on a scoped service (captive dependency) // Enabled by default in Development environment builder.Host.UseDefaultServiceProvider(options => { options.ValidateScopes = true; // ValidateOnBuild: checks at startup that ALL registered services // can actually be constructed — fails fast before any request arrives options.ValidateOnBuild = true; });
💡
ValidateScopes vs ValidateOnBuild
ValidateScopes — detects captive dependencies (short-lived service captured by long-lived one). Throws when the container creates the offending service.

ValidateOnBuild — checks on startup that the container can construct every registered service. Catches missing registrations before your app even starts handling requests. Both are enabled by default in the Development environment.
