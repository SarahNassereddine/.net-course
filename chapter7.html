Chapter 7

Model Binding and Validation in Minimal APIs

How ASP.NET Core automatically extracts values from an HTTP request and turns them into C# handler parameters — and how to validate those values before using them.

1

What is Model Binding?

Model binding extracts values from an incoming HTTP request and uses them to create .NET objects. These objects are then passed as method parameters to your endpoint handler.

This is a one-way population of objects from the request — not the two-way data binding used in desktop or mobile development. ASP.NET Core automatically creates the arguments passed to your handler by examining the request URL, headers, and body.

📋
When does model binding happen?
Model binding happens after routing selects the endpoint but before your handler executes.

The RoutingMiddleware matches the URL and extracts raw string route values.
The EndpointMiddleware then converts those strings to the real C# types your handler needs (e.g. a route value "42" becomes int 42), and also reads the request body and query string.
Request lifecycle — binding's place in the pipeline
① RoutingMiddleware — matches URL, extracts raw string route values
② Filter pipeline runs (if any endpoint filters are registered)
③ Model Binding — converts strings → real types, reads body/query
④ Your endpoint handler executes with fully-populated parameters
🧠 The route template is /products/{id}. A request arrives for GET /products/42. When does the string "42" get converted to int 42 for the handler?
2

Binding Sources

ASP.NET Core can extract handler parameter values from several different parts of the request. These are called binding sources.

🔗
Route values
Values captured from URL segments defined in the route template. /products/{id} → id comes from the URL path.
❓
Query string values
Values appended at the end of the URL after ?. Not used during routing — only during model binding. /products?page=2&size=10
📨
Header values
Values from HTTP request headers. Cannot be bound automatically — requires the explicit [FromHeader] attribute.
📦
JSON body
A single complex type parameter is deserialized from the request body JSON. Minimal APIs only support JSON bodies (not form data) automatically.
💉
Dependency Injected services
Services registered in the DI container are automatically injected as handler parameters. No attribute needed.
🔧
Custom binding
For advanced scenarios, implement TryParse() or BindAsync() on your type to control exactly how binding works.
⚠️
Minimal APIs do NOT auto-bind form data
Unlike MVC controllers and Razor Pages, Minimal APIs only auto-bind JSON request bodies. Form data (application/x-www-form-urlencoded) is not automatically bound. If you need form data in a minimal API handler, access it manually via HttpRequest.Form or use MVC/Razor Pages instead.
3

Binding Simple Types

ASP.NET Core follows a two-step convention for automatically binding simple types (int, string, bool, Guid, etc.) without any attributes:

1
Check the route template first
If the handler parameter name matches a route parameter in the template, bind from the route value.
2
Fall back to the query string
If no matching route parameter exists, look for a query string key with the same name as the handler parameter.
C#Simple type binding — route vs query string
// ① "id" is in the route template → binds from route value // GET /products/42 → id = 42 app.MapGet("/products/{id}", (int id) => $"Product {id}"); // ② "page" and "size" are NOT in the template → bind from query string // GET /products?page=2&size=10 → page = 2, size = 10 app.MapGet("/products", (int page = 1, int size = 20) => $"Page {page}, size {size}"); // ③ Mixed: "id" from route, "verbose" from query string // GET /products/5?verbose=true app.MapGet("/products/{id}", (int id, bool? verbose) => verbose == true ? $"Full details for {id}" : $"Product {id}");
🚫
Binding failure → 400 Bad Request
When the minimal API infrastructure fails to bind a parameter due to an incompatible format (e.g. the URL contains id=abc but the handler expects int id), it throws a BadHttpRequestException and returns a 400 Bad Request response. Your handler never executes.
🧠 Handler: (int id, string? search). Route: /items/{id}. Request: GET /items/7?search=apple. How are the parameters bound?
4

Explicit Binding Attributes

When you need to override the automatic convention and force binding from a specific source, use one of the binding attributes:

AttributeBinds fromNotes
[FromRoute]Route valuesUseful when parameter name differs from route placeholder name
[FromQuery]Query stringUseful to force query string binding even when a route param has the same name
[FromHeader]HTTP request headersRequired for header binding — no automatic convention exists for headers
[FromBody]JSON request bodyUsually inferred automatically for complex types
[FromServices]DI containerUsually inferred automatically; can be made explicit
C#Explicit binding attributes in action
// [FromHeader] — only way to bind from request headers app.MapGet("/secure", ([FromHeader(Name = "X-Api-Key")] string? apiKey) => apiKey == "secret" ? "Authorized" : "Denied"); // [FromRoute] with a different parameter name app.MapGet("/products/{id}", ([FromRoute(Name = "id")] int productId) => $"Product {productId}"); // [FromQuery] — force query string even if route has same name app.MapGet("/search/{id}", (int id, [FromQuery] string? q) => $"Item {id}, query: {q}"); // ⚠️ If [FromRoute] references a parameter that doesn't // exist in the template, you get a runtime exception
⚠️
[FromRoute] requires a matching template placeholder
If you apply [FromRoute(Name = "foo")] but there is no {foo} parameter in the route template, you will get a runtime exception when the app receives its first request — not at startup.
5

Custom Types with TryParse

A simple type is defined as any type that implements one of the following static TryParse methods. Built-in types like int, bool, Guid, and DateTime all implement these, which is why they bind automatically from route values and query strings.

You can create your own types that implement TryParse — ASP.NET Core will treat them as simple types and bind them from a single source (route or query string) automatically.

C#Custom type with TryParse — ProductId example from the slides
// ❶ ProductId is a C# 10 record struct // ❷ Implements TryParse → treated as simple type by minimal APIs // ❸ Binds from route: GET /products/p-42 → ProductId { Value = 42 } app.MapGet("/products/{id}", (ProductId id) => $"Product {id.Value}"); public record struct ProductId(int Value) { // ❹ Must match one of these two signatures: // static bool TryParse(string? value, out T result) // static bool TryParse(string? value, IFormatProvider?, out T result) public static bool TryParse(string? value, out ProductId result) { // ❺ Checks that string starts with 'p' if (value is not null && value[0] == 'p' // ❻ Parses remaining chars as int (skips first char efficiently) && int.TryParse(value.AsSpan(1), out var id)) { // ❼ Parsed successfully → return true with the value result = new ProductId(id); return true; } // ❽ Failed → return false, assign default result = default; return false; } }
💡
Consider implementing IParsable<T> for modern C#
C# 11 introduced the IParsable<T> interface (using static abstract interface members). Implementing it alongside TryParse makes your type compatible with both ASP.NET Core model binding and general parsing APIs.
6

Binding Complex Types from the JSON Body

Complex types (classes, records — anything without a TryParse) are automatically deserialized from the JSON request body. ASP.NET Core uses the built-in System.Text.Json library for deserialization.

C#Complex type binding — POST with JSON body
// POST /product // Body: { "id": 1, "name": "Shoes", "stock": 12 } // ASP.NET Core deserializes the body into a Product instance app.MapPost("/product", (Product product) => { return Results.Created($"/product/{product.Id}", product); }); record Product(int Id, string Name, int Stock);
🚫
Three important rules for body binding
Rule 1 — Only one complex type per handler. If two parameters are both eligible for body binding, you get a runtime exception on the first request.

Rule 2 — Body must be valid JSON. If the body is not JSON (wrong Content-Type), EndpointMiddleware returns 415 Unsupported Media Type.

Rule 3 — Only POST/PUT/PATCH. Trying to bind to the body for GET, HEAD, DELETE, OPTIONS, TRACE, or CONNECT causes a runtime exception — these verbs normally don't carry a body.
🧠 A handler has two complex-type parameters: (OrderRequest order, CustomerInfo customer). What happens when the app receives its first request?
7

Binding Arrays

Arrays of simple types can be bound from the query string by repeating the same key: ?id=1&id=2&id=3. The parameter name in the handler must match the query string key exactly.

C#Array binding from the query string
// GET /products?id=1&id=2&id=3 // int[] id = [1, 2, 3] app.MapGet("/products", (int[] id) => $"IDs: {string.Join(", ", id)}"); // ⚠️ The parameter MUST be named "id" (not "ids") // so it matches the query string key ?id=... // Using "ids" would require ?ids=1&ids=2 in the URL
📋
When does array binding use the query string vs the body?
Arrays bind from the query string only when both conditions are true:
1. The HTTP verb is one that typically has no body (GET, HEAD, DELETE)
2. The array is of a simple type (or string[] or StringValues)

If either condition is false — e.g. a POST request, or an array of complex objects — ASP.NET Core attempts to bind the array from the JSON body instead.
8

Optional & Nullable Parameters

By default, if a required parameter cannot be found in the request, EndpointMiddleware throws an exception and returns a 400 Bad Request. To make a parameter optional, mark it as nullable (int?, string?) or give it a default value.

C#Making parameters optional with nullable and default values
// ① Nullable — parameter is optional, null if absent // GET /stock/123 → id = 123 // GET /stock → id = null app.MapGet("/stock/{id?}", (int? id) => id.HasValue ? $"Stock {id}" : "All stock"); // ② Default value — parameter is optional, uses default if absent // GET /products?page=3 → page = 3 // GET /products → page = 1 (default) app.MapGet("/products", (int page = 1, int pageSize = 20) => $"Page {page} of {pageSize} items"); // ③ Optional string — empty string vs null are different! app.MapGet("/search", (string? q) => q is null ? "No query" : $"Searching: {q}");

10

File Uploads with IFormFile

ASP.NET Core supports file uploads via the IFormFile interface (single file) or IFormFileCollection (multiple files). When your handler declares one of these types, ASP.NET Core reads the multipart form data and populates the parameter automatically.

C#Single and multiple file uploads
// Single file upload app.MapPost("/upload", async (IFormFile file) => { await using var stream = file.OpenReadStream(); // Read file content from stream... return Results.Ok(new { file.FileName, file.ContentType, file.Length // size in bytes }); }); // Multiple files app.MapPost("/upload-many", async (IFormFileCollection files) => { foreach (var file in files) { // Process each file... } return Results.Ok($"{files.Count} files received"); });
⚠️
IFormFile buffers the entire file in memory first
When your handler accepts IFormFile, ASP.NET Core buffers the entire file in memory and on disk before your handler runs. This is fine for small files. For large files (videos, archives), buffering everything may exhaust memory — you should stream the file directly instead, though that is significantly more complex to implement.

Custom Binding with BindAsync

For advanced scenarios where TryParse is not enough (e.g. you need access to HttpContext or need to do async work), implement a static BindAsync method on your type.

C#BindAsync signatures and usage
// Two valid signatures for BindAsync — implement either: // static ValueTask<T?> BindAsync(HttpContext ctx) // static ValueTask<T?> BindAsync(HttpContext ctx, ParameterInfo param) public class PaginationParams { public int Page { get; init; } public int Size { get; init; } public static ValueTask<PaginationParams?> BindAsync(HttpContext ctx) { // Read from query string manually — can access full HttpContext int.TryParse(ctx.Request.Query["page"], out var page); int.TryParse(ctx.Request.Query["size"], out var size); return ValueTask.FromResult<PaginationParams?>( new PaginationParams { Page = page > 0 ? page : 1, Size = size > 0 ? size : 20 }); } } // Usage — PaginationParams binds automatically via BindAsync app.MapGet("/products", (PaginationParams pagination) => $"Page {pagination.Page}, size {pagination.Size}");
12

Binding Source Priority

When ASP.NET Core needs to bind a parameter, it follows a strict priority order. The first matching source wins — it does not try multiple sources for the same parameter.

1
Explicit attribute
[FromRoute], [FromQuery], [FromBody], [FromHeader], [FromServices] — if present, this always wins.
2
Well-known types
HttpContext, HttpRequest, Stream, IFormFile, etc. — hardcoded in the framework.
3
BindAsync
If the type implements a static BindAsync() method, call it.
4
Simple type / TryParse
If the type is a string or has TryParse: bind from route value if name matches a template parameter, otherwise from query string.
5
Array of simple types (on GET/HEAD/DELETE)
If the request has no body, bind arrays from the query string.
6
DI service
If the type is registered in the DI container, inject it from the container.
7
JSON body (last resort)
If nothing else matched, deserialize from the JSON request body. If binding fails and the parameter is not optional, return 400 Bad Request.
🧠 A handler parameter is [FromQuery] int id. The route template is /items/{id}. The URL is /items/5?id=99. What value does id receive?

[AsParameters] — Grouping Parameters

When a handler has many parameters, the signature gets long. [AsParameters] lets you group related parameters into a single struct or class without forcing them to come from the JSON body.

❌ Without [AsParameters]
// Long, hard-to-read signature // Without [AsParameters], if you group // params in a class it binds from BODY app.MapGet("/search", (string? q, int page, int size, string? sort, bool desc) => $"q={q} p={page}");
✅ With [AsParameters]
// Clean handler + grouped params // Each property still binds from its // natural source (route / query) app.MapGet("/search", ([AsParameters] SearchParams p) => $"q={p.Q} page={p.Page}"); record SearchParams( string? Q, int Page = 1, int Size = 20);
📋
Without [AsParameters], a class parameter binds from the body
If you pass a custom class as a handler parameter without [AsParameters], ASP.NET Core treats it as a complex type and tries to deserialize it from the JSON body. [AsParameters] tells the framework to instead bind each property of the class individually using the normal binding rules (route, query string, etc.).
14

Why Validate User Input?

You should validate your endpoint handler parameters before using them to touch your domain logic, database, or infrastructure — or before doing anything that could leak information to an attacker.

What validation catches
📧
Incorrect format
Email fields must look like email addresses. Phone numbers must be valid telephone formats.
🔢
Out-of-range values
You can't buy -1 copies of a book. A month value can't be 13. A page number can't be 0.
✅
Required vs optional
Some fields are required (name on a profile), others are optional (phone number). Missing required fields must be rejected.
🏢
Business rules
You can't convert a currency to itself. A product SKU must follow a specific pattern. These rules are specific to your domain.

DataAnnotations Validation Attributes

DataAnnotations attributes let you declare validation rules directly on your model's properties using standard attributes from System.ComponentModel.DataAnnotations.

C#UserModel with DataAnnotations from the slides
using System.ComponentModel.DataAnnotations; // ❶ Required using public class UserModel { [Required] // ❸ Must not be null/empty [StringLength(100)] // ❹ Max 100 characters [Display(Name = "Full Name")] // ❺ Display name for error messages public string? Name { get; set; } [Required] [EmailAddress] // ❻ Must look like an email public string? Email { get; set; } [Phone] // ❼ Valid phone number format public string? PhoneNumber { get; set; } } // ❷ The API takes a UserModel and binds it from the JSON body app.MapPost("/user", (UserModel user) => Results.Ok(user));

Here is a quick reference of the most commonly used DataAnnotations attributes:

AttributeValidates
[Required]Property must not be null
[StringLength(max)]String has at most max characters
[MinLength(min)]Collection / string has at least min items
[Range(min, max)]Numeric value is between min and max
[EmailAddress]Value has a valid email address format
[Phone]Value has a valid phone number format
[CreditCard]Value has a valid credit card format
[Url]Value has a valid URL format
[RegularExpression(regex)]Value matches the given regex pattern
[Compare("OtherProp")]Value equals another property (e.g. Email == ConfirmEmail)
16

IValidatableObject — Cross-Property Rules

DataAnnotation attributes validate individual properties in isolation. When you need to validate rules that depend on multiple properties together (e.g. "start date must be before end date"), implement IValidatableObject.

C#IValidatableObject for cross-property validation
public class DateRangeModel : IValidatableObject { [Required] public DateTime StartDate { get; set; } [Required] public DateTime EndDate { get; set; } // Called automatically after all [Required] / [Range] etc. pass public IEnumerable<ValidationResult> Validate( ValidationContext validationContext) { if (EndDate <= StartDate) { // yield return produces validation errors lazily yield return new ValidationResult( "EndDate must be after StartDate", new[] { nameof(EndDate) }); } } }
⚠️
IValidatableObject.Validate() only runs if all attribute validations pass first
If any [Required], [Range], or other attribute validation fails, the Validate() method is never called. This means cross-property rules are only checked when the individual property values are already structurally valid.

For very complex validation, consider the FluentValidation library (https://github.com/JeremySkinner/FluentValidation) which gives you full flexibility including access to app services.
17

Adding a Validation Filter to Minimal APIs

Unlike MVC and Razor Pages, Minimal APIs have no built-in DataAnnotations validation. You must add it explicitly. The easiest way is via the MinimalApis.Extensions NuGet package, which provides a WithParameterValidation() extension method.

bashInstall the package
dotnet add package MinimalApis.Extensions
C#WithParameterValidation() — automatic 400 on invalid models
// ❶ Add .WithParameterValidation() to your endpoint app.MapPost("/user", (UserModel user) => Results.Ok(user)) .WithParameterValidation(); // ← adds validation filter // Request lifecycle with WithParameterValidation(): // 1. Model binding runs → UserModel populated from JSON body // 2. Validation filter → checks DataAnnotations attributes // 3a. If VALID → handler executes → 200 OK // 3b. If INVALID → 400 Problem Details returned // handler NEVER runs // The 400 response body looks like this: // { // "type": "...", // "errors": { // "Email": ["The Email field is not a valid e-mail address."], // "Name": ["The Name field is required."] // } // }
⚠️
WithParameterValidation() only works on complex types with DataAnnotations
The validation filter checks the properties of complex types for [Required], [Range], etc. It does not validate simple types like int id directly. Adding attributes to a handler parameter like ([Range(1, 100)] int id) does not work — the attributes must be on properties of a class or record, not on the handler parameter itself.
