Rendering HTML Using Razor Views

How Razor views work, how to pass data into them, use C# logic inside them, and organize them with layouts, sections, and partial views.

1

Quick Review — Core Concepts

TermWhat It Is
Razor Pages The page-based paradigm in ASP.NET Core — combines routing, model binding, and HTML generation using Razor views
Razor Page A single page/endpoint — two files: .cshtml (Razor view) + .cshtml.cs (PageModel)
PageModel Where most of the action happens — defines binding models, validates them, runs page handlers, and prepares view data
Page Handler Each Razor Page handles a single route but can respond to multiple HTTP verbs (GET, POST). Each handler handles one verb.
Razor View Also called a "Razor template" — used to generate the HTML response in the final stage of a request

Note — Views should be read-only
Views shouldn't call methods on the PageModel. The view should generally only be accessing data that has already been collected and exposed as properties. The PageModel prepares everything in the handler, the view just displays it.

Razor View and Code-Behind

Each Razor Page consists of two files with the same name — a .cshtml view and a .cshtml.cs code-behind (PageModel). They're linked not by filename but by the @model directive at the top of the view.

📄 ToDoItem.cshtml (View)
@* ❶ @page makes this file a Razor Page (enables routing) *@ @page @* ❷ @model links this view to its PageModel *@ @model ToDoItemModel <h1>@Model.Title</h1> @foreach (var task in Model.Tasks) { <p>@task</p> }
⚙️ ToDoItem.cshtml.cs (PageModel)
public class ToDoItemModel : PageModel { public string Title { get; set; } public List<string> Tasks { get; set; } public void OnGet() { Title = "Tasks for today"; Tasks = _db.GetTasks(); // Prepare data, view will read it } }
📋
Note — What @page and @model do
@page — makes the file a Razor Page. Without this directive, the Razor Pages framework will not route requests to the page.

@model — specifies which PageModel is associated with this view. The framework uses it to decide which class to instantiate, bind properties on, and run handlers from.

A directive is a statement in a Razor file that changes the way the template is parsed or compiled. Another common directive is @using newNamespace.
💡
@model vs Model — case matters!
@model (lowercase m) is the directive at the top of the view that declares the type. Model (uppercase M) is the property you use anywhere in the view to access the PageModel instance. E.g.: <h1>@Model.Title</h1>.

Introducing Razor Templates

A Razor template is a mix of HTML and C#. The Razor parser compiles it into a C# class — it's not interpreted at runtime line by line, it's compiled, just like any other C# code.

RazorAnatomy of a Razor template
@* Directives at the top *@ @page @model ProductDetailModel @using MyApp.Models @* HTML flows through unchanged *@ <!DOCTYPE html> <html> <body> @* C# inline expression — @ symbol *@ <h1>@Model.ProductName</h1> <p>Price: @Model.Price.ToString("F2")</p> @* C# code block — @{} *@ @{ var isOnSale = Model.Price < 10.00m; var badgeText = isOnSale ? "SALE!" : "Regular price"; } <span>@badgeText</span> </body> </html>

Passing Data to Views

Use the following mechanisms in order of preference:

1
PageModel Properties Preferred
Expose any data to display as public properties on the PageModel. Access via Model.PropertyName in the view. This is the clean, strongly-typed approach.
2
ViewData For layouts
A dictionary of objects with string keys — shared throughout the request. Useful for passing data to layout files (e.g., the page title). Not strongly-typed.
3
TempData PRG pattern
Persists data across a redirect — stored in an encrypted cookie. Deleted after it's read. Commonly used with POST-REDIRECT-GET to pass a success message to the next page.
4
HttpContext
Technically available in both handler and view — but using it to transfer data between them is discouraged. Use PageModel properties instead.
5
@inject Services Use carefully
You can inject a DI service directly into a view with @inject Service myService. Useful in rare cases (e.g., authorization checks in views), but generally the PageModel should call the service and expose the result.
Razor + C#PageModel properties — the preferred approach
// ① The PageModel class — passed to the Razor view when it executes public class ToDoItemModel : PageModel { public List<string> Tasks { get; set; } // ② public properties accessible from the view public string Title { get; set; } public void OnGet(int id) { Title = "Tasks for today"; // ③ building data — normally calls a service/DB Tasks = new List<string> { "Get fuel", "Check oil", "Check tyre pressure" }; } } // In the Razor view: <h1>@Model.Title</h1> // renders: <h1>Tasks for today</h1>
Razor + C#ViewData — passing the page title to a layout
@* In the Razor Page view — set the title in ViewData *@ @{ ViewData["Title"] = "Home Page"; } @* In the layout file — read it *@ <title>@ViewData["Title"] — My App</title> @* TempData — survive a redirect *@ // In the POST handler (before redirect): TempData["Success"] = "Order placed successfully!"; return RedirectToPage("Confirmation"); // In the Confirmation.cshtml view (after redirect): @if (TempData["Success"] is string msg) { <div class="alert">@msg</div> @* TempData is now deleted — gone on next request *@ }
📌 ViewData — two ways to use it (dictionary key OR [ViewData] attribute)
Razor view — setting via dictionary
@{
  ViewData["Title"] = "Home Page";
}
<h2>@ViewData["Title"].</h2>
Set and immediately read — the set makes the key available to the layout too.
PageModel — using [ViewData] attribute
[ViewData] // ① property stored in ViewData dict
public string Title { get; set; }

public void OnGet()
{
  Title = "Home Page"; // ② ViewData["Title"] = "Home Page"
  ViewData["Subtitle"] = "Welcome"; // ③ direct dict
}
❶ Properties marked [ViewData] are automatically stored in ViewData. ❷ Setting the property sets the dict key. ❸ You can also set keys directly.

Using C# in Razor Templates

Razor provides two ways to embed C# into HTML:

🔬 Anatomy of a Razor @ expression
Simple expression — whitespace ends the C# part:
<p>Copyright @DateTime.Now.Year &copy;</p>
HTML — sent to browser unchanged
@ indicates start of C# expression
C# expression — result written to HTML output
Whitespace after DateTime.Now.Year signals the end of the C# expression — the &copy; is back to HTML.
Parenthesized expression — use @() when expression has spaces or operators:
<p>The sum of 1 and 2 is <i>@(1 + 2)</i></p>
The whole C# expression within @() is evaluated and written to the HTML output. Use parentheses whenever the expression contains spaces, operators, or method calls with arguments.
💡 To write a literal @ character in your HTML output, use @@. For example: <p>Email us at info@@myapp.com</p> renders as [email protected].
@ — Inline expression
@* Writes the result of an expression to HTML *@ <h1>@Model.Title</h1> <p>Price: @Model.Price.ToString("F2")</p> <span>@(Model.First + " " + Model.Last)</span> @* Use parentheses for complex expressions *@ <p>Total: @(Model.Qty * Model.Price)</p> @* Literal @ character — use @@ *@ <p>Email us at info@@myapp.com</p>
@{} — Code block
@* Normal C# code — runs during rendering *@ @{ var count = Model.Items.Count; var label = count == 1 ? "item" : "items"; } <p>You have @count @label.</p> @* It's all compiled as real C# *@ @{ Layout = "_Layout"; ViewData["Title"] = "Products"; }
💡
Everything in Razor is compiled C#
A Razor view is compiled into a C# class. All the code inside @{} blocks and @ expressions is real C# — you get full IntelliSense, compile-time checking, and the same performance characteristics as hand-written C# code.

Adding Loops and Conditionals to Razor Templates

Standard C# control flow (foreach, if, switch) works directly in Razor. No special Razor syntax needed — just prefix with @.

Razorforeach, if, else — all standard C#
@* foreach loop — iterate over a list *@ <ul> @foreach (var task in Model.Tasks) { <li>@task.Name — due @task.DueDate.ToShortDateString()</li> } </ul> @* if / else — conditional rendering *@ @if (Model.Tasks.Count == 0) { <p>You have no tasks. Enjoy your day! 🎉</p> } else { <p>You have @Model.Tasks.Count task(s) to complete.</p> } @* for loop with index *@ @for (int i = 0; i < Model.Items.Count; i++) { <p>@(i + 1). @Model.Items[i].Name</p> }
🖥️ How if/foreach controls which HTML gets generated — code vs browser output
PageModel data
Model.IsComplete = false;
Model.Tasks = new List<string> {
  "Get fuel", "Check oil",
  "Check Tyre pressure"
};
Razor markup — if + foreach
@if (Model.IsComplete) {
  <p><strong>Well done!</strong></p>
} else {
  <strong>The following tasks remain:</strong>
  <ul>
    @foreach (var task in Model.Tasks)
    { <li>@task</li> }
  </ul>
}
Only the relevant if/else block renders. The foreach generates one <li> per item.

Rendering Raw HTML — Html.Raw and XSS Safety

By default, Razor HTML-encodes all output — characters like < become &lt;. This protects against Cross-Site Scripting (XSS) attacks. The Html.Raw() helper bypasses this encoding.

✅ Default — HTML Encoded (Safe)
var task = "<strong>Check oil</strong>"; // Razor renders it as literal text: // &lt;strong&gt;Check oil&lt;/strong&gt; <p>@task</p> // Browser displays: <strong>Check oil</strong>
⚠️ Html.Raw — Unencoded (Dangerous)
var task = "<strong>Check oil</strong>"; // Html.Raw skips encoding — renders as HTML: <p>@Html.Raw(task)</p> // Browser displays: Check oil // ❌ NEVER use with user input! // <script>alert('XSS')</script> would execute
🚨
XSS — Never use Html.Raw on user input
Using Html.Raw on user input creates a serious security vulnerability. An attacker could inject <script>alert('stolen: '+document.cookie)</script> as input, which your site would then execute in every visitor's browser. Only use Html.Raw on content you fully trust and control.

Layouts — Shared Markup Across Pages

A layout in Razor is a template that includes common code — headers, footers, nav bars, CSS links — shared across all or most pages. It cannot be rendered directly; it's always rendered in conjunction with a child view.

📋
Note — Why layouts exist
A typical web application has a block-based layout where some blocks are common to every page. The header will likely be identical across your whole application, but the sidebar may differ per section, and the body content differs for every page. Layouts let you define the common structure once.
StructureHow a layout wraps child content
_Layout.cshtml
① Common <head> — CSS links, meta tags, title from ViewData
② Shared header / navbar
③ @RenderBody()
← Child view content inserted here
④ Shared footer
Razor_Layout.cshtml — the layout file
@* Layout files do NOT have @page — they're not routable *@ <!DOCTYPE html> <html> <head> <!-- ❶ ViewData passes the title from the child view --> <title>@ViewData["Title"]</title> <!-- ❷ Common CSS is defined in the layout --> <link rel="stylesheet" href="~/css/site.css" /> </head> <body> <header>My App — shared across all pages</header> <main> <!-- ❸ Child view content rendered here --> @RenderBody() </main> <footer>© 2024 My App</footer> </body> </html>

Conventions to follow with layout files:

  • Prefix layout filenames with an underscore: _Layout.cshtml, _TwoColumn.cshtml
  • Place in Pages/Shared/ so you can refer to them by short name (e.g., "_Layout")
  • Every layout file must call @RenderBody() exactly once
  • ASP.NET Core apps can have multiple layouts, and layouts can reference other layouts

Overriding Parent Layouts Using Sections

Sections let a child view inject content into specific named spots in the layout — beyond just the main body. A common use case is injecting page-specific JavaScript at the bottom of the layout.

📄 Child View — defines a section
@page @model ProductModel <h1>Products</h1> <p>Main content here...</p> @* Define a Scripts section — placed anywhere in the file *@ @section Scripts { <script src="~/js/products.js"></script> } @* Define a Sidebar section *@ @section Sidebar { <aside>Filter by category...</aside> }
🖼️ Layout — renders sections
@* ❷ @RenderBody() — all non-section content *@ @RenderBody() @* ❸ Required section — error if view doesn't define it *@ @RenderSection("Sidebar", required: true) @* ❹ Optional section — silently ignored if not defined *@ @RenderSection("Scripts", required: false)







📐 Sections in practice — defining in child view, rendering in layout
Child view (defining sections)
@{
  Layout = "_TwoColumn";
}
@section Sidebar { ❶
  <p>This is the sidebar content</p> ❶
} ❶
<p>This is the main content</p> ❷
❶ All content inside @section Sidebar {} goes to the Sidebar slot — not the main body.
❷ Any content outside a section block is rendered by @RenderBody().
_TwoColumn.cshtml layout (rendering sections)
@{
  Layout = "_Layout"; ❶
}
<div class="main-content">
  @RenderBody() ❷
</div>
<div class="side-bar">
  @RenderSection("Sidebar", required: true) ❸
</div>
@RenderSection("Scripts", required: false) ❹
❶ This layout is itself nested inside _Layout.cshtml.
❷ Renders all non-section content from the child view.
❸ Required — throws an error if the child view doesn't define this section.
❹ Optional — silently ignored if the child view doesn't define it.

Using Partial Views to Encapsulate Markup

Partial views are reusable chunks of Razor markup — like a component or a helper. They break up large views into smaller, focused pieces. They are rendered using the <partial /> Tag Helper.

📋
Note — What partial views are
Partial views are a bit like Razor Pages without the PageModel and handlers. They are purely about rendering small sections of HTML — no routing, no model binding, no validation, no handlers. They just render.
🧩 Partial views — the partial file vs the page that calls it
_ToDo.cshtml — the partial view
@model ToDoItemViewModel // ❶ has its own @model
<h2>@Model.Title</h2> // ❷ renders model data
<ul>
  @foreach (var task in Model.Tasks)
  { <li>@task</li> }
</ul>
❶ Partial views bind to data in their own Model property — just like a Razor Page view uses its PageModel.
❷ The content of the partial — previously in the full view file, now extracted here.
RecentToDos.cshtml — the calling page
@page // ❶ Razor Pages use @page; partials do NOT
@model RecentToDoListModel

@foreach (var todo in Model.RecentItems)
{
  <partial name="_ToDo" model="todo" /> // ❹
}
❶ This is a Razor Page so it uses @page. Partial views do not use @page.
❷ The PageModel contains the list of recent items to render.
❸ Loops through recent items; each todo is a ToDoItemViewModel.
❹ Uses the <partial> tag helper to render _ToDo.cshtml, passing the model.
RazorUsing a partial view with <partial />
@* Render partial by name — framework searches for _ToDo.cshtml *@ <partial name="_ToDo" model="@item" /> @* Render with absolute path *@ <partial name="~/Pages/Shared/_ProductCard.cshtml" model="@product" /> @* Loop and render a partial for each item *@ @foreach (var item in Model.ToDos) { <partial name="_ToDo" model="@item" /> }

When you use a short name like _ToDo, the framework searches for the file starting from the current page's folder, working up:

Search orderFramework looks for _ToDo.cshtml in these locations
// If the calling page is at: Pages/Agenda/ToDos/RecentToDos.cshtml 1. Pages/Agenda/ToDos/ ← current page's folder (first) 2. Pages/Agenda/ 3. Pages/ 4. Pages/Shared/ ← shared folder (recommended location) 5. Views/Shared/

Importing Common Directives with _ViewImports

The _ViewImports.cshtml file contains directives that are automatically inserted at the top of every Razor view in the same folder and subfolders. Avoids repeating @using statements in every view.

Razor_ViewImports.cshtml — shared across all views
@* Make namespaces available in all views *@ @using MyApp.Models @using MyApp.ViewModels @using Microsoft.AspNetCore.Mvc.RazorPages @* Activate Tag Helpers (required for asp-* attributes) *@ @addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers @* Only directives go here — no C# logic, no @{} blocks *@ @* For logic that runs before every view, use _ViewStart.cshtml *@
⚠️
Only directives — no arbitrary C#
You should only put Razor directives in _ViewImports.cshtml (like @using, @addTagHelper, @model). You can't put arbitrary C# code there. For code that should run before every view (like setting the Layout), use _ViewStart.cshtml instead.
📁 _ViewImports.cshtml applies to all views in its folder and subfolders
Example _ViewImports.cshtml
@using WebApplication1 ❶
@using WebApplication1.Pages ❶
@using WebApplication1.Models ❷
@addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers ❸
❶ Default namespace of the app and Pages folder.
❷ Add this to avoid repeating it in every view that uses models.
❸ Makes Tag Helpers available (e.g. asp-for, asp-page) — added by default in new projects.
Folder scope — where it applies
📁 Pages/
📄 _ViewImports.cshtml ← applies here…
📄 Index.cshtml ✓ inherits
📄 Privacy.cshtml ✓ inherits
📁 Shared/
📄 _Layout.cshtml ✓ inherits
📁 Products/
📄 Index.cshtml ✓ inherits
📄 _ViewImports.cshtml ← can override/add more
Place _ViewImports.cshtml in Pages/ to apply to all views. You can add a more specific one in a subfolder to add or override directives for just that section.

Running Code for Every View with _ViewStart

The _ViewStart.cshtml file can contain any Razor code, and it executes before each view in the same folder (and subfolders) renders. Its most common use is setting the default layout for all pages.

Razor_ViewStart.cshtml — set the default layout for all pages
@* Runs before every Razor Page view *@ @{ // All pages will use _Layout.cshtml by default Layout = "_Layout"; } // Individual views can override this: @{ Layout = "_TwoColumnLayout"; // override for this page only } // Or disable the layout entirely: @{ Layout = null; // render without any layout }
📋 _ViewImports.cshtml
What: Razor directives only (@using, @addTagHelper)
When: Applied as if added to top of every view
Runs for: Views, layouts, partial views
⚡ _ViewStart.cshtml
What: Any Razor code (most commonly sets Layout)
When: Runs before each view renders
Runs for: Razor Page views only — NOT layouts or partials
⚡ What _ViewStart.cshtml does — and what it doesn't run for
Execution order for a Razor Page request
1
Page Handler runs (OnGet(), OnPost(), etc.)
2
_ViewStart.cshtml executes — sets Layout = "_Layout"
3
The Razor Page view renders — can override Layout or set it to null
4
The layout renders (wraps the view output)
⚠️ _ViewStart does NOT run for:
❌ Layout files (_Layout.cshtml) — they are rendered after the view, not as a standalone page request
❌ Partial views (_ToDo.cshtml) — they have no request of their own, they are rendered inline
✅ Only runs for Razor Page views that are the direct response to a request (files with @page)
💡 Individual views can still override: @{ Layout = "_TwoColumn"; } or disable: @{ Layout = null; }

Runtime Compilation of Razor Pages

By default, Razor Pages are compiled at build time — they become C# classes in your application. ASP.NET Core also supports runtime compilation, which lets you modify .cshtml files while the app is running without stopping and rebuilding.

✅ Build-time compilation (default)
Behavior: Views compiled when you run dotnet build
Performance: No overhead at runtime
Errors: Caught at build time
Use: Production & CI/CD — always
⚡ Runtime compilation (opt-in)
Behavior: .cshtml changes picked up without rebuild
Performance: Adds overhead — recompiles on change
Errors: Caught only at request time
Use: Local development only — disable in production
C#Enabling runtime compilation — Program.cs + NuGet
// 1. Install the NuGet package: // Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation // 2. Add to Program.cs — only for Development: var builder = WebApplication.CreateBuilder(args); builder.Services .AddRazorPages() .AddRazorRuntimeCompilation(); // ← add this // Or conditionally, only in Development: if (builder.Environment.IsDevelopment()) { builder.Services .AddRazorPages() .AddRazorRuntimeCompilation(); }
⚠️
Avoid runtime compilation in production
Runtime compilation is handy when developing locally — you can edit a view and refresh the browser without rebuilding. However, it adds startup and request overhead and delays error detection. Never enable it in production deployments.

Injecting Services into Views with @inject

Beyond passing data via the PageModel, ASP.NET Core lets you inject services directly into a view using the @inject directive. This gives the view access to a DI-registered service without going through the PageModel.

⚠️
Use sparingly — prefer PageModel properties
Injecting services in views can seem convenient, but it violates the MVC separation of concerns — the view shouldn't be responsible for fetching or computing data. Use @inject only for view-specific helpers (e.g., localization, authorization checks, feature flags), never to replace PageModel data-loading logic.
Razor@inject — inject a service directly into a view
@* Syntax: @inject ServiceType variableName *@ @inject IStringLocalizer<SharedResources> Localizer @inject IAuthorizationService AuthService @* Use it anywhere in the view *@ <h1>@Localizer["WelcomeTitle"]</h1> @{ var canEdit = await AuthService.AuthorizeAsync( User, "EditPolicy"); } @if (canEdit.Succeeded) { <a asp-page="Edit">Edit</a> } @* Common legitimate use: feature flags *@ @inject IFeatureManager Features @if (await Features.IsEnabledAsync("NewUI")) { <partial name="_NewDashboard" /> } else { <partial name="_OldDashboard" /> }
MechanismBest ForAvoid When
PageModel Properties All page-specific data — products, users, counts, anything from a DB —
ViewData Cross-cutting UI data like page titles shared with layouts Strongly typed data — use PageModel instead
@inject View-layer helpers: localization, auth checks, feature flags Loading data from a database or service — do that in PageModel

View Components — Partial Views with Logic

A View Component is like a partial view that can also execute code and call services. It replaces the old "child action" pattern from legacy ASP.NET. Use it when a reusable UI fragment needs its own data-fetching logic — for example, a shopping cart badge, a notification bell, or a user avatar dropdown.

📄 Partial view
Data: Passed in from the parent view/page
Logic: None — display only
Services: Cannot call services
Use: Simple reusable UI fragments
⚙️ View Component
Data: Fetches its own data independently
Logic: Has an InvokeAsync() method
Services: Full DI — can call any service
Use: Self-contained UI widgets with business logic
C# + RazorAnatomy of a View Component
// 1. The component class — in /ViewComponents/ folder public class CartSummaryViewComponent : ViewComponent { private readonly ICartService _cart; public CartSummaryViewComponent(ICartService cart) => _cart = cart; // ← full dependency injection public async Task<IViewComponentResult> InvokeAsync() { var count = await _cart.GetItemCountAsync(HttpContext.User); return View(count); // renders Default.cshtml with count as model } } // 2. The view — at /Pages/Shared/Components/CartSummary/Default.cshtml @model int <span class="cart-badge">🛒 @Model items</span> // 3. Invoke from any Razor Page or layout: @await Component.InvokeAsync("CartSummary") // Or using Tag Helper syntax (cleaner — requires @addTagHelper in _ViewImports): <vc:cart-summary />
💡
View component file location convention
View component views are located at /Pages/Shared/Components/{ComponentName}/Default.cshtml. You can use a different view name by passing it as a string to View("ViewName", model), but Default.cshtml is the convention.

AJAX with Razor — Partial Views and Dynamic Updates

Razor is a server-side rendering (SSR) engine — it generates HTML on the server and sends it to the browser. For highly dynamic pages, you can combine SSR with AJAX: the browser requests just a partial update, and the server returns only the HTML for a section of the page.

ℹ️
AJAX with Razor = partial views + dedicated handlers
To use AJAX with Razor Pages, expose your partial views via dedicated Razor Page handlers (e.g., OnGetProductList() that returns a PartialViewResult). Your JavaScript then fetches that URL and injects the returned HTML into the DOM.
C# + JSAJAX pattern with Razor Pages — handler + fetch
// PageModel — dedicated handler that returns a partial public class ProductsModel : PageModel { public IList<Product> Products { get; set; } public IActionResult OnGetProductList(string category) { Products = _db.GetByCategory(category); return Partial("_ProductList", Products); // returns only the partial } } // _ProductList.cshtml — the partial view @model IList<Product> @foreach (var p in Model) { <div class="product-card">@p.Name — @p.Price.ToString("C")</div> } // JavaScript — fetch updated HTML from the server async function loadCategory(category) { const res = await fetch(`/Products?handler=ProductList&category=${category}`); const html = await res.text(); document.getElementById('product-container').innerHTML = html; }
✅ Good use of AJAX + Razor
Filtering a product list without a full page reload
Loading more items (infinite scroll)
Refreshing a notification count badge
Updating a small widget in a dashboard
❌ Consider a SPA instead
The whole app is highly interactive (think Figma, Gmail)
Near-instant UI updates with complex local state
Offline support required
Team already knows React/Angular/Vue
💡
Consider Blazor for rich interactivity
If you need a highly dynamic UI but want to stay in the .NET ecosystem, Blazor is designed exactly for that. Blazor Server and Blazor WebAssembly offer component-based interactivity without writing JavaScript. AJAX + Razor works well for simpler scenarios, but Blazor is the ASP.NET Core answer to SPAs.

Chapter Summary

Everything covered in Chapter 17 — a complete reference of Razor view concepts, patterns, and decisions.

ConceptKey Point
Razor View Mixes HTML and C# to generate HTML dynamically. Compiled into a C# class at build time. Views should only read PageModel properties — never call methods.
@page / @model @page makes a file a routable Razor Page. @model (lowercase) declares the PageModel type. Model (uppercase) accesses the instance in the view.
Passing Data Use PageModel properties first. Fall back to ViewData for cross-view data (titles, layout). Use TempData to survive one redirect. Use @inject sparingly for view-layer helpers.
@ expressions / @{} blocks @expr writes a value to HTML (no semicolons). @(expr) for expressions with spaces. @{ } runs C# code without output (needs semicolons). Use @@ for a literal @.
HTML Encoding / Html.Raw All @ expressions are HTML-encoded by default (XSS protection). Use Html.Raw() only for trusted HTML — never on user input.
Layouts Extract shared markup (header, footer, CSS links) into a layout. The layout calls @RenderBody() to render the child view. Set via Layout = "_Layout" in the view or in _ViewStart.cshtml.
Sections Allow child views to inject content into specific slots of a layout. Defined with @section Name { }, rendered with @RenderSection("Name", required: bool).
Partial Views Reusable view fragments. No routing, no PageModel. Rendered via <partial name="_Name" model="data" />. Cannot run _ViewStart.cshtml.
View Components Like a partial view with its own logic and DI. Use for self-contained UI widgets that need data from services (cart, notifications). Replaces the old "child action" pattern.
_ViewImports Adds directives (@using, @addTagHelper) to all views in the folder and subfolders. Directives only — no arbitrary C#.
_ViewStart Runs arbitrary C# before each Razor Page view (not layouts/partials). Most common use: set default layout. Individual views can override it.
Runtime Compilation Opt-in feature for local development. Recompiles .cshtml files on-the-fly without a rebuild. Must NOT be enabled in production.
AJAX + Razor Combine partial views with named page handlers (OnGetName() returning Partial(...)) and JavaScript fetch(). For heavy interactivity, consider Blazor instead.
