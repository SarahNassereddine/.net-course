Chapter 13

Creating a Website with Razor Pages

Learn how to build server-rendered, page-based web applications using Razor Pages in ASP.NET Core. Understand the MVC pattern and how requests flow through Razor Pages applications.

1

Your First Razor Pages Application

So far in this course you've built one type of ASP.NET Core application: minimal API apps that return JSON. In this chapter, you'll learn how to build server-rendered, page-based applications using Razor Pages.

Types of ASP.NET Core Applications

Most ASP.NET Core apps fall into one of three categories:

ASP.NET Core Application Types
🔌
API for Machine Consumption
Returns JSON / XML to backend processes, mobile apps, SPAs
🌐
HTML Web App (Razor Pages)
Handles form requests, generates HTML directly for users ← this chapter
🔄
Both HTML and API
Serves both needs, shares logic across different clients

In this chapter we'll focus on the second option: creating server-side rendered HTML pages using Razor Pages.

📄
Server-Rendered vs. Client-Side Frameworks
With server-rendering, your application generates the HTML on the server and the browser displays this HTML directly. No extra client-side framework is required. You can still add dynamic client-side behavior using JavaScript, but fundamentally each page in your app is a standalone request and response, creating a simpler developer experience.

WebApplicationBuilder builder = WebApplication.CreateBuilder(args);

builder.Services.AddRazorPages();  // Register Razor Pages services

WebApplication app = builder.Build();

// These two are only added outside Development:
if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/Error");  // friendly error page
    app.UseHsts();                      // security headers
}

app.UseHttpsRedirection();  // ensure HTTPS only
app.UseStaticFiles();       // serve files from wwwroot
app.UseRouting();           // select the right endpoint
app.UseAuthorization();     // check if endpoint is allowed to run

app.MapRazorPages();  // register Razor Page endpoints

app.Run();

Middleware Pipeline Overview

Middleware Pipeline — Request ↓ / Response ↑
HTTP Request
🌐
Only added when not in Development environment:
ExceptionHandlerMiddleware
Catches exceptions from later middleware → friendly error page
HstsMiddleware
Adds security headers (HSTS) in line with industry best practices
Always added:
HttpsRedirectionMiddleware
Redirects HTTP requests to HTTPS — ensures secure connections only
StaticFileMiddleware
Serves .css, .js, images from wwwroot/ — short-circuits here if file found
RoutingMiddleware
Selects which endpoint to invoke for the request. Runs after StaticFiles so static files are not routed
AuthorizationMiddleware
Controls whether the selected endpoint is allowed to run for the requesting user
EndpointMiddleware
Auto-added at the end of the pipeline. Executes the selected Razor Page handler to generate the response
🌐
HTML Response
🔄
Request Flow
As with minimal APIs, the RoutingMiddleware selects which endpoint handler to execute, and the EndpointMiddleware executes the handler to generate a response. The request flows through all middleware in order, and the response flows back through them in reverse.

Generating HTML with Razor Pages

Razor Pages are stored in .cshtml files (a hybrid of .cs and .xhtml) within the Pages folder of your project. The routing middleware maps request URL paths to a Razor Page by looking in the Pages folder for a Razor Page with the same path.

URL Path → Razor Page File Mapping
GET /
Pages/Index.cshtml
GET /Index
Pages/Index.cshtml
GET /Privacy
Pages/Privacy.cshtml
GET /Admin/Users
Pages/Admin/Users.cshtml
Routing is based on file path within the Pages folder — no explicit route registration needed

Anatomy of a Razor Page

Let's look at the Privacy.cshtml Razor Page:

RazorPages/Privacy.cshtml
@page                          // ← marks this file as a Razor Page (MUST be first line)
@model PrivacyModel            // ← links to the PageModel class
@{
    ViewData["Title"] = "Privacy Policy";   // ← C# code block (no output)
}
&lt;h1&gt;@ViewData["Title"]&lt;/h1&gt;   // ← @ writes C# value into the HTML
<p>Use this page to detail your site's privacy policy.</p>
Razor Page Directives & Syntax
@page
Marks the file as a Razor Page. Must be on the very first line. Without it the page won't be routable.
@model
Links the Razor Page to a specific PageModel class (the code-behind). Enables Model.Property access in the view.
@{ }
A C# code block that executes but writes nothing to the response output (used for setting variables, ViewData, etc.)
@value
Dynamically writes a C# expression/variable into the HTML output. Values are HTML-encoded automatically.
<html>
Static HTML rendered as-is in the response without any transformation
⚠️
Critical: @page Directive
The @page directive must always be placed on the first line of the file, as it tells ASP.NET Core that the .cshtml file is a Razor Page. Without it, you won't be able to view your page correctly.

Layouts

Razor Pages have the concept of layouts, which are base templates that define the common elements of your application, such as headers and footers. The HTML of the layout combines with the Razor Page template to produce the final HTML sent to the browser.

How Layouts Compose the Final HTML
_Layout.cshtml
<html>
<head>…</head>
<header> NAV </header>
@RenderBody()
<footer> … </footer>
</html>
Shared across all pages
+
Privacy.cshtml
@page
@model PrivacyModel
<h1>Privacy</h1>
<p>…</p>
Page-specific content
=
Final HTML
<html>
<header> NAV </header>
<h1>Privacy</h1>
<p>…</p>
<footer> … </footer>
</html>
Sent to the browser
Layouts avoid duplicating the header/footer in every page. Shared folder: Pages/Shared/_Layout.cshtml
✨ Benefits of Layouts
Layouts prevent you from having to duplicate code for the header and footer in every page. If you need to tweak something, you only need to do it in one place. You can find layouts in the Pages/Shared folder of your project.

Handling Request Logic with Page Models and Handlers

The @page directive marks a file as a Razor Page, but most Razor Pages also have an associated page model. By convention, this page model is placed in a code-behind file that has a .cs extension.

Page Model
A class that derives from the PageModel base class and typically contains one or more methods called page handlers that define how to handle requests to the Razor Page.

File Naming Convention

Page File Pairing Convention
📄
Privacy.cshtml
Razor Page — the View
nested under
in Solution Explorer
📄
Privacy.cshtml.cs
Page Model — the Controller

Visual Studio nests these files underneath the Razor Page .cshtml files in Solution Explorer, making the relationship clear.

Example Page Model

C#Pages/Privacy.cshtml.cs
using Microsoft.AspNetCore.Mvc.RazorPages;

namespace WebApplication1.Pages
{
    public class PrivacyModel : PageModel   // ← derives from PageModel
    {
        private readonly ILogger<PrivacyModel> _logger;

        public PrivacyModel(ILogger<PrivacyModel> logger)
        {
            _logger = logger;   // ← injected via constructor DI
        }

        public void OnGet()     // ← page handler for GET requests
        {
            // Handler for GET requests — load data here
        }
    }
}

Page Handlers

Page handlers are methods on the PageModel that handle different HTTP methods:

Page Handler Naming Convention
OnGet() / OnGetAsync()
HTTP GET — called when a user navigates to the page in their browser
OnPost() / OnPostAsync()
HTTP POST — called when a user submits an HTML form
OnPut() / OnPutAsync()
HTTP PUT — update a resource
OnDelete() / OnDeleteAsync()
HTTP DELETE — delete a resource
Pattern: On{HttpVerb}() — ASP.NET Core automatically matches the method by HTTP verb
💡
Page Handler Return Types
Page handlers are typically named by convention, based on the HTTP verb they respond to. They return either void, indicating that the Razor Page's template should be rendered, or an IActionResult that contains other instructions for generating the response, such as redirecting the user to a different page.

Razor Pages are focused on building page-based apps, so you typically want to return HTML rather than JSON or XML. You can also use an IActionResult to return any sort of data, to redirect users to a new page, or to send an error.
📝
Separation of Concerns
You can include C# code in your Razor Pages using @{ }, but generally speaking, you'll want to limit the code in your .cshtml file to presentational concerns only. Complex logic, code to access services such as a database, and data manipulation should be handled in the PageModel instead.

Exploring a Typical Razor Page

Let's look at a more complete example that shows how all the pieces fit together. Consider a product listing page that displays products from a database.

RazorPages/Products.cshtml — the View
@page
@model ProductsModel          // ← binds this view to ProductsModel

&lt;h1&gt;Our Products&lt;/h1&gt;

@if (Model.Products.Count == 0)   // ← access PageModel properties via Model
{
    &lt;p&gt;No products found.&lt;/p&gt;
}
else
{
    &lt;div class="product-grid"&gt;
        @foreach (var product in Model.Products)   // ← iterate over list from PageModel
        {
            &lt;div class="product-card"&gt;
                &lt;h3&gt;@product.Name&lt;/h3&gt;
                &lt;p&gt;@product.Price.ToString("C")&lt;/p&gt;
            &lt;/div&gt;
        }
    &lt;/div&gt;
}
C#Pages/Products.cshtml.cs — the PageModel
public class ProductsModel : PageModel
{
    private readonly IProductService _productService;

    public ProductsModel(IProductService productService)
    {
        _productService = productService;   // ← injected from DI container
    }

    // View model property — data exposed to the Razor view
    public List<Product> Products { get; set; } = new();

    public async Task OnGetAsync()   // ← handles GET /Products
    {
        // Load from service and populate the view model property
        Products = await _productService.GetAllProductsAsync();
    }
}
How a Razor Page Request Works — Step by Step
1
User navigates to /Products — browser sends GET request
2
Routing middleware matches URL → selects Pages/Products.cshtml
3
OnGetAsync() handler is called — it fetches data from the service
4
Products list is stored in the Products property on the PageModel
5
Razor view reads @Model.Products and generates HTML for each product
6
Final HTML is sent back to the browser and displayed to the user

Understanding the MVC Design Pattern

Razor Pages is built on top of the Model-View-Controller (MVC) design pattern. Understanding this pattern helps you understand how Razor Pages applications are structured.

MVC Design Pattern
A design pattern that separates an application into three main components: Model (data and business logic), View (UI presentation), and Controller (handles user input and coordinates between model and view).
The MVC Pattern
📊
Model
Data & business logic. Domain entities, services, repositories. Independent of the UI.
👁️
View
UI presentation. Receives data from the model and formats it into HTML for the user.
🎮
Controller
Handles user input. Coordinates between model and view. Validates input, selects response.
Controller
calls →
Model
returns data →
Controller
passes to →
View

Benefits of MVC Pattern

  • Separation of concerns — UI, business logic, and data access are separate
  • Testability — Each component can be tested independently
  • Maintainability — Changes to one component don't necessarily affect others
  • Reusability — Business logic can be shared across different views

Applying the MVC Design Pattern to Razor Pages

Razor Pages implements the MVC pattern in a slightly different way than traditional MVC applications. Let's see how the three components map:

MVC Components → Razor Pages Mapping
M
Model
→
Application Model (Services, Entities)
Your business logic and domain entities — IProductService, EF Core DbContext, etc. Injected into the PageModel constructor via DI.
V
View
→
The .cshtml Razor Page
Generates HTML by reading data from the PageModel (via the Model property). Focuses purely on presentation — no business logic here.
C
Controller
→
PageModel class + Page Handlers (OnGet, OnPost…)
Coordinates between the application model and the view. Validates input, calls services, populates PageModel properties, and selects the response.

The Role of Page Handlers

Page handlers in the PageModel should perform a limited number of actions:

✓
Validate data
Check that data in the binding model is valid for the request
✓
Invoke application model
Call the appropriate actions on the application model using services
✓
Select response
Choose an appropriate response based on the result from the application model

Building the View Model

When the page handler has called out to the application model (business logic), it's time to generate a response. A view model captures the details necessary for the Razor view to generate a response.

Razor Pages use the PageModel class itself as the view model for the Razor view by exposing the required data as properties. The Razor view uses the data exposed in the PageModel to generate the final HTML response.

C#PageModel as View Model
public class ProductsModel : PageModel
{
    // These public properties serve as the view model —
    // the Razor view reads them via @Model.Products etc.
    public List<Product> Products { get; set; }
    public int TotalProducts { get; set; }
    public string CategoryName { get; set; }

    public async Task OnGetAsync(int categoryId)
    {
        // Load data from the application model (services)
        // and populate the view model properties
        Products = await _productService.GetByCategoryAsync(categoryId);
        TotalProducts = Products.Count;
        CategoryName = await _productService.GetCategoryNameAsync(categoryId);
    }
}

Do Razor Pages Use MVC or MVVM?

You may occasionally see Razor Pages described as using the Model-View-View Model (MVVM) design pattern rather than MVC. It is worth understanding the difference.

MVC vs MVVM — Key Difference
MVC (Razor Pages)
Controller / PageModel
calls ↓
passes to ↓
Model
View
One-directional. The controller drives both the model and the view. No two-way binding.
MVVM (WPF, MAUI)
Model
↕ data flows both ways
ViewModel
⇄ two-way binding
View
Two-way binding. The View and ViewModel synchronize automatically (e.g. WPF data binding).
📌
The Verdict
While the Razor Pages PageModel class superficially resembles a ViewModel, Razor Pages does not use two-way data binding. It is firmly MVC — it is built directly on the ASP.NET Core MVC framework infrastructure. The PageModel fulfils the roles of both Controller and ViewModel within the MVC pattern, but that does not make it MVVM.

Routing a Request and Building a Binding Model

The first step when your app receives a request is routing it to the correct Razor Page handler inside the routing middleware. Once a handler is selected, the endpoint middleware builds the binding model before the handler executes.

Binding Model
One or more objects that act as a container for data provided in a request — data required by a page handler. Values are extracted from the route template, query string, and request body and automatically populated into method parameters or properties marked with [BindProperty] on the PageModel.

Example: /category/Simple

Consider a CategoryModel Razor Page that displays to-do items for a given category. A request to /category/Simple triggers the following sequence:

Request → Binding Model → Handler Execution
🌐
GET /category/Simple
→
Routing Middleware
Maps to
CategoryModel.OnGet
→
Binding Model Built
category = "Simple"
extracted from URL path
→
Handler Executes
OnGet("Simple")
C#CategoryModel — binding model via method parameter
public class CategoryModel : PageModel
{
    private readonly ToDoService _service;

    public CategoryModel(ToDoService service)
    {
        _service = service;   // ← injected via DI
    }

    // `category` is the binding model — auto-populated from the URL path segment
    public ActionResult OnGet(string category)
    {
        Items = _service.GetItemsForCategory(category);
        return Page();   // ← renders the associated Razor view (.cshtml)
    }

    // View model property — read by the Razor view via @Model.Items
    public List<ToDoListModel> Items { get; set; }
}
💡
Binding Model vs Minimal API Parameters
The binding model for Razor Pages is conceptually the same as the parameters you pass into a minimal API endpoint handler. Simple types (string, int) bind from route values or the query string. Complex types bind from the JSON request body or can use [BindProperty] on the PageModel. You will learn more about this in chapter 16.
⚠️
PageModel ≠ The Model in MVC
Naming in ASP.NET Core can be confusing. The PageModel class is not the "Model" in the MVC pattern — that role belongs to your business logic layer (services, domain classes). The PageModel acts as the Controller (through handlers) and the View Model (through its public properties). The actual application model is accessed via injected services.

Building HTML Using the View Model

When the page handler has called out to the application model, it's time to generate a response. A view model captures the details necessary for the view to generate a response.

Definition: View Model
A view model in the MVC pattern is all the data required by the view to render a UI. It's typically some transformation of the data contained in the application model, plus extra information required to render the page (such as the page's title).

View Model in Razor Pages vs MVC

How the View Model is Passed to the View
📦 ASP.NET Core MVC (Controllers)
Controller Action
creates separate object ↓
ViewModel class
passed to ↓
Razor View (.cshtml)
View model is a separate dedicated class passed to View(model)
📄 Razor Pages
PageModel (Controller + ViewModel)
exposes properties ↓
@Model.Products
@Model.CategoryName
read by ↓
Razor View (.cshtml)
The PageModel itself is the view model — properties are exposed directly
⚠️
Important Distinction
Razor Pages use the PageModel class itself as the view model for the Razor view by exposing the required data as properties. This is different from MVC controllers where you typically create a separate view model class.

How the Razor View Uses the View Model

1
Page Handler Sets Properties
The page handler (e.g., OnGet) sets properties on the PageModel: Items = _service.GetItemsForCategory(category);
2
Handler Returns Page()
The handler signals that the Razor view should be rendered by returning Page(). Returning void implicitly does the same.
3
View Accesses PageModel Properties
The Razor view (.cshtml) accesses the PageModel through the Model property: @Model.Items, @Model.CategoryName
4
HTML Response Generated
The view generates HTML using the data and sends it back through the middleware pipeline to the user's browser. The page handler doesn't control what HTML is generated — the view itself decides.

Putting It All Together: A Complete Razor Page Request

Let's walk through a complete request to a Razor Page to see how all the pieces fit together:

Complete Razor Page Request Flow
🌐 Browser — User navigates to /Products
MIDDLEWARE PIPELINE
StaticFiles, HTTPS redirect, HSTS, Exception handler…
RoutingMiddleware → matches /Products → selects Pages/Products.cshtml
AuthorizationMiddleware → checks if endpoint is allowed
② MODEL BINDING
ASP.NET Core reads route values, query string, form data, request body → builds binding model → passes to handler parameters or [BindProperty] properties
③ PAGE HANDLER EXECUTION — OnGetAsync()
Validates the binding model → calls application services → populates PageModel properties (view model)
④ APPLICATION MODEL (Services / DB)
Business logic executes, data is fetched (e.g., from the database), results returned to handler
⑤ RAZOR VIEW EXECUTION — Products.cshtml
Reads PageModel properties via @Model → combines with Layout → generates complete HTML
🌐 Browser — receives & renders complete HTML page
Detailed Step Walk-through
1
User Makes Request
User navigates to /Products in their browser. The request enters the middleware pipeline.
2
Routing Middleware Selects Endpoint
The RoutingMiddleware examines the URL path and maps it to Pages/Products.cshtml. It also extracts any route parameters from the URL.
3
Model Binding
ASP.NET Core creates a binding model by reading data from the request (route values, query string, form data, request body) and binding it to parameters of the page handler method or properties on the PageModel.
4
Page Handler Execution
The appropriate page handler is called (e.g., OnGetAsync() for a GET request). The handler:
• Validates the binding model
• Calls application services to perform business logic
• Populates properties on the PageModel for the view
5
Razor View Execution
The Razor view (.cshtml file) executes, using the data exposed by the PageModel (Model property) to generate HTML. The view combines with any layouts to produce the final HTML.
6
Response Sent to User
The complete HTML is sent back through the middleware pipeline to the user's browser, which renders the page.

Why Separation of Concerns Matters

You might be thinking: "This whole MVC process seems rather convoluted. So many steps to display some HTML! Why not allow the application model to create the view directly?"

The Key Benefits

The key benefit throughout this process is the separation of concerns. Each component has a single, well-defined responsibility:

Separation of Concerns — What Each Layer Is Responsible For
🎨 The View (.cshtml)
✓ Responsible for:
Taking data and generating HTML
✗ Not responsible for:
Business logic, database access, or deciding what data to show
🧠 The Application Model (Services)
✓ Responsible for:
Executing business logic and providing/updating data
✗ Not responsible for:
HTML generation or any UI concerns
🎯 The Controller (Page Handler)
✓ Responsible for:
Validating requests and coordinating between model and view
✗ Not responsible for:
Business logic or HTML generation

Real-World Example: The Dangers of Tight Coupling

⚠️
Story from the Trenches
A few years ago while working on a small web app, the team hadn't decoupled their business logic from HTML generation properly. Initially there were no obvious problems — the code worked, so they shipped it!

A few months later, someone new started working on the app and immediately "helped" by renaming an innocuous spelling error in a class in the business layer.

Unfortunately, the names of those classes had been used to generate their HTML code, so renaming the class caused the whole website to break in users' browsers!

Needless to say, they made a concerted effort to apply the MVC pattern thereafter and ensure proper separation of concerns.

Benefits of Clear Boundaries

  • Easier to update: UI logic changes won't necessarily require modifying business logic classes
  • Less likely to introduce errors: Changes in one area don't affect unrelated areas
  • Better testability: The model remains easily testable without UI dependencies
  • Team collaboration: Different developers can work on view, model, and controller simultaneously
💡
Testing Advantage
UI code is classically hard to test, dependent on the environment. By keeping the model independent of the view, you ensure that the model (which often contains your application's business logic) stays easily testable without any dependencies on UI constructs.
🎯 Chapter Summary
Razor Pages provides a page-based approach for building server-rendered web applications in ASP.NET Core.

Razor Pages are stored in .cshtml files in the Pages folder. The @page directive (must be on the first line) marks a file as a Razor Page.

Page Models are code-behind classes that derive from PageModel and handle request logic through page handlers like OnGet() and OnPost().

The MVC pattern separates applications into Model (business logic), View (UI), and Controller (request handling). In Razor Pages, the PageModel serves as both Controller and View Model. Note: Razor Pages uses MVC — not MVVM — because there is no two-way binding between the view and the PageModel.

Binding models are built automatically from the incoming request (route values, query string, form data). They are passed as method parameters to the page handler or via [BindProperty] properties on the PageModel — exactly like parameters in minimal API endpoint handlers.

Layouts are base templates that define common elements (headers, footers) shared across pages, preventing code duplication.

Request flow: User request → Routing middleware selects page → Binding model built → Page handler execution → Razor view generates HTML → Response sent to browser.
