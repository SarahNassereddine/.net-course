Chapter 4

Handling Requests with the Middleware Pipeline

A deep dive into how ASP.NET Core middleware works — defining it, combining it, handling errors, and serving static files.

1

Defining Middleware

ASP.NET Core middleware comprises C# classes or functions that manage HTTP requests or responses. They are chained together, creating a pipeline. The output of one middleware serves as the input for the next.

📋
Note — Cross-Cutting Concerns
Logging, error handling, and security are classic cross-cutting concerns — required by many parts of your application. Middleware is the mechanism ASP.NET Core uses to handle these concerns in a composable, reusable way.

A piece of middleware can do one of three things:

A
Handle the request completely
Generate an HTTP response and short-circuit the pipeline — don't pass to next middleware. Example: UseWelcomePage().
B
Process the request and pass it on
Inspect or modify the incoming request, then call next() to pass to the next middleware. Example: Authentication middleware.
C
Process the outgoing response
Inspect or modify the response on the way back out. Example: Error handling middleware, CORS headers.
C#Middleware with authentication + authorization
var app = builder.Build(); // Authentication middleware runs first: // → Inspects the request, associates it with a user (or anonymous) app.UseAuthentication(); // Authorization middleware runs second: // → Uses the user identity set by authentication above // → Verifies the user has permission to make this specific request app.UseAuthorization(); // Both are Type B: they inspect the request and call next() // Order is CRITICAL — authorization needs authentication to run first app.MapGet("/secure", [Authorize] () => "Secret data"); app.Run();
🧠 Authentication middleware runs before Authorization middleware. Why does this order matter?
2

The HttpContext Object

For every incoming request, the ASP.NET Core web server (Kestrel) constructs an HttpContext object. Think of it as a storage box dedicated to a single request — it holds everything about that request and its eventual response for as long as the request lives.

📋
Full Request Lifecycle
1. Browser sends an HTTP request.
2. IIS (Reverse Proxy) receives the raw request and checks for security threats.
3. IIS forwards the request to Kestrel.
4. Kestrel builds an HttpContext and runs your C# middleware pipeline.
5. Each middleware layer reads and may update the HttpContext (credentials, routing, posted data, etc.).
6. Your application code sets the response on the HttpContext.
7. Kestrel converts the updated HttpContext into a raw HTTP response.
8. Kestrel returns it to IIS, which forwards the response back to the browser.
Construction by the web server

When Kestrel receives a request from the reverse proxy, it immediately builds an HttpContext and populates it with the raw request details. This object is then threaded through every middleware component in your pipeline.

Contents of HttpContext
HttpContext PropertyWhat it Contains
RequestURL, method (GET/POST), headers, body, query string, form data
ResponseStatus code, headers, body being written
UserThe ClaimsPrincipal representing the current user (set by Auth middleware)
RequestServicesThe DI service provider scoped to this request
ItemsDictionary for middleware to share data within a single request
ConnectionClient IP, port, TLS certificate information
Middleware access to HttpContext
01
Identify the requested page
Routing middleware maps ctx.Request.Path to the matching endpoint handler in your application.
02
Examine user credentials
Authentication middleware reads and sets ctx.User — a ClaimsPrincipal identifying who is making the request.
03
Retrieve posted data
Endpoint handlers read ctx.Request.Body or ctx.Request.Form to access data submitted by the client.
Processing, updating, and returning the response

After your application processes the request, it writes the result back into the same HttpContext via ctx.Response. The updated object travels back through the middleware pipeline in reverse, giving each layer a chance to add headers, compress the body, and more.

1
Application code writes status code, headers, and body into ctx.Response. The object passes back through middleware in reverse order.
2
Kestrel serializes the updated HttpContext into a raw HTTP response.
3
The raw response is sent back to IIS (the reverse proxy), which strips internal headers and forwards it toward the client.
4
IIS delivers the final HTTP response to the user's browser.
C# Accessing HttpContext in a Minimal API endpoint
app.MapGet("/info", (HttpContext ctx) => { var path = ctx.Request.Path; // e.g. "/info" var method = ctx.Request.Method; // "GET" var clientIp = ctx.Connection.RemoteIpAddress; // caller's IP var user = ctx.User.Identity?.Name; // null if not authenticated ctx.Response.StatusCode = 200; return $"Path: {path}, Method: {method}"; });
3

Combining Middleware — A Holding Page

WelcomePageMiddleware is a simple middleware designed for early development. It returns a sample HTML page for every request at any path.

📋
Note
WelcomePageMiddleware is designed to provide a sample HTML page quickly when you're first developing an application. The Welcome-page middleware response: every request to the application, at any path, will return the same Welcome-page response.
C#Using the Welcome Page middleware
var app = builder.Build(); // This middleware intercepts ALL requests and returns a welcome page. // Nothing after it in the pipeline will ever run. // Useful as a holding page while the app is being built. app.UseWelcomePage(); app.Run(); // GET / → Welcome page ✓ // GET /about → Welcome page ✓ (same for all paths) // GET /api/data → Welcome page ✓ (pipeline is short-circuited)
4

Handling Static Files

The UseStaticFiles() middleware serves files from the wwwroot folder directly, without invoking your application logic.

⚠️
Security — wwwroot Only
By default, the wwwroot folder is the only folder ASP.NET Core will serve files from. It doesn't serve files from other folders for security reasons. The wwwroot folder in an ASP.NET Core project is typically deployed as-is to production, including all the files and folders it contains.
C#Serving static files from wwwroot
var app = builder.Build(); // UseStaticFiles checks if the request matches a file in wwwroot. // If yes → serves the file directly and short-circuits the pipeline. // If no → passes the request to the next middleware. app.UseStaticFiles(); app.MapGet("/hello", () => "Hello from API"); app.Run(); // GET /css/site.css → served from wwwroot/css/site.css ✓ // GET /images/logo.png → served from wwwroot/images/logo.png ✓ // GET /hello → passes to MapGet handler ✓ // GET /secret/data.txt → 404 (not in wwwroot)

HTTP Response Status Codes

Every HTTP response contains a status code and, optionally, a reason phrase describing the status. Knowing these is essential for building correct APIs and middleware.

200 OK
201 Created
204 No Content
400 Bad Request
404 Not Found
409 Conflict
500 Server Error
200 OK — The request was handled and processed successfully. The response body contains the requested data.
201 Created — A new resource was successfully created (typically after a POST). The Location header points to the new resource.
204 No Content — Success, but there is nothing to return (e.g., after a DELETE). Response body is empty.
400 Bad Request — The client sent invalid data. The request failed validation — body was malformed, required fields missing, etc.
404 Not Found — The requested resource does not exist at this URL. The server understood the request but found nothing to return.
409 Conflict — The request could not be completed due to a conflict with the current state of the resource — e.g., creating a resource that already exists.
500 Internal Server Error — An unexpected error occurred on the server. The client did nothing wrong — this is a server-side bug.
RangeCategoryDescription
1xxInformationNot often used; provides a general acknowledgment
2xxSuccessThe request was successfully handled and processed
3xxRedirectionBrowser must follow the provided link (e.g., login redirect)
4xxClient ErrorProblem with the request — invalid data or unauthorized
5xxServer ErrorA problem on the server caused the request to fail

A Minimal API Application in the Pipeline

When you use MapGet() and friends, you are not adding middleware. You are defining endpoints. The endpoint middleware (added automatically by WebApplication) is what executes your handlers.

📋
Note — MapGet vs Middleware
It's important to understand that the MapGet method does not add middleware to the pipeline; it defines an endpoint in your application. You can define endpoints anywhere in Program.cs before the call to app.Run(), but they are typically placed after the middleware pipeline definition.

WebApplication automatically adds the EndpointMiddleware to the end of the pipeline. It also automatically adds the developer exception page middleware to the start when running in development.

The routing middleware selects a minimal API endpoint based on the request URL. The endpoint middleware executes the selected endpoint.
💡
The Pipeline is Bidirectional
All examples so far try to handle an incoming request and generate a response, but it's important to remember that the middleware pipeline is bidirectional. Each middleware component gets an opportunity to handle both the incoming request and the outgoing response. The order of middleware is most important for those components that create or modify the outgoing response.
C#Complete minimal API pipeline example
var builder = WebApplication.CreateBuilder(args); var app = builder.Build(); // Middleware added automatically by WebApplication: // [DeveloperExceptionPage] — auto-added in Development // [RoutingMiddleware] — auto-added // Your middleware app.UseHttpsRedirection(); app.UseAuthorization(); // Define endpoints (NOT middleware) app.MapGet("/fruit", () => { return new[] { "Apple", "Banana", "Cherry" }; }); app.MapGet("/fruit/{name}", (string name) => { return new Fruit(name, 10); }); // Middleware added automatically by WebApplication: // [EndpointMiddleware] — auto-added at the end, executes handler app.Run(); record Fruit(string Name, int Stock);

Handling Errors Using Middleware

When an unhandled exception occurs in your application, you don't want to show the raw stack trace to users. Error-handling middleware intercepts these exceptions and returns a clean response.

📋
Note — How Error Middleware Works
Error-handling middleware attempts to address these problems by modifying the response before the app returns it to the user. Typically, error-handling middleware returns either details on the error that occurred (Development) or a generic but friendly HTML page to the user (Production).

Key pattern: Error-handling middleware ignores the incoming request as it arrives in the pipeline. Instead, it inspects the outgoing response, modifying it only when an error has occurred.
🔬 Development
DeveloperExceptionPage
Shows the full stack trace, exception message, source code line, and request details.

Added automatically by WebApplication in Development. Never show in Production!
🏭 Production
ExceptionHandlerMiddleware
Shows a friendly, custom error page that matches your app's look and feel.

Hides stack traces and internal details from end users.
8

Viewing Exceptions in Development: DeveloperExceptionPage

📋
Note
As shown previously, WebApplication automatically adds this middleware to your middleware pipeline when you're running in the Development environment, so you don't need to add it explicitly.
C#DeveloperExceptionPage — auto-added in Development
var app = builder.Build(); // WebApplication automatically adds this when ASPNETCORE_ENVIRONMENT=Development: // app.UseDeveloperExceptionPage(); // // You can also add it explicitly if needed: if (app.Environment.IsDevelopment()) { app.UseDeveloperExceptionPage(); // shows stack trace in browser } else { app.UseExceptionHandler("/Error"); // friendly page in production } // DeveloperExceptionPage catches exceptions ANYWHERE downstream // and shows detailed information — stack trace, source line, request data app.MapGet("/boom", () => { throw new InvalidOperationException("Something went wrong!"); // In Development: browser shows full stack trace // In Production: browser shows /Error page });
9

Handling Exceptions in Production: ExceptionHandlerMiddleware

📋
Note
The custom error page created by ExceptionHandlerMiddleware can have the same look and feel as the rest of the application by reusing elements such as the header and footer. More importantly, you can easily control the error details displayed to users.
C#ExceptionHandlerMiddleware — production error handling
var app = builder.Build(); // For production: redirect to a friendly /error page app.UseExceptionHandler("/error"); app.MapGet("/error", () => Results.Problem("An error occurred.", statusCode: 500)); // Alternative: use a lambda directly app.UseExceptionHandler(errApp => { errApp.Run(async ctx => { ctx.Response.StatusCode = 500; ctx.Response.ContentType = "text/html"; await ctx.Response.WriteAsync("<h1>Oops! Something went wrong.</h1>"); }); });
⚠️
Never expose stack traces in production
The DeveloperExceptionPage shows sensitive information including file paths, connection strings, and internal logic. Always use ExceptionHandlerMiddleware in production to show a safe, friendly error message.
