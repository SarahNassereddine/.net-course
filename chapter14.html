Chapter 14

Mapping URLs to Razor Pages Using Routing

Understand how routing works in Razor Pages, learn to customize route templates, and discover how to generate URLs dynamically for navigation in your applications.

1

Routing in ASP.NET Core

Routing in ASP.NET Core uses the same infrastructure and middleware whether you're building minimal APIs, Razor Pages, or MVC controllers. However, there are some differences in how you define the mapping between your route templates and your handlers in each case.

Routing Fundamentals
🎯 Core Concept
Routing maps incoming request URLs to specific endpoints that execute code. It's the bridge between a URL like /products/42 and the handler that processes it.
🔄 Shared Infrastructure
The same RoutingMiddleware and EndpointMiddleware work across all application types, providing consistency and predictability.
💡
What You Already Know
In Chapter 6, you learned about routing for minimal APIs. The same core concepts apply to Razor Pages—route templates, parameters, constraints—but with conventions that make Razor Pages even easier to work with.
2

Convention-Based vs. Explicit Routing

Routing is a key part of ASP.NET Core, as it maps the incoming request's URL to a specific endpoint to execute. There are two ways to define these URL-endpoint mappings in your application:

Routing Type Description Used By
Convention-Based Defined globally for your application with patterns that all endpoints must follow MVC controllers (traditional approach)
Explicit (Attribute) Routing Each endpoint is directly mapped with its own route template Minimal APIs, Razor Pages, Modern MVC

Convention-Based Routing

Convention-based routing is defined globally for your application. You can use it to map endpoints (MVC controller actions specifically) to URLs, but those MVC controllers must adhere strictly to the conventions you define.

Explicit Routing (Attribute Routing)

With explicit routing, you tie a given URL to a specific endpoint directly. You've seen this approach with minimal APIs, where each endpoint is directly associated with a route template. You can also use explicit routing with MVC controllers by placing [Route] attributes on the action methods themselves.

✨ Razor Pages Approach
Razor Pages use explicit routing by creating route templates based on conventions. This gives you the best of both worlds: the simplicity of conventions with the flexibility of explicit routes.
🧠 Which routing type does Razor Pages use?
3

Routing Requests to Razor Pages

For every Razor Page in your application, the framework uses the path of the Razor Page file relative to the Razor Pages root directory (Pages/), excluding the file extension (.cshtml), to create a route template.

File Path → Route Template
Pages/Index.cshtml → /
Pages/Privacy.cshtml → /Privacy
Pages/Products/View.cshtml → /Products/View
Pages/Admin/Users/List.cshtml → /Admin/Users/List
⚠️
Case Insensitivity
Remember that routing is not case-sensitive. The URLs /Privacy, /privacy, and /PRIVACY all map to the same Razor Page.
📂 Convention Benefits
This file-based routing convention makes it immediately obvious which URL a Razor Page responds to just by looking at the file structure. No need to hunt through routing configuration!
4

Index.cshtml Pages: Special Routing Behavior

By default, each Razor Page creates a single route template based on its file path. However, there's an important exception to this rule: Razor Pages named Index.cshtml.

🔍
Index.cshtml Creates TWO Route Templates
Index.cshtml pages create two route templates: one ending with "/Index" and one without this suffix. This matches the common behavior of index.html files in traditional HTML applications.

Example: Pages/ToDo/Index.cshtml

If you have a Razor Page at the path Pages/ToDo/Index.cshtml, it creates two route templates:

✅ Route Template 1: "ToDo"
URL: https://example.org/ToDo or https://example.org/todo (case-insensitive)
✅ Route Template 2: "ToDo/Index"
URL: https://example.org/ToDo/Index or https://example.org/todo/index

Both URLs execute the same Index.cshtml Razor Page.

Default Template Example

The standard Razor Pages template includes three pages in the Pages directory:

  • Pages/Error.cshtml
  • Pages/Index.cshtml
  • Pages/Privacy.cshtml

This creates a collection of four routes for the application:

Route Templates
"" → Index.cshtml
"Index" → Index.cshtml
"Error" → Error.cshtml
"Privacy" → Privacy.cshtml
⚠️
Watch Out for Overlapping Routes!
Be careful when using Index.cshtml pages. If you add Pages/ToDo/Index.cshtml, you must not add Pages/ToDo.cshtml, as you'll get an exception at runtime when you navigate to /todo because both pages match the route!
5

Customizing Razor Page Route Templates

The route templates for a Razor Page are based on the file path by default, but you're also able to customize or replace the final template for each page. By default, Razor Pages have URLs consisting of a series of literal segments, such as /ToDo/Index.

To customize the Razor Page route template, you update the @page directive at the top of the Razor Page's .cshtml file.

🎨
Two Types of Customization
You can either add segments to the default route (enhancing it) or replace the entire route with a custom template.
6

Adding a Segment to a Razor Page Route Template

To add an extra segment to a Razor Page's route template, add a space followed by the extra route template segment after the @page statement.

RazorAdding a literal segment
@* Pages/Privacy.cshtml *@ @page "Extra" @model PrivacyModel <!-- Default route: /Privacy --> <!-- New route: /Privacy/Extra -->

Adding Route Parameters

The most common reason for customizing a Razor Page's route template is to add a route parameter. You can use all the same routing features you learned about in Chapter 6, including optional parameters, default parameters, and constraints.

RazorAdding route parameters to Razor Pages
@* Pages/Products.cshtml - Single parameter *@ @page "{category}/{name}" @model ProductsModel @* This matches URLs like: /products/bags/white-rucksack → category="bags", name="white-rucksack" /products/shoes/black-size9 → category="shoes", name="black-size9" /Products/phones/iPhoneX → category="phones", name="iPhoneX" *@ @* With optional parameter *@ @page "{id:int?}" @* Matches: /products and /products/42 *@ @* With default value *@ @page "{page:int=1}" @* /products → page=1, /products/5 → page=5 *@ @* With constraint *@ @page "{id:int:min(1)}" @* Only matches positive integers *@
Route Features Available
Feature Syntax Example
Required parameter {name} @page "{id}"
Optional parameter {name?} @page "{id?}"
Default value {name=value} @page "{page=1}"
Type constraint {name:type} @page "{id:int}"
Multiple constraints {name:type:constraint} @page "{id:int:min(1)}"
Catch-all {**name} @page "{**path}"

7

Replacing a Razor Page Route Template Completely

To specify a custom route for a Razor Page, prefix the route with / in the @page directive. This indicates that this is a custom route template, instead of an addition to the default.

RazorReplacing the entire route template
@* Pages/Products/View.cshtml *@ @* Default route would be: /Products/View *@ @page "/{category}/{name}" @model ViewModel @* Route is now: /{category}/{name} The /Products/View part is completely ignored *@ @* Another example: *@ @page "/checkout" @* Completely custom route, ignores file path *@
⚠️
Use with Caution
Replacing the route template completely breaks the intuitive file-path-to-URL convention that makes Razor Pages easy to understand. Use this feature sparingly, and only when you have a good reason.
⚡ When to Replace Routes
Valid use case: You have a page at Pages/Checkout/Payment.cshtml but want it accessible at /checkout for simplicity.

Better approach: Move the page to Pages/Checkout.cshtml instead, which naturally gives you the /checkout route.
8

Generating URLs for Razor Pages

One of the benefits of using convention-based routing in Razor Pages is that your URLs can be somewhat fluid. If you rename a Razor Page, the URL associated with that page also changes automatically.

Example: Renaming Impact
Original: Pages/Cart.cshtml → /Cart
Renamed: Pages/Basket/View.cshtml → /Basket/View

The problem: if you've hardcoded URLs throughout your application, renaming breaks all those links. The solution: generate URLs dynamically using the page's file path as a reference.

✅
Benefits of Generated URLs
• Refactoring-friendly: Rename or move pages without breaking links
• Type-safe: Compiler checks ensure referenced pages exist
• Maintainable: Single source of truth for routing
9

Generating URLs for a Razor Page Using IUrlHelper

The Url property on a Razor Page is an instance of IUrlHelper that allows you to easily generate URLs for your application by referencing other Razor Pages by their file path.
IUrlHelper is a wrapper around the LinkGenerator class you learned about in chapter 6. It adds some shortcuts for generating URLs based on the current request.

RazorUsing Url.Page() in Razor views
@* Generate URL to Index page *@ <a href="@Url.Page("Index")">Home</a> @* Generate URL with route parameter *@ <a href="@Url.Page("Product", new { id = 5 })">View Product</a> @* Results in: /Product/5 *@ @* Generate URL to page in subfolder *@ <a href="@Url.Page("/Admin/Users")">Manage Users</a> @* Generate absolute URL *@ var absoluteUrl = Url.Page("/Privacy", null, null, Request.Scheme); @* Results in: https://yoursite.com/Privacy *@

IUrlHelper Methods

IUrlHelper has several different overloads of the Page() method:

Method Signature Purpose
Page(string pageName) Generate relative URL to a page
Page(string pageName, object values) Generate URL with route parameters
Page(string pageName, string handler) Generate URL to specific page handler
Page(string pageName, object values, string protocol, string host) Generate absolute URL with custom protocol/host
🎯
For MVC Controllers
Similar to Razor Pages, but use the Action() method on IUrlHelper. You provide an MVC controller name and action name instead of a page path:

@Url.Action("Index", "Home")
@Url.Action("Details", "Products", new { id = 5 })
10

Generating URLs for MVC Controllers

If you're using MVC controllers (covered in detail in chapters 19-20), generating URLs is similar to Razor Pages but uses different method names on IUrlHelper.

💡
MVC Controllers vs Razor Pages
For Razor Pages, use Url.Page(). For MVC controllers, use Url.Action(). You can use both methods from either Razor Pages or MVC controllers, allowing cross-navigation.

Using Url.Action() for MVC Controllers

The main difference from Razor Pages is that you provide an action name and controller name instead of a page path:

public class CurrencyController : Controller
{
    [HttpGet("currency/index")]
    public IActionResult Index()
    {
        // Generate URL to the View action in this controller
        var url = Url.Action("View", "Currency", new { code = "USD" });
        
        return Content($"The URL is {url}");
        // Returns: "The URL is /Currency/View/USD"
    }

    [HttpGet("currency/view/{code}")]
    public IActionResult View(string code)
    {
        // This action is the target of the generated URL
        return View();
    }
}

Method Signature Breakdown

Url.Action(actionName, controllerName, routeValues)
actionName
The name of the action method (e.g., "View", "Edit", "Delete")
controllerName
The name of the controller without the "Controller" suffix (e.g., "Currency" for CurrencyController)
routeValues
Anonymous object with additional route parameters (e.g., new { code = "USD" })

Using nameof for Refactor-Safety

💡
Pro Tip: Use nameof()
Instead of using strings for action names, use the C# 6 nameof operator to make your code refactor-safe:

var url = Url.Action(nameof(View), "Currency", new { code = "USD" });

Now if you rename the View method, the compiler will update this reference automatically!

Ambient Values in Same Controller

If you're routing to an action in the same controller, you can use a different overload that omits the controller name:

public class CurrencyController : Controller
{
    public IActionResult Index()
    {
        // Omit controller name when targeting same controller
        var url = Url.Action(nameof(View), new { code = "USD" });
        return Content($"The URL is {url}");
    }

    public IActionResult View(string code)
    {
        return View();
    }
}

The IUrlHelper uses ambient values from the current request and overrides them with any specific values you provide.

⚠️ Definition: Ambient Values
Ambient values are the route values for the current request. They include Controller and Action when called from an MVC controller, and Page when called from a Razor Page. Ambient values can also include additional route values that were set when the action or Razor Page was initially located using routing.

Generating URLs with LinkGenerator

LinkGenerator is a service available through dependency injection that provides various methods for generating URLs for Razor Pages and MVC actions. It's useful when you need to generate URLs outside of a Razor Page context.

C#Using LinkGenerator in services or middleware
public class EmailService { private readonly LinkGenerator _linkGenerator; public EmailService(LinkGenerator linkGenerator) { _linkGenerator = linkGenerator; } public string GenerateConfirmationUrl(HttpContext httpContext, string userId) { // Generate path for Razor Page var path = _linkGenerator.GetPathByPage( httpContext, page: "/Account/Confirm", values: new { userId }); // Generate full URI var uri = _linkGenerator.GetUriByPage( httpContext, page: "/Account/Confirm", values: new { userId }, scheme: "https"); return uri; } }

LinkGenerator Methods

Method Returns Use Case
GetPathByPage() Relative path Generate URL path for a Razor Page
GetUriByPage() Absolute URI Generate full URL for a Razor Page
GetPathByAction() Relative path Generate URL path for MVC action
GetUriByAction() Absolute URI Generate full URL for MVC action
🧠 When would you use LinkGenerator instead of IUrlHelper?
Note
If you need to generate URLs from parts of your application outside the Razor Page or MVC infrastructure, you won’t be able to use the IUrlHelper helper. Instead, you can use the LinkGenerator class
By contrast with IUrlHelper, LinkGenerator requires that you always provide sufficient arguments to uniquely define the URL to generate. This makes it more verbose but also more consistent and has the advantage that it can be used anywhere in your application. This differs from IUrlHelper, which should be used only inside the context of a request.

Customizing Conventions with Razor Pages

Razor Pages is built on a series of conventions designed to reduce boilerplate code. You can customize these conventions to get full control of your application's URLs without manually customizing every Razor Page.

Common Convention Customizations

These are some of the most common changes developers make to Razor Pages routing:

  • Making URLs lowercase
  • Ensuring URLs always end with a trailing slash (/product-details/)
  • Using kebab-case instead of PascalCase (product-details instead of ProductDetails)

1. Lowercase URLs and Trailing Slash

You can configure these using RouteOptions in Program.cs:

WebApplicationBuilder builder = WebApplication.CreateBuilder(args);

builder.Services.AddRazorPages();

builder.Services.Configure<RouteOptions>(options =>
{
    options.LowercaseUrls = true;           // /products instead of /Products
    options.LowercaseQueryStrings = true;   // ?page=1 instead of ?Page=1
    options.AppendTrailingSlash = true;     // /products/ instead of /products
});

WebApplication app = builder.Build();
app.MapRazorPages();
app.Run();

2. Kebab-Case URLs

For kebab-case, you need to create a custom parameter transformer. This is more advanced but straightforward:

using System.Text.RegularExpressions;
using Microsoft.AspNetCore.Routing;

public class KebabCaseParameterTransformer : IOutboundParameterTransformer
{
    public string? TransformOutbound(object? value)
    {
        if (value is null) return null;
        
        // Replace PascalCase with kebab-case
        return Regex.Replace(
            value.ToString()!, 
            "([a-z])([A-Z])", 
            "$1-$2"
        ).ToLower();
    }
}

Then register it in Program.cs:

builder.Services.AddRazorPages()
    .AddRazorPagesOptions(options =>
    {
        // Apply kebab-case transformer to all Razor Pages
        options.Conventions.Add(
            new PageRouteTransformerConvention(
                new KebabCaseParameterTransformer()
            )
        );
    });

3. Adding Extra Routes to Specific Pages

You can add an alternative route template to a single Razor Page using AddPageRoute():

builder.Services.AddRazorPages()
    .AddRazorPagesOptions(options =>
    {
        // Add alternative route to specific page
        options.Conventions.AddPageRoute(
            "/Search/Products/StartSearch",  // Page path
            "/search-products"               // Additional route
        );
    });
⚠️
AddPageRoute vs @page Directive
Unlike customizing the route template using the @page directive (which replaces the default), AddPageRoute() adds an extra route template to the page. The default route template still works.

4. Custom Page Route Conventions

For even more control, you can implement IPageRouteModelConvention to customize how route templates are generated for all pages:

public class PrefixingPageRouteModelConvention : IPageRouteModelConvention
{
    public void Apply(PageRouteModel model)
    {
        // Create new selectors with "page/" prefix
        var selectors = model.Selectors
            .Select(selector => new SelectorModel
            {
                AttributeRouteModel = new AttributeRouteModel
                {
                    Template = AttributeRouteModel.CombineTemplates(
                        "page",  // Prefix
                        selector.AttributeRouteModel!.Template
                    ),
                }
            })
            .ToList();

        // Add new selectors to the page
        foreach(var newSelector in selectors)
        {
            model.Selectors.Add(newSelector);
        }
    }
}

Register it in Program.cs:

builder.Services.AddRazorPages()
    .AddRazorPagesOptions(options =>
    {
        options.Conventions.Add(new PrefixingPageRouteModelConvention());
    });

Available Convention Interfaces

Razor Pages provides three powerful interfaces for customization:

IPageRouteModelConvention
Used to customize the route templates for all Razor Pages in your app. Add prefixes, change patterns, etc.
IPageApplicationModelConvention
Used to customize how the Razor Page is processed, such as to add filters to your Razor Pages automatically (covered in chapters 21-22).
IPageHandlerModelConvention
Used to customize how page handlers are discovered and selected for Razor Pages.
📚
Learn More
For more details on customizing conventions, see Microsoft's "Razor Pages route and app conventions in ASP.NET Core" documentation at http://mng.bz/A0BK

Best Practices and Advice

Based on experience with Razor Pages routing, here are some recommendations to keep your applications maintainable:

✅ DO: Add route parameters to @page directive
Use @page "{name}" or @page "{id:int}" to make your routes dynamic. This is a great use of the @page directive customization.
✅ DO: Consider global conventions for route templates
When you want to change route templates for all your Razor Pages (like using kebab-case), use global conventions rather than customizing each page individually.
❌ AVOID: Replacing the route template with absolute path
Using @page "/custom-route" breaks the file-path convention and makes your app harder to navigate. Restructure your folders instead.
❌ AVOID: Adding literal segments to @page directive
Don't use @page "Extra/Segment" to add literal path segments. Rely on the file hierarchy instead—move the page to the appropriate folder.
❌ AVOID: Multiple route templates per page
Don't use AddPageRoute() convention to add additional URLs to a page. Having multiple URLs to access the same page is confusing and bad for SEO.
🎯 Chapter Summary
Routing in Razor Pages uses the same infrastructure as minimal APIs and MVC, but with convention-based explicit routing that maps file paths to URLs automatically.

File path determines URL: Pages/Products/View.cshtml → /Products/View. Routing is case-insensitive.

Customize routes with the @page directive: add parameters with @page "{id}", or completely replace with @page "/custom".

All routing features work: optional parameters ({id?}), defaults ({page=1}), constraints ({id:int}), and catch-all ({**path}).

Generate URLs dynamically using Url.Page() in views or LinkGenerator in services to maintain refactoring-friendly code.

Best practice: Rely on file hierarchy for URL structure, add route parameters when needed, avoid replacing routes completely.
