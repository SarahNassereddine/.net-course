Chapter 15

Generating Responses with Page Handlers in Razor Pages

Learn how page handlers work in Razor Pages, how to accept parameters, return different response types, and handle HTTP status codes effectively for a better user experience.

1

Razor Pages and Page Handlers

For Razor Pages, the entry point is the page handler that resides in a Razor Page's PageModel. A page handler is a method that runs in response to a request.

Page Handler Responsibilities
1️⃣ Validate the Request
Confirm that the incoming request is valid and contains the necessary data.
2️⃣ Invoke Business Logic
Call the appropriate business logic corresponding to the incoming request.
3️⃣ Choose the Response
Select the appropriate kind of response to return based on the result.

What Page Handlers Typically Return

Page handlers typically return one of three things:

Return Type Behavior
PageResult Causes the associated Razor view to generate an HTML response
void / Task Same as PageResult—causes the Razor view to generate an HTML response
RedirectToPageResult Indicates that the user should be redirected to a different page
🎯
The Handler Doesn't Generate the Response — It Selects It
It's important to realize that a page handler doesn't generate a response directly; it selects the type of response and prepares the data for it. For example, returning a PageResult doesn't generate any HTML at that point — it merely indicates that a view should be rendered. It's the view engine that generates the actual response.
⚠️
Keep Business Logic Out of Page Handlers
Page handlers generally shouldn't perform business logic directly. If a page handler receives a request to add a product to a cart, it shouldn't manipulate the database or recalculate cart totals directly. Instead, it should call another class (a service) to handle the details. This separation of concerns ensures your code stays testable and maintainable as it grows.
2

Selecting a Page Handler to Invoke

When EndpointMiddleware executes a selected Razor Page, it selects a page handler to invoke based on two variables:

1. HTTP Verb
The HTTP method used in the request: GET, POST, PUT, DELETE, etc.
2. Handler Route Value
The value of the handler route value that comes from:
  • A query string value: /Search?handler=CustomSearch
  • A route parameter: @page "{handler?}"

Page Handler Naming Convention

Page handlers follow a strict naming pattern: On{Verb}{Handler}

C#Page handler naming patterns
public class SearchModel : PageModel { // Standard GET handler - matches GET requests with no handler value public void OnGet() { } // Standard POST handler - matches POST requests with no handler value public void OnPost() { } // Named GET handler - matches GET /Search?handler=CustomSearch public void OnGetCustomSearch() { } // Named POST handler - matches POST /Search?handler=Async public async Task OnPostAsync() { } // Named DELETE handler - matches DELETE /Search?handler=Item public void OnDeleteItem() { } }
Request → Handler Mapping
Request Handler Invoked
GET /Search OnGet()
POST /Search OnPost()
GET /Search?handler=CustomSearch OnGetCustomSearch()
POST /Search?handler=Async OnPostAsync()
DELETE /Search?handler=Item OnDeleteItem()

Implicit Page Handlers

What happens if you get a request that doesn't match any handlers? The EndpointMiddleware executes an implicit page handler instead.

ℹ️ Implicit Handlers
Implicit page handlers contain no logic—they simply render the Razor view. Think of them as:
public void OnDelete() { }

This allows the page to render even when no matching handler exists.
💡
Special Case: HEAD Requests
There's one exception to the implicit page handler rule: if a request uses the HEAD verb and there is no corresponding OnHead handler, the EndpointMiddleware executes the OnGet handler instead (if it exists). HEAD requests are typically sent automatically by the browser and don't return a response body — they're often used for caching and security purposes.

3

Accepting Parameters to Page Handlers

ASP.NET Core can bind to two different targets in Razor Pages:

1. Method Arguments
If a page handler has method parameters, the arguments are bound and created from values in the request.
2. Properties with [BindProperty]
Any properties on the PageModel marked with [BindProperty] are bound to the request.
Important: By default, this attribute does nothing for GET requests.
To bind properties for GET requests too, use: [BindProperty(SupportsGet = true)]
C#Binding to page handler parameters
public class ProductModel : PageModel { // Method parameter binding - works for all HTTP verbs public IActionResult OnGet(int id) { // id bound from route, query string, etc. return Page(); } // Property binding - only for POST by default [BindProperty] public ProductDto Product { get; set; } public IActionResult OnPost() { // Product property is bound from form data if (!ModelState.IsValid) return Page(); // Save Product... return RedirectToPage("Success"); } }
C#Property binding for GET requests
public class SearchModel : PageModel { // This property binds for GET requests too [BindProperty(SupportsGet = true)] public string SearchTerm { get; set; } [BindProperty(SupportsGet = true)] public int Page { get; set; } = 1; public void OnGet() { // SearchTerm and Page are already bound from query string // Example: /Search?SearchTerm=laptop&Page=2 var results = _searchService.Search(SearchTerm, Page); } }
⚠️
GET Request Binding
Remember: [BindProperty] does not work for GET requests by default. You must explicitly set SupportsGet = true if you want properties bound on GET requests. This is a security feature to prevent mass assignment attacks.
4

Returning IActionResult Responses

ASP.NET Core has many types of IActionResult that you can return from page handlers. Each represents a different type of HTTP response:

⚠️
IActionResult vs IResult — Don't Confuse Them
Note that the interface type is IActionResult, not IResult. IResult is used in minimal APIs and should generally be avoided in Razor Pages. In .NET 7+, IResult types returned from Razor Pages execute as expected, but they don't have all the same features as IActionResult, so always favor IActionResult in Razor Pages.
Common IActionResult Types
Type Response Status Code
PageResult Generates HTML view for the Razor Page 200 OK
ViewResult Generates HTML view for a given Razor view (MVC) 200 OK
PartialViewResult Renders part of an HTML page (for AJAX) 200 OK
RedirectToPageResult Redirects to another Razor Page 302 Redirect
RedirectResult Redirects to a specified URL 302 Redirect
FileResult Returns a file as the response 200 OK
ContentResult Returns a provided string 200 OK
StatusCodeResult Sends a raw HTTP status code Custom
NotFoundResult Sends a 404 response 404 Not Found

FileResult Variations

FileResult is a base class with several derived types for returning files:

FileContentResult
Returns a byte[] as a file response to the browser
FileStreamResult
Returns the contents of a Stream as a file response
PhysicalFileResult
Returns the contents of a file on disk as a file response
5

PageResult and RedirectToPageResult

In a typical Razor Pages flow, whenever you return HTML you use a PageResult; when you redirect to a new page, you use a RedirectToPageResult.

PageResult — Rendering HTML

A PageResult renders the Razor Page's view and returns it as HTML. This is what happens when you return Page() or simply return void.

C#PageResult Example
public class CheckoutModel : PageModel { public IActionResult OnGet() { // Returns the same page (Checkout.cshtml) return Page(); } }

RedirectToPageResult — Navigating to Another Page

A RedirectToPageResult sends a redirect response (302) telling the browser to navigate to a different page. This is crucial for the Post-Redirect-Get pattern.

Why POST-Redirect-GET Matters
Without redirect: If you hit F5 on the result page, the browser re-submits the POST request → duplicate order!

With redirect: The browser URL is /payment with a GET request. Pressing F5 just refreshes the payment page → no duplicate order. The POST data is gone from browser history.

Complete Example with Both Patterns

C#Common page handler patterns
public class CreateProductModel : PageModel { [BindProperty] public ProductDto Product { get; set; } // GET request - display the form public void OnGet() { // Returning void renders the Razor Page (PageResult) } // POST request - process the form public IActionResult OnPost() { // Validation failed - redisplay the form if (!ModelState.IsValid) { return Page(); // Returns PageResult } // Save the product... _productService.Create(Product); // Success - redirect to avoid duplicate submissions return RedirectToPage("Success"); // Returns RedirectToPageResult } }
✨ Post-Redirect-Get Pattern
The pattern shown above is called Post-Redirect-Get (PRG). After successfully processing a POST request (like creating a product), redirect to another page. This prevents users from accidentally resubmitting the form if they refresh the page.
C#RedirectToPage variants
// Redirect to Index page return RedirectToPage("Index"); // Redirect with route values return RedirectToPage("Details", new { id = product.Id }); // Results in: /Details?id=42 or /Details/42 depending on route // Redirect to page in subfolder return RedirectToPage("/Admin/Dashboard"); // Redirect with fragment return RedirectToPage("Details", new { id = 5 }, "reviews"); // Results in: /Details/5#reviews
💡
Tip: Helper Methods on PageModel
Most IActionResult implementations have a helper method on the base PageModel class. They're typically named after the type, and the result generated is called TypeResult. For example, the Page() method returns a PageResult, RedirectToPage() returns a RedirectToPageResult, and NotFound() returns a NotFoundResult. These are thin wrappers around new TypeResult().
📝
Persisting Data Between Pages — TempData
Razor Pages are generally designed to be stateless. If you want to persist data between multiple pages, you need to place it in a database or similar store. If you only need to store data for a single request (e.g., a success message to show after a redirect), you can use TempData, which stores small amounts of data in cookies for a single request.

NotFoundResult and StatusCodeResult

Razor Pages can return a NotFoundResult when a requested resource doesn't exist, which returns a raw 404 HTTP status code. You could achieve a similar result using StatusCodeResult and setting the status code explicitly.

C#Returning 404 Not Found
public class ProductModel : PageModel { public Product Product { get; set; } public IActionResult OnGet(int id) { Product = _productService.GetById(id); if (Product == null) { // Resource not found - return 404 return NotFound(); // Returns NotFoundResult } return Page(); } }
C#Using StatusCodeResult for other status codes
// Return 404 explicitly return StatusCode(404); // Return 500 Internal Server Error return StatusCode(500); // Return 503 Service Unavailable return StatusCode(503); // Return custom status with content return StatusCode(403, "Access denied");
⚠️
Raw Status Codes in Web Apps
Returning "raw" status codes without additional content is generally OK for API applications. For apps consumed directly by users (like Razor Pages), this can result in a poor user experience—users see a blank page or browser default error page instead of a helpful message.
7

Handling Status Codes with StatusCodePagesMiddleware

Razor Pages applications can return a wide range of HTTP status codes that indicate error states. Microsoft provides StatusCodePagesMiddleware to handle this use case and provide a better user experience.

C#Basic usage of StatusCodePagesMiddleware
var app = builder.Build(); // Basic usage - shows simple text response app.UseStatusCodePages(); app.MapRazorPages(); app.Run();

Re-executing the Pipeline

A more typical approach in production is to re-execute the pipeline when an error is captured, using a similar technique to the ExceptionHandlerMiddleware.

C#Re-executing with a custom error page
var app = builder.Build(); // Re-execute with custom error path // {0} is replaced with the status code number app.UseStatusCodePagesWithReExecute("/{0}"); // Now you can create pages like: // - Pages/404.cshtml for Not Found // - Pages/500.cshtml for Internal Server Error // - Pages/403.cshtml for Forbidden app.MapRazorPages(); app.Run();
How Re-execution Works
1. A page handler returns NotFound() → 404 status code
2. StatusCodePagesMiddleware intercepts the 404 response
3. The path format string "/{0}" becomes "/404"
4. Pipeline is re-executed, routing to Pages/404.cshtml
5. Custom error page is displayed to the user

Redirecting vs. Re-executing

Instead of re-executing the pipeline, you can redirect the browser to the error page:

C#Redirecting to error pages
app.UseStatusCodePagesWithRedirects("/{0}"); // This sends a 302 redirect to the browser // Browser URL changes to /404 // Different from re-execute which keeps original URL
Approach Browser URL Final Status Code Recommended?
Re-execute Stays at original URL Preserves original (404, 500, etc.) ✅ Yes
Redirect Changes to error page URL Becomes 200 OK ⚠️ Use with caution
✅
Recommended: Re-execute
UseStatusCodePagesWithReExecute is generally preferred because:
  • Preserves the original URL in the browser
  • Maintains the correct HTTP status code
  • Better for SEO and debugging
💡
Combining with ExceptionHandlerMiddleware
You can use StatusCodePagesMiddleware in combination with other exception-handling middleware by adding both to the pipeline. StatusCodePagesMiddleware modifies the response only if no response body has been written. So if another component (such as ExceptionHandlerMiddleware) returns a message body along with an error code, it won't be modified by StatusCodePagesMiddleware.
🎯 Chapter Summary
Page handlers are methods on the PageModel that run in response to requests. They validate input, invoke business logic, and choose the response type.

Handler selection is based on HTTP verb and optional handler route value. Naming pattern: On{Verb}{Handler} (e.g., OnGet(), OnPostDelete()).

Parameter binding works on method arguments (all verbs) and [BindProperty] properties (POST by default, or all verbs with SupportsGet = true).

IActionResult types: PageResult (render view), RedirectToPageResult (redirect), NotFoundResult (404), StatusCodeResult (custom status), FileResult (download files).

Post-Redirect-Get pattern: After successful POST, redirect to prevent duplicate submissions on refresh. Return RedirectToPage() after processing.

StatusCodePagesMiddleware provides user-friendly error pages. Use UseStatusCodePagesWithReExecute("/{0}") to display custom error pages while preserving status codes.
