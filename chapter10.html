Chapter 10

Configuring an ASP.NET Core Application

Load settings from JSON files, environment variables, and command-line arguments. Store secrets safely. Use strongly typed options objects. Adapt behavior per hosting environment.

1

Introducing the ASP.NET Core Configuration Model

Configuration is the set of external parameters that control your application's behavior at runtime — without recompiling your code. It's one of the most elegant parts of ASP.NET Core.

Setting vs Secret
A setting is any value that changes your application's behavior (e.g., a default zoom level or page title). A secret is a special, sensitive setting — passwords, API keys, connection strings — that must never be committed to source control.

Why externalize configuration?

🔄
No recompilation
Change a database connection string or API key at deploy time without touching code.
🔒
Security
Hardcoded secrets can be extracted from compiled binaries and end up in source control. Externalizing them reduces this risk.
🌍
Multiple environments
Use a local SQLite database in development and a production SQL Server in production, with no code change.
Configuration Provider
ASP.NET Core uses configuration providers to load key-value pairs from a variety of sources. Applications can use multiple providers simultaneously. Examples: JSON files, environment variables, command-line arguments, Azure Key Vault.

Unlike the old ConfigurationManager in ASP.NET Framework (which read from a single web.config file), ASP.NET Core lets you aggregate settings from many sources, with later providers able to override values from earlier ones.

📌
ConfigurationManager (new in .NET 6)
The ConfigurationManager class implements both IConfigurationBuilder (describes how to build configuration) and IConfigurationRoot (holds the values). When you add a provider, values are loaded immediately — no need to call Build() first. You interact with it mostly through the IConfiguration interface.
2

Adding a Configuration Provider in Program.cs

WebApplicationBuilder creates a ConfigurationManager instance and exposes it as builder.Configuration. By default it already registers several providers — but you can add your own or start from scratch.

Default providers added by WebApplicationBuilder

appsettings.json
Base settings file, loaded first. Optional by default.
appsettings.ENVIRONMENT.json
Environment-specific overrides (e.g., appsettings.Development.json). Optional.
User Secrets
Per-app secrets stored outside the project folder. Development only.
Environment variables
OS-level key-value pairs. Great for storing secrets in production.
Command-line arguments
Values passed when running dotnet run. Highest priority by default.

The default appsettings.json

JSONappsettings.json — default from the Empty template
{ "Logging": { "LogLevel": { "Default": "Information", "Microsoft.AspNetCore": "Warning" } }, "AllowedHosts": "*" }

Adding your own settings

Nest your settings under a named section to keep them organized. Values can be any JSON type (numbers, strings, booleans, objects) — they are all stored internally as strings.

JSONappsettings.json — adding MapSettings section
{ "Logging": { /* ... */ }, "AllowedHosts": "*", "MapSettings": { // ← namespace your settings "DefaultZoomLevel": 9, // ← numbers are stored as strings internally "DefaultLocation": { // ← deeply nested structures are fine "Latitude": 50.500, "Longitude": -4.000 } } }

Loading and reading values in Program.cs

C#Program.cs — loading from a single JSON file manually
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); builder.Configuration.Sources.Clear(); // remove all defaults builder.Configuration.AddJsonFile("appsettings.json", optional: true); // won't throw if missing WebApplication app = builder.Build(); // Access by key — colon (:) denotes section separator var zoom = builder.Configuration["MapSettings:DefaultZoomLevel"]; // "9" var lat = builder.Configuration["MapSettings:DefaultLocation:Latitude"]; // "50.5" // Or grab a whole section: var lat2 = builder.Configuration .GetSection("MapSettings")["DefaultLocation:Latitude"]; // same result // You can also inject IConfiguration directly into endpoint handlers: app.MapGet("/", (IConfiguration config) => config.AsEnumerable()); app.Run();
⚠️
Keys are NOT case-sensitive, values are always strings
Configuration keys are case-insensitive — MapSettings:DefaultZoomLevel and mapsettings:defaultzoomlevel refer to the same key. All values, including numbers and booleans in JSON, are stored internally as strings. You must parse them yourself when using raw IConfiguration.
🚫
Never store secrets in appsettings.json!
appsettings.json is typically committed to source control. Never put passwords, API keys, or connection strings in it. Use User Secrets in development and environment variables in production. You'll learn how in section 4.

3

Using Multiple Providers to Override Configuration Values

When you add providers to ConfigurationManager, the order matters. Think of providers as stacking layers: each new layer can add new keys or overwrite values from a previous layer.

🔑
Golden Rule — Order determines priority
Later providers win. If two providers define the same key, the value from the provider added last is used. The ConfigurationManager is essentially a dictionary — adding a second provider can add new keys or overwrite old values.
C#Program.cs — loading from three providers (order matters)
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); builder.Configuration.Sources.Clear(); builder.Configuration.AddJsonFile("sharedSettings.json", optional: true); // layer 1 — base builder.Configuration.AddJsonFile("appsettings.json", optional: true); // layer 2 — overrides builder.Configuration.AddEnvironmentVariables(); // layer 3 — wins WebApplication app = builder.Build(); app.Run();

Concrete override example

appsettings.json
{ "MyAppConnString": "localDB;", "MapSettings": { "DefaultZoomLevel": 5 } }
Environment Variables (added after)
// Environment variables on the machine: MyAppConnString="productionDB;" GoogleMapsApiKey="123456ABCD" // Result in IConfiguration: // MyAppConnString → "productionDB;" ← overwritten! // GoogleMapsApiKey → "123456ABCD" ← new key added // MapSettings:DefaultZoomLevel → 5 ← unchanged

Hierarchical keys in environment variables

To represent nested JSON keys in environment variables, use colon (:) or double underscore (__) as the separator:

BashSetting hierarchical config via environment variables
# Equivalent to JSON: { "MapSettings": { "DefaultZoomLevel": 5 } } # Using colon — works on Windows and macOS MapSettings:DefaultZoomLevel=5 # Using double underscore — required on Linux (colon not allowed in env var names) MapSettings__DefaultZoomLevel=5 # Both are converted to "MapSettings:DefaultZoomLevel" in IConfiguration
💡
Avoid env var collisions with a prefix
If you only want to load env variables that belong to your app, use AddEnvironmentVariables("MYAPP_"). The prefix is stripped from the key, so MYAPP_DatabaseName becomes DatabaseName in IConfiguration.

4

Storing Configuration Secrets Safely

Sensitive values — passwords, connection strings, API keys — should never be stored in appsettings.json, which gets committed to source control. ASP.NET Core provides two approaches depending on where your app is running.

Development: User Secrets Manager
// Stored OUTSIDE the project folder // in your OS user profile // Won't be committed to git // Setup: dotnet user-secrets init // Add a secret: dotnet user-secrets set \ "MapSettings:GoogleMapsApiKey" "F5RJT9" // In .csproj — auto-generated: <UserSecretsId>96eb2a39...</UserSecretsId>
Production: Environment Variables
// Set on the server / Docker container // Never in source control // Added by default in WebApplicationBuilder // On Linux server: export MyAppConnString="Server=prod;..." // Docker compose: environment: - MyAppConnString=Server=prod;... // Azure App Service: set in portal UI

Adding User Secrets in Program.cs

WebApplicationBuilder adds the User Secrets provider automatically in development. If you cleared sources and need to add it manually:

C#Program.cs — conditionally adding User Secrets
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); if (builder.Environment.IsDevelopment()) { // Only in development — not in staging or production // Program class is used to locate the UserSecretsId in the .csproj builder.Configuration.AddUserSecrets<Program>(); } WebApplication app = builder.Build(); app.Run();
⚠️
Secrets are NOT encrypted
Neither User Secrets nor environment variables encrypt their values — they are stored as plain text. Their benefit is keeping secrets outside your project folder so they don't end up in source control. For true encryption, use Azure Key Vault or HashiCorp Vault.

5

Reloading Configuration Values When They Change

File-based configuration providers can watch for file changes and rebuild IConfiguration automatically — no restart required. This is especially useful for toggling log levels in production without downtime.

C#Program.cs — enabling reloadOnChange
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); builder.Configuration.Sources.Clear(); builder.Configuration.AddJsonFile( "appsettings.json", optional: true, reloadOnChange: true); // ← IConfiguration rebuilt when file changes on disk WebApplication app = builder.Build(); app.Run();
📌
reloadOnChange is ON by default
WebApplicationBuilder adds appsettings.json with reloadOnChange: true by default. You only need to set it explicitly if you cleared all sources and are adding providers manually. Note: environment variable and command-line providers cannot reload — only file-based providers support this feature.
⚠️
File watchers have a cost
ASP.NET Core sets up a background file watcher for each file with reloadOnChange: true. This is normally fine, but watching thousands of files could cause issues. Stick to watching a small number of critical config files.
6

Strongly Typed Settings with the Options Pattern

Accessing config via builder.Configuration["MapSettings:DefaultZoomLevel"] is error-prone: typos in key strings, manual type conversion, no IntelliSense. The options pattern solves this by binding config sections to typed POCO classes.

❌ Raw IConfiguration — messy
app.MapGet("/settings", (IConfiguration cfg) => { // Strings everywhere — typos silently return null string title = cfg["AppDisplaySettings:Title"]; // Manual type conversion — ugly bool show = bool.Parse( cfg["AppDisplaySettings:ShowCopyright"]); return new { title, show }; });
✅ Strongly typed options — clean
app.MapGet("/settings", (IOptions<AppDisplaySettings> opts) => { AppDisplaySettings s = opts.Value; // Fully typed, IntelliSense, no parsing string title = s.Title; bool show = s.ShowCopyright; return new { title, show }; });

1. Define your POCO options class

C#AppDisplaySettings.cs — a simple options POCO
public class AppDisplaySettings { // Must be: nonabstract, public, parameterless constructor, public properties public string Title { get; set; } public bool ShowCopyright { get; set; } } public class MapSettings { public int DefaultZoomLevel { get; set; } public LocationSettings DefaultLocation { get; set; } } public class LocationSettings { public double Latitude { get; set; } public double Longitude { get; set; } }

2. Register with Configure<T> in Program.cs

C#Program.cs — binding config sections to POCO options
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); // Bind the "MapSettings" section to the MapSettings POCO class builder.Services.Configure<MapSettings>( builder.Configuration.GetSection("MapSettings")); // ❶ // Bind "AppDisplaySettings" section to AppDisplaySettings POCO builder.Services.Configure<AppDisplaySettings>( builder.Configuration.GetSection("AppDisplaySettings")); // ❷ // ❶ Registers IOptions<MapSettings> as a singleton in the DI container // ❷ The section name doesn't have to match the class name, but it's good convention WebApplication app = builder.Build(); app.Run();
💡
Use nameof() to avoid typos
You can call GetSection(nameof(MapSettings)) instead of GetSection("MapSettings"). The nameof() operator is evaluated at compile time, so a renamed class will produce a compile error instead of a silent runtime mismatch.
🧠 You forgot to call builder.Services.Configure<MapSettings>() but still inject IOptions<MapSettings> into a handler. What happens?
7

The IOptions<T> Interface

IOptions<T> is a simple wrapper interface with a single Value property that holds your typed settings POCO. It is registered as a singleton in the DI container — the options object is created once and never changes, even if the underlying config file is edited.

C#Injecting IOptions<T> into an endpoint handler
app.MapGet("/display-settings", (IOptions<AppDisplaySettings> options) => // ← injected by DI { AppDisplaySettings settings = options.Value; // ← access your POCO here return new { title = settings.Title, // ← fully typed, no parsing showCopyright = settings.ShowCopyright }; });

What Configure<T> does internally

1
Creates a ConfigureOptions<T> instance that holds the binding instructions.
2
When first requested, the binder reads the IConfigurationSection and sets all matching public properties on a new instance of your POCO class.
3
The bound POCO is stored inside IOptions<T> as a singleton. It is never re-created, even if you edit appsettings.json at runtime.
⚠️
IOptions<T> does not reload
Even if you set reloadOnChange: true on the JSON provider, IOptions<T> will not pick up the new values. The underlying IConfiguration is rebuilt, but the singleton options object stays frozen at its original values. Use IOptionsSnapshot<T> (next section) to get reloading behaviour.
8

Reloading with IOptionsSnapshot<T>

IOptionsSnapshot<T> is conceptually identical to IOptions<T>, but is registered as a scoped service — a new instance is created per request, and it reflects any configuration changes since the last request.

InterfaceLifetimeReloads on config change?Use when
IOptions<T> Singleton ❌ No Settings don't change at runtime (most cases)
IOptionsSnapshot<T> Scoped ✅ Yes (per request) Settings may change and you want the latest value
IOptionsMonitor<T> Singleton ✅ Yes (event-driven) Singleton services that need live config updates
C#Using IOptionsSnapshot<T> — always reflects latest config
app.MapGet("/display-settings", (IOptionsSnapshot<AppDisplaySettings> options) => // ← scoped, reloads automatically { AppDisplaySettings settings = options.Value; // ← current config at this request return new { title = settings.Title, showCopyright = settings.ShowCopyright }; });
⚠️
Cannot inject IOptionsSnapshot into a Singleton
Because IOptionsSnapshot<T> is scoped, injecting it into a singleton service creates a captive dependency — the singleton would hold a scoped object beyond its intended lifetime. Use IOptionsMonitor<T> for singleton services that need live config updates.

Options Binding Rules — What Gets Bound and What Doesn't

The options binder uses reflection to populate your POCO. Your class must be non-abstract with a public parameterless constructor. The binder then loops through all properties and binds those that qualify.

✅ Bindable properties
public class BindableOptions { // Simple built-in types public string Name { get; set; } public int Count { get; set; } // Complex nested types public SubClass Child { get; set; } // Read-only with default value public SubClass RO { get; } = new(); // Collections (interface or concrete) public List<SubClass> List { get; set; } public Dictionary<string,SubClass> Dict { get; set; } public IEnumerable<SubClass> IEnum { get; set; } }
❌ NOT bindable
public class UnbindableOptions { // Not public internal string Hidden { get; set; } // No getter public string SetOnly { set; } // Null read-only — binder can't write to it public SubClass NullRO { get; } = null; // Indexer public string this[int i] { get => ...; } // Non-string dictionary key public Dictionary<int,SubClass> Bad { get; set; } }
📌
Collection binding rules
If a collection property is already initialized, the binder adds to it. If it is null, the binder creates a new List<> for IEnumerable<>, ICollection<>, etc. Exception: an initialized IEnumerable<> cannot be added to (it has no Add() method) — leave it null if you need the binder to populate it. Dictionary keys must be string, enum, or an integer type.
10

Binding Strongly Typed Settings Without IOptions

Some developers find the IOptions<T> wrapper annoying — services must take a dependency on it just to call .Value immediately. You can bind a POCO directly and register it in the DI container, allowing direct injection.

C#Program.cs — create, bind, and register as singleton
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); var settings = new MapSettings(); // ❶ create instance builder.Configuration.GetSection("MapSettings").Bind(settings); // ❷ bind section to it builder.Services.AddSingleton(settings); // ❸ register directly in DI WebApplication app = builder.Build(); // Now inject MapSettings directly — no IOptions wrapper needed! app.MapGet("/", (MapSettings map) => map); app.Run();
C#Program.cs — delegate to IOptions registration
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); builder.Services.Configure<MapSettings>( // ❶ register IOptions normally builder.Configuration.GetSection("MapSettings")); builder.Services.AddSingleton(provider => // ❷ also register MapSettings directly provider.GetRequiredService<IOptions<MapSettings>>().Value); // by extracting from IOptions WebApplication app = builder.Build(); // Inject MapSettings directly — no wrapper app.MapGet("/", (MapSettings map) => map); app.Run();
⚠️
Trade-off: no automatic reloading
Both approaches register the POCO as a singleton. This means you lose the automatic reloading capability of IOptionsSnapshot<T>. In most production apps that's fine, but factor this in before using direct injection.

Configuring an App for Multiple Environments

Almost every real app runs in multiple environments — a local dev machine, a staging server, and production. ASP.NET Core has built-in support for detecting the current environment and loading the right configuration for it.

Hosting Environment
The current running context of your application. ASP.NET Core reads the ASPNETCORE_ENVIRONMENT environment variable to determine this. If not set, it defaults to Production. The value is exposed as IHostEnvironment.EnvironmentName.

The three standard environments

Development
Your local machine. Developer exception page on. Verbose logging. User Secrets for sensitive config.
Staging
Pre-production testing. Mimics production config. Error handling on. Environment variables for secrets.
Production
Live server. Strict error handling. Minimal logging. Azure Key Vault or environment vars for secrets.
C#Program.cs — using IHostEnvironment to customize behavior
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); IHostEnvironment env = builder.Environment; // ← access current environment // Use helper methods — always case-insensitive if (env.IsDevelopment()) { /* dev-only setup */ } if (env.IsStaging()) { /* staging-only setup */ } if (env.IsProduction()) { /* production-only setup */ } // Or check by name for custom environments: if (env.IsEnvironment("UAT")) { /* user acceptance testing */ } builder.Services.AddProblemDetails(); WebApplication app = builder.Build(); // Customize the middleware pipeline per environment if (!builder.Environment.IsDevelopment()) { app.UseExceptionHandler(); // friendly error page in staging/production } app.Run();
💡
Default is Production — deploy safely
If ASPNETCORE_ENVIRONMENT is not set, ASP.NET Core defaults to Production. This means that on a freshly configured server where you forgot to set the variable, your app will still behave safely (no developer exceptions, minimal logging). The current environment is also logged to the console at startup so you can verify it.

Loading Environment-Specific Configuration Files

A common pattern is to have a base appsettings.json (shared across all environments) and additional optional files per environment. Any key in the environment-specific file overwrites the same key from the base file.

C#Program.cs — environment-specific JSON files (WebApplicationBuilder does this by default)
WebApplicationBuilder builder = WebApplication.CreateBuilder(args); IHostEnvironment env = builder.Environment; builder.Configuration.Sources.Clear(); builder.Configuration .AddJsonFile("appsettings.json", optional: false) // ← base, required .AddJsonFile($"appsettings.{env.EnvironmentName}.json", // ← e.g. appsettings.Development.json optional: true); // ← optional — skip if missing if (env.IsDevelopment()) { builder.Configuration.AddUserSecrets<Program>(); // ← secrets only in dev } builder.Configuration.AddEnvironmentVariables(); // ← always last (highest priority) WebApplication app = builder.Build(); app.Run();

File loading order (later overrides earlier)

①
appsettings.json
base config — lowest priority
②
appsettings.{Environment}.json
overrides base
③
User Secrets
development only
④
Environment Variables
highest priority — always wins

13

Setting the Environment with launchSettings.json

launchSettings.json (in the Properties folder) lets you define profiles — named run configurations for local development. Each profile can set its own environment variables, URLs, and launch behavior.

📌
Development only — not deployed to production
launchSettings.json is for local development. It is not deployed to production servers. Use actual OS environment variables on your servers.
JSONProperties/launchSettings.json — typical three-profile setup
{ "profiles": { "http": { // ← used on macOS by default "commandName": "Project", // ← equivalent to dotnet run "launchBrowser": true, "applicationUrl": "http://localhost:5063", "environmentVariables": { "ASPNETCORE_ENVIRONMENT": "Development" // ← sets env for this profile only } }, "https": { // ← used by Visual Studio on Windows "commandName": "Project", "launchBrowser": true, "applicationUrl": "https://localhost:7202;http://localhost:5063", "environmentVariables": { "ASPNETCORE_ENVIRONMENT": "Development" } }, "IIS Express": { // ← Windows only, behind IIS Express "commandName": "IISExpress", "launchBrowser": true, "environmentVariables": { "ASPNETCORE_ENVIRONMENT": "Development" } } } }

Running with a specific profile

BashCLI profile commands
# Run with the "https" profile dotnet run --launch-profile https # Run ignoring launchSettings.json entirely dotnet run --no-launch-profile # Set environment via command-line argument (no launchSettings.json needed) dotnet run --no-launch-profile --environment Staging
💡
Per-project environment isolation
launchSettings.json scopes environment variables to a single project. If you set ASPNETCORE_ENVIRONMENT=Development in launchSettings.json, it applies only when running this project — unlike a system-level environment variable that would affect all apps on your machine.
🎯 Chapter Summary
Configuration providers load key-value string pairs from JSON files, environment variables, command-line args, User Secrets, and more. You add them to ConfigurationManager and later providers override earlier ones with the same key.

Never commit secrets to appsettings.json. Use User Secrets locally and environment variables (or Azure Key Vault) in production.

Strongly typed options (POCO classes + Configure<T>) are preferred over raw IConfiguration string keys. Use IOptions<T> (singleton) for stable config and IOptionsSnapshot<T> (scoped) when you need live reloading.

Hosting environments are determined by the ASPNETCORE_ENVIRONMENT variable. Default is Production. Use IsDevelopment(), IsStaging(), IsProduction() helper methods for case-insensitive checks.

Environment-specific JSON files (e.g., appsettings.Development.json) are loaded after the base file and override matching keys, allowing per-environment configuration without code changes.

launchSettings.json scopes environment variables to individual projects during local development and is never deployed to production.
