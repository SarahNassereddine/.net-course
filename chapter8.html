Chapter 8

An Introduction to Dependency Injection

Why manually creating objects hurts your code — and how the DI container takes that responsibility away, giving you modular, testable, loosely coupled apps.

1

SOLID Principles — Why DI Exists

ASP.NET Core was designed from the ground up to be modular and to follow good software engineering practices. The backbone of those practices is the set of SOLID principles — five guidelines for writing maintainable, extensible object-oriented code.

S
Single Responsibility
A class should have only one reason to change. Don't make one class do everything.
O
Open-Closed
Open for extension, closed for modification. Add behavior without changing existing code.
L
Liskov Substitution
A subclass should be replaceable for its parent class without breaking the program. If Dog extends Animal, anywhere you use Animal must also work with Dog.
I
Interface Segregation
Don't force clients to depend on methods they don't use. Prefer small, focused interfaces over fat ones.
D
Dependency Inversion ← this is what DI is about
High-level modules should not depend on low-level modules directly. Both should depend on abstractions (interfaces). Don't hard-code new EmailSender() — depend on IEmailSender and let something else provide the implementation.
💡
Dependency Injection is the mechanism that implements Dependency Inversion
The D in SOLID says "depend on abstractions, not concretions." DI is how you actually achieve this — a container creates objects for you and hands them in (injects them), so your code never has to call new on its dependencies.
2

The Problem Without DI

Imagine a RegisterUser endpoint that sends a welcome email. Without DI, the handler creates EmailSender itself — and EmailSender has its own dependencies that also need to be created manually.

C#Without DI — creating the whole dependency graph manually
// ❶ RegisterUser has an IMPLICIT dependency on EmailSender // You can only know this by reading the source code app.MapPost("/register", (UserDto user) => { // ❷ Must create MessageFactory first var factory = new MessageFactory(); // ❸ NetworkClient also has dependencies var settings = new EmailServerSettings("smtp.example.com", 587); var network = new NetworkClient(settings); // ❹ Now finally can create EmailSender var sender = new EmailSender(network, factory); // ❺ Only these two lines are the actual "work" sender.SendWelcomeEmail(user.Email); return Results.Ok(); });

Notice how only the last two lines do meaningful work. Everything above is ceremony — boilerplate you write just to construct the objects you actually need. The more complex your app, the deeper this dependency graph grows.

The graph can be many layers deep. In real applications, a single root service can pull in dozens of dependencies transitively. Managing this manually is unsustainable.

⚠️
The dependency graph is the root of the problem
The dependency graph is the set of objects that must be created to produce one "root" object. When you create these manually, every call site in your app becomes responsible for knowing how to build the entire graph — and if any class gains a new dependency, you have to update every place it's used.
3

Problems With Manual Object Creation

The previous code violates several of the SOLID principles. Here's exactly what goes wrong:

🚫
Violates SRP — Doing two jobs at once
The handler is responsible for both constructing an EmailSender object and using it to send email. Those are two separate responsibilities.
📜
Ceremony — Boilerplate that adds no value
In the example, only 2 of the 5+ lines of code do real work. The rest is just wiring up objects. This obscures the intent of the method and makes it harder to read.
🔗
Tied to the implementation — Hard to change
If you refactor EmailSender and add a new dependency, you must update every place that creates it. A change in one class ripples outward to all its call sites.
🧪
Impossible to unit test
If you test RegisterUser, a real email will be sent every time the test runs. You can't swap EmailSender for a fake (mock) version because the handler creates it directly.
🧠 A handler creates new EmailSender() directly inside its body. A colleague adds a new required dependency to the EmailSender constructor. What problem does this cause?
4

How Dependency Injection Solves It

With DI, your handler declares what it needs as a constructor/parameter, and the DI container (also called the IoC container — Inversion of Control) takes care of building the entire dependency graph and providing it.

❌ Without DI
app.MapPost("/register", (UserDto user) => { // Handler builds everything itself var settings = new EmailServerSettings(...); var network = new NetworkClient(settings); var factory = new MessageFactory(); var sender = new EmailSender(network, factory); sender.SendWelcomeEmail(user.Email); return Results.Ok(); });
✅ With DI
app.MapPost("/register", (UserDto user, IEmailSender sender) => { // Container provides sender automatically sender.SendWelcomeEmail(user.Email); return Results.Ok(); });

The handler no longer knows or cares how to create IEmailSender. It just uses it. The DI container is the only thing with a single responsibility of creating objects — it reads your registrations, builds the graph, and hands the ready-made service to your handler.

Who is responsible for what
DI Container — knows how to create every registered service and its full dependency graph
Minimal API Framework — asks the container for whatever services the handler's parameters declare
Your Handler — just declares its dependencies as parameters and uses them; never calls new
💡
Implicit vs. Explicit Dependencies
Without DI, RegisterUser has an implicit dependency on EmailSender — you can only discover it by reading the source code. With DI, the parameter IEmailSender sender makes the dependency explicit in the method signature — it's visible at a glance, and tools like IDEs and DI validators can check it.
5

Loose Coupling & Coding to Interfaces

Coupling describes how much one class needs to know about another to use it. The tighter the coupling, the harder your code is to change and test.

🔒 Tight Coupling
Handler creates new EmailSender(). It knows the concrete class, knows how to build it, and knows all its deps. Any change to EmailSender breaks the handler.
🔓 Loose Coupling
Handler depends on IEmailSender interface. It only knows about the contract (what methods exist). Any implementation can be swapped in without touching the handler.

The key pattern is: code to abstractions (interfaces), not to concrete types. This unlocks two huge benefits:

🔄
Swappable implementations
Register SmtpEmailSender in production and DummyEmailSender in development. The handler code never changes — only the DI registration changes.
🧪
Easy unit testing
In tests, inject a mock IEmailSender that records calls instead of sending real emails. No test runner ever touches your mail server.
C#Coding to an interface — the contract
// The abstraction (contract) — only this is what the handler sees public interface IEmailSender { void SendWelcomeEmail(string toAddress); } // Production implementation public class EmailSender : IEmailSender { public EmailSender(NetworkClient network, MessageFactory factory) { ... } public void SendWelcomeEmail(string toAddress) { /* real SMTP logic */ } } // Test/dummy implementation — no actual email is ever sent public class DummyEmailSender : IEmailSender { public void SendWelcomeEmail(string toAddress) { /* do nothing */ } }
🧠 Your handler accepts IEmailSender sender. In production the DI container injects EmailSender; in tests you inject DummyEmailSender. Why does this work?
6

Registration — Telling the Container What to Use

The DI container needs to know: "When someone asks for IEmailSender, which concrete class should I create?" This act of telling the container is called registration.

📋
The registration statement in plain English
"When any class or handler declares a dependency on IEmailSender, create an instance of EmailSender and provide it."

This is the process that decouples the contract (interface) from the implementation (concrete class).
C#Program.cs — registering services
var builder = WebApplication.CreateBuilder(args); // ❶ Register: "When IEmailSender is needed, use EmailSender" builder.Services.AddTransient<IEmailSender, EmailSender>(); var app = builder.Build(); // ❷ The handler just asks for IEmailSender — container resolves it app.MapPost("/register", (UserDto user, IEmailSender sender) => { sender.SendWelcomeEmail(user.Email); return Results.Ok(); }); app.Run();

To swap implementations, you change only the registration line. The handler itself is untouched — that's the power of coding to an interface.

7

DI in ASP.NET Core — How It Flows

In ASP.NET Core minimal APIs, dependency injection is wired into the framework itself. The RequestDelegateFactory (part of the minimal API infrastructure) inspects your handler's parameters and automatically requests matching services from the DI container before invoking your handler.

Request flow with DI
① HTTP Request arrives → RoutingMiddleware selects the endpoint
② RequestDelegateFactory inspects handler parameter types
③ For each service-type parameter → asks DI container to resolve it
④ DI container builds the full dependency graph and returns the service
⑤ Handler is invoked with all parameters populated — ready to use
🔍
How does the framework tell apart a service from a model binding parameter?
When a handler parameter is a type that's registered in the DI container, the framework injects it as a service. When it's not registered, the framework falls back to model binding (route, query string, body). This is why adding a parameter like IEmailSender sender just works — as long as you've registered the type.
8

Adding Framework Services to the Container

ASP.NET Core itself uses DI to configure its own internal components (like the Kestrel web server) and optional features (like Razor Pages or authentication). You register these services via builder.Services in Program.cs.

C#builder.Services — the IServiceCollection
var builder = WebApplication.CreateBuilder(args); // builder.Services is of type IServiceCollection // This is where you declare all your app's services // ❶ Core services (Kestrel, routing, etc.) are added automatically // by WebApplicationBuilder — you don't need to do anything // ❷ Optional framework features need explicit registration builder.Services.AddRazorPages(); // registers all Razor Pages services builder.Services.AddAuthentication(); // registers auth middleware builder.Services.AddAuthorization(); // registers authorization services var app = builder.Build();

The convention is that every library with required services exposes an Add*() extension method on IServiceCollection. This method adds everything the library needs internally, so you don't have to wire it up manually.

📋
Registration — telling the container which implementation to use
The process of telling the DI container "when someone asks for IEmailSender, provide an EmailSender" is called registration. In production: builder.Services.AddTransient<IEmailSender, EmailSender>(). In tests: swap for AddTransient<IEmailSender, DummyEmailSender>(). The handler never changes.
⚠️
Forgot to register a service? You'll get an exception.
There's no way to know at a glance which libraries require explicit registration — you have to check the docs. If you forget, ASP.NET Core will throw an InvalidOperationException at runtime with a message like "Unable to resolve service for type '...' while attempting to activate '...'". Always check for this exception in your logs when adding a new package.

You can also configure services by passing an options object or using an options lambda when calling Add*():

C#Configuring services with options
// Using a lambda to configure options inline builder.Services.AddRazorPages(options => { options.Conventions.AuthorizeFolder("/Admin"); });
9

Using Services From the DI Container

In a minimal API application, you have two main ways to access services:

C# Injecting into an endpoint handler (canonical approach)
// ASP.NET Core's RequestDelegateFactory inspects the parameters. // Parameters that are registered services are resolved from DI. // Simple types (strings, ints) are bound from the route/query string. // Complex types (records, classes) are bound from the request body. app.MapGet("/users/{id}", (int id, LinkGenerator linker) => { // 'id' comes from the route segment // 'linker' is a service from the DI container var link = linker.GetPathByName("GetUser", new { id }); return Results.Ok(link); });
🔍
How does it know what's a service vs a route param?
RequestDelegateFactory checks if a parameter type is registered in the DI container. If it is, it pulls it from there. If it isn't, it tries to bind it from the request (route, query string, or body). This is automatic — you never need attributes for registered services.
C# Accessing services in Program.cs directly
var app = builder.Build(); // Use app.Services (IServiceProvider) to resolve services directly // Useful for startup tasks (e.g., database migrations) var dbContext = app.Services.GetRequiredService<AppDbContext>(); await dbContext.Database.MigrateAsync(); // run before app starts app.Run();
10

IServiceProvider — Direct Access Outside Requests

Sometimes you need a service during app startup — before any HTTP request arrives. For example, to run a database migration or seed initial data. In these cases, inject from the container directly using app.Services.

📋
Two key IServiceProvider methods
GetService<T>() — returns the service T if registered; returns null if not found.

GetRequiredService<T>() — returns the service T if registered; throws InvalidOperationException if not found. Prefer this when you're certain the service should be there.
C#Method 2 — access services in Program.cs via IServiceProvider
var builder = WebApplication.CreateBuilder(args); builder.Services.AddTransient<IEmailSender, EmailSender>(); var app = builder.Build(); // app.Services is an IServiceProvider — the DI container's public API var sender = app.Services.GetRequiredService<IEmailSender>(); sender.SendWelcomeEmail("admin@example.com"); // send startup notification app.Run();
💡
IServiceCollection vs. IServiceProvider — a crucial distinction
IServiceCollection (builder.Services) — used before the app is built. This is where you register services.

IServiceProvider (app.Services) — used after the app is built. This is where you resolve (retrieve) services.

Think of IServiceCollection as the recipe book, and IServiceProvider as the kitchen that uses it.
