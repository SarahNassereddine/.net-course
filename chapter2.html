Understanding ASP.NET Core

A deeper look at why ASP.NET Core was created, how .NET 8 is structured, the Kestrel hosting model, and how NuGet manages dependencies.

1

Using a Web Framework

ASP.NET Core makes writing web applications faster, easier, and more secure than building everything from scratch. It provides ready-made solutions for the most common web development tasks, so you focus on what makes your application unique.

🖼️

2

Why ASP.NET Core Was Created

The original ASP.NET was tightly coupled to Windows and IIS. ASP.NET Core was a ground-up rewrite to address modern development requirements.

🌍
Cross-Platform
Run and develop on Windows, macOS, and Linux. Deploy anywhere — including Docker and cloud-native environments.
🧩
Modular Architecture
Only include the packages your application actually needs. Smaller footprint, easier maintenance, independent evolution of features.
🔓
Open Source
Developed completely in the open on GitHub. Community involvement in design decisions, transparent bug tracking, and free forever.
📏
Web Standards
Adheres to modern HTTP standards and specifications. Works correctly with all compliant HTTP clients and proxies.
☁️
Modern Web Trends
Built for client-side SPA architectures, cloud deployment, containerization, and microservices — trends that were emerging when the original ASP.NET was designed.
📚
Note — .NET Core vs .NET Framework
.NET Core (and its successor, .NET 5+) employs many of the same APIs as .NET Framework but is more modular. It implements a different set of features, with the goal of providing a simpler programming model and modern APIs. It's a separate platform rather than a fork of .NET Framework, though it uses similar code for many of its APIs.

.NET 7 / .NET 8 Platform

Every ASP.NET Core app starts life as a plain .NET console application. You then add the Kestrel web server library, which turns that console app into a web server. From there, you layer in whatever features your app needs — logging, configuration, HTML generation — each as a separate library.

Think of it like building with blocks: you start with the smallest block and stack on top.

💡
Why start as a console app?
Because it keeps things simple and modular. A console app is just a process that starts and runs — adding Kestrel gives it the ability to listen for HTTP requests. Adding more libraries gives it more capabilities. You only include what you actually need.

How Kestrel and your logic connect
Kestrel receives an HTTP request from the client and calls down into your web application logic. Your logic processes the request (using logging, configuration, etc.) and returns a response back up to Kestrel, which sends it to the client.

Kestrel & Hosting Models

Unlike classic ASP.NET which required IIS to function, ASP.NET Core ships with its own built-in web server called Kestrel. This is a fundamental architectural change.

⚠️
Classic ASP.NET — Tightly coupled to IIS
In the old ASP.NET, IIS is deeply integrated into the application. IIS handles the request pipeline directly — it knows about your app's internals, manages the application pool, and is required even in development. You can't run classic ASP.NET without IIS on Windows.

ASP.NET Core — Kestrel is self-hosted
ASP.NET Core apps include Kestrel, a self-hosted web server, directly inside the application process. Any external server (like Nginx or IIS) simply acts as a reverse proxy — it forwards requests to Kestrel and receives responses, but has no deeper knowledge of the app.

🪟
Running inside IIS on Windows
By default on Windows, ASP.NET Core can run inside IIS (in-process hosting), which can provide better performance than the reverse-proxy model. IIS still has no deep knowledge of your app — it just hosts the process more efficiently. This is primarily a deployment detail and doesn't change how you build your app.
💡
Key takeaway
Whether you deploy to IIS, Nginx, Docker, or bare-metal Linux — your ASP.NET Core code is exactly the same. The hosting model is a deployment concern, not a development concern.

NuGet — The .NET Package Manager

NuGet is the package manager for .NET. It enables you to import libraries (packages) into your projects with a single command or click. Most ASP.NET Core applications have many NuGet dependencies.

📋
Note — How Dependencies Work
Most ASP.NET Core applications have dependencies on various external libraries, which are managed through NuGet. These dependencies are listed in the project, but the files of the libraries themselves aren't included in source control. Before you can build and run your application, you need to ensure that there are local copies of each dependency on your machine — typically done via dotnet restore.
XMLProject.csproj — NuGet dependencies
<Project Sdk="Microsoft.NET.Sdk.Web"> <PropertyGroup> <TargetFramework>net8.0</TargetFramework> </PropertyGroup> <ItemGroup> <!-- Each dependency listed as a PackageReference node --> <PackageReference Include="Microsoft.EntityFrameworkCore" Version="8.0.0" /> <PackageReference Include="Serilog.AspNetCore" Version="8.0.0" /> </ItemGroup> </Project> // Restore all packages listed above: dotnet restore
💡
Random Port Assignment
The template chooses a random port for your application's URL, which will be opened in the browser automatically when you run from Visual Studio. You can change this in launchSettings.json.
