Chapter 18

Building Forms with Tag Helpers

Tag Helpers let you write clean, semantic HTML that automatically generates correct form attributes, validates inputs, and protects against CSRF — all driven by your C# model.

1

Introduction to Tag Helpers

Tag Helpers are a feature of ASP.NET Core for server-side HTML rendering. They are Razor components that customize the HTML generated in your templates — either by augmenting standard HTML elements with asp-* attributes, or as entirely new standalone elements.

⚠️
Server-side only — not for SPAs
Razor and Tag Helpers are for server-side HTML rendering. You can't use Tag Helpers directly in frontend frameworks like Angular, React, or Vue. For those, you'd use Minimal APIs or Web API controllers instead.
❌ Plain HTML — manual, error-prone
<form action="/Checkout" method="post"> <label for="FirstName">First Name</label> <input type="text" id="FirstName" name="Input.FirstName" value="@Model.Input.FirstName" required /> <!-- Easy to get id/name wrong! --> <!-- No validation attributes generated --> </form>
✅ Tag Helpers — declarative, correct
<form asp-page="Checkout"> <label asp-for="Input.FirstName"></label> <input asp-for="Input.FirstName" /> <!-- Tag Helpers generate correct id, name, --> <!-- value, type AND validation attributes --> <!-- Cannot mis-spell the property name --> </form>
💡
Visual distinction in Visual Studio
In Visual Studio, Tag Helpers are distinguishable from normal elements — they appear bold and in a different color. C# expressions are shaded, and IntelliSense is available for the model properties, giving you compile-time safety when referencing model members.
2

Why Tag Helpers?

Tag Helpers solve four interconnected problems when building forms in ASP.NET Core:

🔗
Auto-populate values
Automatically fills the value attribute from the PageModel property — no manual value="@Model.X".
🏷️
Correct id and name
Generates the exact id and name attributes that model binding needs — no mismatches.
🎯
Correct input type
Picks the right HTML type attribute (email, number, checkbox, etc.) based on the C# type and DataAnnotations.
⚡
Validation attributes
Generates data-val-* attributes for jQuery Unobtrusive Validation — client-side rules from server-side annotations.
3

Enabling Tag Helpers — _ViewImports.cshtml

Tag Helpers are not available by default in every Razor file. You must explicitly opt in by adding an @addTagHelper directive. This is typically done once in the _ViewImports.cshtml file so every view in the application inherits it.

RazorPages/_ViewImports.cshtml — enabling Tag Helpers
@* This single line enables ALL built-in ASP.NET Core Tag Helpers *@ @addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers @* Breaking down the syntax: * → wildcard — import ALL Tag Helpers from this assembly Microsoft.AspNetCore.Mvc.TagHelpers → the assembly containing the built-in helpers You can also import Tag Helpers from other assemblies (e.g. your own): *@ @addTagHelper *, MyApp.TagHelpers @* Or remove a specific Tag Helper if you need to opt out: *@ @removeTagHelper Microsoft.AspNetCore.Mvc.TagHelpers.FormTagHelper, Microsoft.AspNetCore.Mvc.TagHelpers
💡
Already done in new project templates
If you create a new ASP.NET Core Razor Pages project using dotnet new webapp or Visual Studio, the _ViewImports.cshtml already contains the @addTagHelper directive. You only need to add it manually if you created the file by hand or are working in a project that didn't include it. If Tag Helpers appear to be doing nothing (no asp-* IntelliSense), a missing _ViewImports.cshtml is usually the cause.
📋
Opting out of a specific Tag Helper with @ prefix
You can also suppress a Tag Helper on a single element using the ! prefix. For example, <!label asp-for="Email"></!label> prevents the Label Tag Helper from running on that element, rendering it as plain HTML. This is useful when you need the raw element for a specific reason but don't want to remove the Tag Helper globally.

DataAnnotations and Tag Helpers Working Together

DataAnnotations on your model serve double duty: they validate data on the server AND provide metadata that Tag Helpers use to generate the correct HTML.

C#A view model decorated with DataAnnotations
public class UserBindingModel { [Required] [StringLength(100, ErrorMessage = "Maximum length is {1}")] [Display(Name = "Your name")] public string FirstName { get; set; } [Required] [EmailAddress] public string Email { get; set; } [Phone(ErrorMessage = "Not a valid phone number.")] [Display(Name = "Phone number")] public string PhoneNumber { get; set; } [Range(18, 120)] public int Age { get; set; } } // In the PageModel, expose it with [BindProperty]: public class CheckoutModel : PageModel { [BindProperty] public UserBindingModel Input { get; set; } }
💡
Binding model vs view model — use two separate objects in production
For simplicity in examples, the same class is used for both binding and display. In production, you should use two separate objects — a binding model (what comes from the request) and a view model (what gets displayed). This prevents mass-assignment attacks where a user crafts a POST body with extra fields they shouldn't be able to set.

The Form Tag Helper

The Form Tag Helper generates the action and method attributes on a <form> element using Razor Page routing — and also injects a hidden anti-CSRF token automatically.

AttributeWhat it generatesExample
asp-page Sets the action attribute to the URL of the specified Razor Page asp-page="Checkout" → action="/Checkout"
asp-route-* Appends route parameters to the action URL asp-route-id="5" → action="/Product/5"
asp-page-handler Sets which page handler to invoke on POST asp-page-handler="Delete" → calls OnPostDelete()
asp-antiforgery Controls hidden CSRF token generation (default: true) asp-antiforgery="false" disables the token
Razor → HTMLForm Tag Helper examples
@* Basic — posts back to the current page (must add method="post" manually) *@ <form method="post">...</form> @* Post to a named Razor Page *@ <form asp-page="Checkout">...</form> → generates: <form action="/Checkout" method="post">...</form> @* With route parameter *@ <form asp-page="Product" asp-route-id="@Model.ProductId">...</form> → generates: <form action="/Product/5" method="post"> @* With specific page handler — calls OnPostDelete() *@ <form asp-page-handler="Delete">...</form> @* Disable CSRF token (rare — only for APIs or external consumers) *@ <form asp-page="Checkout" asp-antiforgery="false">...</form>
🔒
Hidden Anti-CSRF Token — Automatic
The main job of the Form Tag Helper is to generate the action attribute, but it also performs one critical hidden function: it automatically generates a hidden field containing an anti-forgery token to prevent Cross-Site Request Forgery (CSRF) attacks. You don't write this manually — it's always there when using the Form Tag Helper.
⚠️
Never use GET for forms that modify data
If you omit asp-page, you must manually add method="post". Using GET for forms that modify data is a security risk — GET requests can be triggered by visiting a URL (e.g., via a malicious image tag), whereas POST requires an explicit form submission.

The Label Tag Helper

The Label Tag Helper generates the caption and for attribute for a <label> element, based on the model property referenced via asp-for.

C# + Razor → HTMLLabel Tag Helper — how the caption is derived
// Model property: [Display(Name = "Your name")] public string FirstName { get; set; } public string Email { get; set; } // No [Display] attribute // Razor: <label asp-for="FirstName"></label> <label asp-for="Email"></label> // Generated HTML: <label for="FirstName">Your name</label> ← uses [Display] Name <label for="Email">Email</label> ← falls back to property name // Override the caption manually: <label asp-for="Email">Please enter your Email</label> → <label for="Email">Please enter your Email</label> // Sub-properties (dot notation works just like C#): <label asp-for="Input.FirstName"></label> <label asp-for="Input.Address.City"></label>
♿
Accessibility — the for attribute matters
The for attribute is important for accessibility. It specifies the ID of the <input> to which the label refers, allowing screen readers to announce the label when the input is focused, and allowing users to click the label to focus the input. The Label Tag Helper sets this correctly, matching the id of the corresponding input.

The Input and Textarea Tag Helpers

The Input Tag Helper generates an <input> element with the correct type, id, name, value, and data-val-* validation attributes — all derived automatically from the model property.

C# + Razor → HTMLInput Tag Helper — full generated output for [EmailAddress]
// Model: [Required] [EmailAddress] public string Email { get; set; } // Razor: <input asp-for="Input.Email" /> // Generated HTML (from the [EmailAddress] attribute): <input type="email" id="Input_Email" name="Input.Email" value="[email protected]" data-val="true" data-val-email="The Email Address field is not a valid e-mail address." data-val-required="The Email Address field is required." /> // The data-val-* attributes are picked up by jQuery Unobtrusive Validation // for client-side validation — no extra JavaScript code needed.

How the Input Tag Helper picks the type:

C# type / AnnotationGenerated type
stringtype="text"
string + [EmailAddress]type="email"
string + [Phone]type="tel"
string + [Url]type="url"
string + [DataType(DataType.Password)]type="password"
[HiddenInput] attributetype="hidden"
booltype="checkbox"
byte, int, short, longtype="number"
decimal, double, floattype="text" (culture-safe)
DateTime or [DataType(DataType.Date)]type="datetime-local"
💡
Why decimal renders as text, not number
decimal, double, and float types render as type="text" rather than type="number". This is intentional — different cultures render decimal points and number group separators differently (e.g. 1.5 vs 1,5). Rendering as text avoids parse errors that would appear only in certain browser-culture combinations.
💡
asp-format — controlling display format
Use asp-format to customize how the value is displayed in the input. For example: <input asp-for="Dec" asp-format="{0:0.000}" /> — if Dec is 1.2, this generates value="1.200". Under the covers, this performs a string.Format() on the property's value. You can also set the format via [DisplayFormat("{0:0.000}")] on the model property.
Razor → HTMLTextarea Tag Helper
// Model: [StringLength(200)] [Required] public string Multiline { get; set; } = "This is some text."; // Razor: <textarea asp-for="Multiline"></textarea> // Generated HTML: <textarea data-val="true" id="Multiline" name="Multiline" data-val-length="Maximum length 200." data-val-length-max="200" data-val-required="The Multiline field is required."> This is some text. <!-- value goes inside the tag for textarea --> </textarea>

Overriding Tag Helper Output

    


    
Tag Helpers are designed to not get in your way. Any attribute you set manually in your Razor markup is passed through to the output unchanged — even if the Tag Helper would normally generate that same attribute. You can always override the defaults.


    

      
RazorPassing through custom attributes & overriding defaults

      
// The 'class' attribute is yours — Tag Helper won't touch it:
<input class="form-control" asp-for="Input.FirstName" />
→ generates: <input class="form-control" type="text" id="Input_FirstName" ... />

// Override the 'type' attribute — force an HTML5 color picker:
<input type="color" asp-for="FavoriteColor" />
// Even though FavoriteColor is a string (which maps to type="text"),
// your explicit type="color" wins and is preserved in the output.

// Add any extra HTML5 attribute — they all pass through:
<input asp-for="Input.Age" placeholder="e.g. 25" min="18" max="120" />

// Force a range slider for a numeric property:
<input type="range" min="0" max="100" asp-for="Input.Quantity" />

    


    

      
💡

      

        
This is better than legacy HTML Helpers

        
In legacy ASP.NET (HTML Helpers), adding extra attributes to generated elements required passing an anonymous object as a parameter — e.g., @Html.TextBoxFor(m => m.Name, new { @class = "form-control" }). Tag Helpers let you add attributes naturally in HTML, which is more readable and works properly with all HTML editors and IDEs.

      

    


    

      

        
❌ Legacy HTML Helper — awkward

        

@* Had to pass attributes as an anonymous object *@
@Html.TextBoxFor(m => m.Email,
    new { @class = "form-control",
          placeholder = "Enter email" })

@* Confusing @ escaping, hard to read, no HTML IntelliSense *@

      

      

        
✅ Tag Helper — natural HTML

        

@* Just write normal HTML attributes inline *@
<input asp-for="Email"
       class="form-control"
       placeholder="Enter email" />

@* Reads like HTML, full editor support *@

The Select Tag Helper

    


    
The Select Tag Helper generates <select> dropdowns and multiselect lists. It needs two PageModel properties: one to hold the selected value(s), one to hold the list of options.


    

      
⚠️

      

        
Only works with SelectListItem

        
The Select Tag Helper requires the items list to be of type IEnumerable<SelectListItem>. Each SelectListItem exposes a Value (what gets submitted) and Text (what the user sees). The items list property should NOT be in the binding model — it shouldn't be model-bound from the request. Normally you'd load it in OnGet().

      

    


    

      
C# + RazorSelect Tag Helper — dropdown and multiselect

      
// PageModel:
public class SelectModel : PageModel
{
    [BindProperty]
    public string SelectedCurrency { get; set; }   // single value

    [BindProperty]
    public IEnumerable<string> SelectedCurrencies { get; set; } // multiselect

    // NOT BindProperty — loaded from application, not from request
    public IEnumerable<SelectListItem> Currencies { get; set; }

    public void OnGet()
    {
        Currencies = new[]
        {
            new SelectListItem { Value = "USD", Text = "US Dollar" },
            new SelectListItem { Value = "GBP", Text = "British Pound" },
            new SelectListItem { Value = "EUR", Text = "Euro" },
        };
    }
}

// Razor — single-select dropdown:
<select asp-for="SelectedCurrency"
        asp-items="Model.Currencies"></select>

// Razor — multiselect list box (IEnumerable property → adds 'multiple' automatically):
<select asp-for="SelectedCurrencies"
        asp-items="Model.Currencies"></select>

// Razor — list box of fixed height (standard HTML 'size' attribute):
<select asp-for="SelectedCurrency"
        asp-items="Model.Currencies" size="4"></select>

// Razor — for enum types (built-in helper generates SelectListItem from enum):
<select asp-for="Status"
        asp-items="Html.GetEnumSelectList<OrderStatus>()"></select>

    


    

      
💡

      

        
asp-for vs asp-items — prefix rules

        
asp-for must not include the Model. prefix (it's implied). asp-items, on the other hand, must include it if referencing a view model property — e.g., asp-items="Model.Currencies". The Select Tag Helper automatically adds the multiple HTML attribute if the property bound by asp-for is an IEnumerable.

Select — OptGroups and the "No Selection" Option

    


    
Two common requirements come up frequently when working with <select> elements: grouping items with <optgroup> labels, and adding a "no selection" placeholder at the top of the list.


    
OptGroups — grouping items with SelectListGroup


    
To add <optgroup> elements to a select list, assign a SelectListGroup to the Group property of each SelectListItem. Items without a group are rendered outside any group, after the grouped items.


    

      
C# + Razor → HTMLSelect Tag Helper — grouping with optgroup

      
// PageModel setup — create groups, assign to items:
public class SelectListsModel : PageModel
{
    [BindProperty]
    public IEnumerable<string> SelectedValues { get; set; }

    public IEnumerable<SelectListItem> Items { get; set; }

    public SelectListsModel()
    {
        // ① Create group objects:
        var dynamic = new SelectListGroup { Name = "Dynamic Languages" };
        var @static = new SelectListGroup { Name = "Static Languages" };

        // ② Assign groups to items:
        Items = new List<SelectListItem>
        {
            new SelectListItem { Value = "js",     Text = "JavaScript", Group = dynamic },
            new SelectListItem { Value = "python", Text = "Python",     Group = dynamic },
            new SelectListItem { Value = "cpp",    Text = "C++",         Group = @static },
            new SelectListItem { Value = "csharp", Text = "C#"            } // no group
        };
    }
}

// Razor — no change needed, Tag Helper handles it:
<select asp-for="SelectedValues" asp-items="Model.Items"></select>

// Generated HTML:
<select id="SelectedValues" name="SelectedValues" multiple="multiple">
    <optgroup label="Dynamic Languages">
        <option value="js">JavaScript</option>
        <option value="python">Python</option>
    </optgroup>
    <optgroup label="Static Languages">
        <option value="cpp">C++</option>
    </optgroup>
    <option value="csharp">C#</option>  <!-- no group → rendered outside -->
</select>

    


    
Adding a "No Selection" Option


    
Without a placeholder, a <select> dropdown always has a value — it defaults to the first item. This can be misleading. Add an explicit "not selected" <option> inline in your Razor — Tag Helpers pass extra options through unchanged.


    

      

        
❌ No placeholder — always has a value

        

<select asp-for="SelectedValue"
        asp-items="Model.Items">
</select>
<!-- Renders "JavaScript" as selected by default -->
<!-- User might not notice they need to make a choice -->

      

      

        
✅ With placeholder — clear intent

        

<select asp-for="SelectedValue"
        asp-items="Model.Items">
    <option value="">** Not selected **</option>
</select>
<!-- Extra <option> is placed at the top -->
<!-- Blank Value="" = null/empty when submitted -->

      

    


    

      
💡

      

        
Extra <option> elements are always passed through unchanged

        
Any <option> elements you add manually inside the <select> in your Razor markup are rendered at the top of the list, before the Tag Helper's generated options. The Tag Helper doesn't remove or modify them. Use value="" so that submitting without a selection gives you an empty/null value that your [Required] validation can catch.

The Validation Message Tag Helper

    


    
The Validation Message Tag Helper displays the error message for a specific property when validation fails — applied to a <span> using asp-validation-for.


    

      
Razor → HTMLShowing validation errors per-field

      
@* Razor — place next to each input *@
<div>
    <label asp-for="Input.Email"></label>
    <input asp-for="Input.Email" />

</div>

// Generated HTML (before error):
<span class="field-validation-valid"
      data-valmsg-for="Input.Email"
      data-valmsg-replace="true"></span>

// After failed validation — server-side error shown:
<span class="field-validation-error"
      data-valmsg-for="Input.Email">
    The Email Address field is required.
</span>

// jQuery Validation also populates this span for client-side errors,
// clearing it again when the user corrects their input.

The Validation Summary Tag Helper

    


    
The Validation Summary Tag Helper displays a summary of all validation errors in one place — useful especially for showing model-level errors that aren't tied to a specific property.


    

      
RazorValidation summary — three modes

      
@* Applied to a div using asp-validation-summary *@

// None — don't display a summary (pointless but available)
<div asp-validation-summary="None"></div>

// ModelOnly — only errors NOT associated with a property
// Best when field-level spans already show per-field errors:
<div asp-validation-summary="ModelOnly"></div>

// All — every error, both property-level and model-level
<div asp-validation-summary="All"></div>

// Adding a model-level error in your handler (blank key = model error):
if (Input.CurrencyFrom == Input.CurrencyTo)
{
    ModelState.AddModelError(string.Empty, "Cannot convert currency to itself");
}
if (!ModelState.IsValid) return Page();

    


    

      
📋

      

        
When to use ModelOnly vs All

        
Use ModelOnly when you already have per-field <span asp-validation-for="..."> elements next to each input — those handle property errors. ModelOnly then only shows business rule errors that span multiple fields (like "start date must be before end date"). Use All if you want a single summary at the top of the form and don't use per-field spans.

Enabling Client-Side Validation

    


    
The Input Tag Helper generates data-val-* attributes on your inputs, but those attributes do nothing on their own. To activate client-side validation, you must include the jQuery Validation and Unobtrusive Validation JavaScript libraries in your page.


    

      
📋

      

        
How it works end-to-end

        
The Input Tag Helper writes data-val="true", data-val-required="...", data-val-email="...", and similar attributes directly onto the <input>. When the page loads, jQuery Unobtrusive Validation reads these attributes and registers the validation rules automatically — no additional JavaScript needed.

      

    


    

      
RazorAdding the validation scripts to a view — two approaches

      
@* Approach 1 — Use the built-in partial view (recommended for new projects) *@
@* Add this section at the bottom of any Razor Page that needs client validation: *@

@section Scripts {
    @await Html.PartialAsync("_ValidationScriptsPartial")
}

// _ValidationScriptsPartial.cshtml contains:
//   <script src="~/lib/jquery-validation/dist/jquery.validate.min.js"></script>
//   <script src="~/lib/jquery-validation-unobtrusive/jquery.validate.unobtrusive.min.js"></script>


@* Approach 2 — Add scripts directly (for full control) *@
<script src="~/lib/jquery-validation/dist/jquery.validate.min.js"></script>
<script src="~/lib/jquery-validation-unobtrusive/jquery.validate.unobtrusive.min.js"></script>

// NOTE: jQuery itself must also be included — the default layout template
// includes it globally, so you usually don't need to add it separately.

    


    

      

        
❌ Without validation scripts

        

<!-- data-val-* attributes are present in HTML -->
<input data-val="true"
       data-val-required="Required."
       name="Email" />

<!-- But nothing reads them — user can submit
     an empty form, errors only caught server-side -->

      

      

        
✅ With validation scripts

        

@section Scripts {
    @await Html.PartialAsync(
        "_ValidationScriptsPartial")
}
<!-- jQuery Unobtrusive Validation reads the
     data-val-* attributes and shows errors
     instantly as the user types or blurs each field -->

      

    


    

      
⚠️

      

        
Client-side validation is supplemental — always validate server-side too

        
Client-side validation improves the user experience by giving instant feedback, but it can be bypassed (users can disable JavaScript or send raw HTTP requests). Always check ModelState.IsValid in your page handler — server-side validation is the authoritative check. Client-side is a UX layer on top, not a replacement.

The Anchor Tag Helper

    


    
The Anchor Tag Helper generates the href attribute of an <a> element using routing — the same routing system used by forms. This means your links always point to the correct URL even if you change your routes.


    

      
Razor → HTMLAnchor Tag Helper — generating links via routing

      
@* Link to the Index page (generates "/" not "/Index") *@
<a asp-page="/Index">Home</a>
→ <a href="/">Home</a>

@* Link to a page with a route parameter *@
<a asp-page="/Product/Detail" asp-route-id="@item.Id">View</a>
→ <a href="/Product/Detail/42">View</a>

@* Link to a specific page handler — calls OnGetExport() *@
<a asp-page="/Orders" asp-page-handler="Export">Export</a>
→ <a href="/Orders?handler=Export">Export</a>

@* Generate an absolute URL (includes protocol + host) *@
<a asp-page="/Index" asp-protocol="https" asp-host="myapp.com">Home</a>
→ <a href="https://myapp.com/">Home</a>

    


    

      
AttributePurpose

      
asp-pageThe Razor Page to link to

      
asp-page-handlerSpecific page handler method (e.g., OnGetExport)

      
asp-areaSets the area route parameter

      
asp-hostGenerates an absolute URL to the specified host

      
asp-protocolGenerates http:// or https:// absolute URL

      
asp-routeUses a named route to generate the URL (not the same as asp-route-*)

      
asp-route-*Sets individual route parameters (repeat for multiple)

    


    

      
⚠️

      

        
asp-route vs asp-route-* — easy to confuse

        
asp-route (no suffix) is for specifying a named route (e.g., a named minimal API endpoint). asp-route-* (with suffix) sets individual route parameter values during URL generation. These are two different attributes and serve different purposes — don't confuse them.

Cache-Busting with asp-append-version

    


    
Browsers cache CSS, JS, and image files for performance. When you update a file, users may still see the old cached version. Cache-busting solves this by appending a unique version string to the URL.


    

      
📋

      

        
How it works

        
A cache-busting query string adds a parameter like ?v=1. When the file changes, the hash changes too (e.g., ?v=2). Browsers see this as a new URL and fetch the fresh file, rather than using the cached version. The asp-append-version attribute computes the hash automatically from the file's contents — you never need to update it manually.

      

    


    

      

        
❌ Manual — you forget to update

        

<!-- Hard-coded version — easy to forget to bump -->
<link href="/css/site.css?v=1" rel="stylesheet" />
<script src="/js/site.js?v=1"></script>
<!-- Update every reference everywhere = mistakes -->

      

      

        
✅ asp-append-version — automatic

        

<link rel="stylesheet"
      href="~/css/site.css"
      asp-append-version="true" />
<script src="~/js/site.js"
        asp-append-version="true"></script>

→ Generated HTML:
<link href="/css/site.css?v=EWaMeWsJ..." />

      

    


    

      
💡

      

        
Hash-based — updates automatically on any file change

        
The asp-append-version attribute loads the referenced file and generates a unique hash based on its contents. Every time you change the file, the hash changes. The browser sees the new hash as a new URL and fetches the updated version. Works on <script>, <link>, and <img> elements.

Conditional Markup with the Environment Tag Helper

    


    
The Environment Tag Helper renders its content only when the app is running in the specified environments. Unlike most Tag Helpers, the whole element is the Tag Helper — not just an attribute on an existing element.


    

      

        
C# if statement — verbose

        

@if (env == "Testing" || env == "Staging")
{
    <div class="warning">
        You are on a testing environment
    </div>
}

      

      

        
<environment> Tag Helper — clean

        

<environment include="Testing,Staging">
    <div class="warning">
        You are on a testing environment
    </div>
</environment>

<environment exclude="Development">
    <!-- Only shown in non-Development -->
    <script src="site.min.js"></script>
</environment>

      

    


    

      
💡

      

        
Common use case — minified vs development bundles

        
A very common pattern is to serve unminified, readable JavaScript files in Development (for debugging) and minified bundles in Production (for performance). The Environment Tag Helper makes this a one-liner in your layout file. Use include="Development" for the verbose version and exclude="Development" for the minified one.

Tag Helpers vs HTML Helpers

    


    
If you've worked with legacy ASP.NET MVC, you may have used HTML Helpers — C# methods that generated HTML from your views. Tag Helpers are their modern replacement, but HTML Helpers still exist in ASP.NET Core for backward compatibility.


    

      
FeatureHTML Helpers (legacy)Tag Helpers (modern)

      

        Syntax
        @Html.TextBoxFor(m => m.Name, new { @class = "x" })
        <input asp-for="Name" class="x" />
      

      

        Looks like
        C# method calls embedded in HTML
        Normal HTML attributes
      

      

        Editor support
        Confused most non-VS HTML editors
        Works in any HTML editor, full IntelliSense
      

      

        Adding extra attributes
        Anonymous object parameter — verbose and error-prone
        Just write them inline naturally
      

      

        Multiple helpers on one element
        Not easily possible
        Multiple asp-* attributes on same element — fully supported
      

      

        Available in ASP.NET Core?
        Yes — kept for migration scenarios
        Yes — preferred for all new code
      

    


    

      

        
❌ HTML Helper — C#-centric, hard to read

        

@Html.LabelFor(m => m.Email)
@Html.TextBoxFor(m => m.Email,
    new { @class = "form-control" })
@Html.ValidationMessageFor(m => m.Email)

@* Reads like C#, confuses HTML editors,
   awkward anonymous object for attributes *@

      

      

        
✅ Tag Helper — HTML-centric, readable

        

<label asp-for="Email"></label>
<input asp-for="Email"
       class="form-control" />
<span asp-validation-for="Email"></span>

@* Reads like HTML, any editor supports it,
   attributes just go inline naturally *@

      

    


    

      
📋

      

        
When you might still see HTML Helpers

        
You'll encounter HTML Helpers in older codebases being migrated to ASP.NET Core, and in some built-in framework helpers like @Html.PartialAsync() and @Html.GetEnumSelectList<T>() — the latter is commonly used inside asp-items on a select element. HTML Helpers are not going away, but for any new form markup, use Tag Helpers exclusively.

CSRF Protection and ValidateAntiForgeryToken

    


    
A CSRF (Cross-Site Request Forgery) attack tricks a user into unknowingly submitting a malicious request to a site where they're authenticated — like changing their password or making a purchase without knowing it.


    
    

      

        AttackHow a CSRF attack works
      

      

        

          

            1
            User logs into their bank at bank.com and stays logged in (auth cookie stored)
          

          

            2
            User visits evil.com — which silently sends a POST to bank.com/transfer?to=hacker&amount=1000
          

          

            3
            Browser automatically sends the bank cookie — server thinks it's a legitimate request!
          

          

            ✓
            Anti-forgery token prevents this — the malicious site can't read the token from bank.com (cross-origin blocked), so validation fails
          

        

      

    


    

      
C#Anti-forgery token — how it works in ASP.NET Core

      
// ① Server generates a unique token and embeds it in the form
// (Form Tag Helper does this automatically — hidden field)
<form asp-page="Submit">
    <!-- Hidden field added automatically by the Form Tag Helper: -->
    <!-- <input name="__RequestVerificationToken" type="hidden" value="..." /> -->
    <input asp-for="Input.Username" />
    <button type="submit">Submit</button>
</form>

// ② On POST, [ValidateAntiForgeryToken] checks the token
[ValidateAntiForgeryToken]
public IActionResult OnPost()
{
    // Only executes if token is present and valid
    // If token is missing or invalid → 400 Bad Request
    TempData["Success"] = "Your message has been sent!";
    return RedirectToPage();
}

// ③ In ASP.NET Core Razor Pages — validation is automatic!
// Razor Pages validates the anti-forgery token on every POST by default.
// [ValidateAntiForgeryToken] is NOT required explicitly — it's built in.

    


    

      
💡

      

        
Razor Pages validates CSRF automatically on every POST

        
In Razor Pages, anti-forgery validation is automatic for all POST requests. You don't need to add [ValidateAntiForgeryToken] manually. The Form Tag Helper always adds the token to forms. The framework always validates it on POST. This is one of the built-in security features that makes Razor Pages safer than rolling your own forms.

      

    


    

      
🧠 A user visits a malicious website that tries to send a POST to your Razor Pages app (CSRF attack). Why does the anti-forgery token protect against this?

      

        
        
        
        
      

      

    


    

      
📋

      

        
Old Razor syntax vs Tag Helper (for reference)

        

          Old MVC Razor syntax: @Html.AntiForgeryToken() — manually adds the hidden token field.

          Modern Razor Pages / Tag Helper: just use <form asp-page="..."> — the token is added automatically. No manual @Html.AntiForgeryToken() needed.
