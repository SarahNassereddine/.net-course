Saving Data with Entity Framework Core

Learn how to use Entity Framework Core as an ORM to interact with databases in your ASP.NET Core applications, from basic CRUD operations to managing schema migrations.

Introducing Entity Framework Core

Interacting with databases from application code is often a messy affair, and you can take many approaches. A task as simple as reading data from a database requires handling network connections, writing SQL statements, and managing variable result data.

The .NET ecosystem has a whole array of libraries you can use for database access, ranging from low-level ADO.NET libraries to higher-level abstractions such as Entity Framework Core (EF Core).

What is EF Core?

Entity Framework Core is a library that provides an object-oriented way to access databases. It acts as an object-relational mapper (ORM), communicating with the database for you and mapping database responses to .NET classes and objects.

Object-Relational Mapper (ORM)
With an ORM, you can manipulate a database with object-oriented concepts such as classes and objects by mapping them to database concepts such as tables and columns.
🗺️ EF Core Mapping

Database Support

The most commonly used family is relational databases, accessed via Structured Query Language (SQL). EF Core can map:

  • Microsoft SQL Server
  • SQLite
  • MySQL
  • PostgreSQL
  • Many other relational databases
  • In-memory database (for testing)

EF Core uses a provider model, so support for other relational databases can be plugged in later as they become available. As of .NET Core 3.0, EF Core also works with nonrelational, NoSQL, or document databases like Cosmos DB.

Why Use an Object-Relational Mapper?

Speed of Development

One of the biggest advantages of an ORM is the speed with which it allows you to develop an application. You can stay in the familiar territory of object-oriented .NET, often without needing to manipulate a database directly or write custom SQL.

Without an ORM (Traditional Approach)
To load product details from a database, you'd have to:
1. Open a connection to the database
2. Write the necessary SQL with correct table and column names
3. Read the data over the connection
4. Create a POCO (Plain Old CLR Object) to hold the data
5. Set properties on the object, converting data to the correct format manually
With EF Core
EF Core handles all the plumbing for you:
• Handles database connections
• Generates the SQL statements
• Maps data back to your POCO objects
• All you provide is a LINQ query describing the data you want

Change Tracking

ORMs like EF Core keep track of which properties have changed on any objects they retrieve from the database. This lets you:

  • Load an object from the database
  • Modify it in .NET code
  • Ask the ORM to update the associated record

The ORM works out which properties have changed and issues update statements for the appropriate columns, saving you a bunch of work.

Drawbacks to Consider

Using an ORM has some tradeoffs:

⚠️ Database Abstraction
The high level of abstraction can lead to problematic query patterns. A classic example is the N+1 problem, where what should be a single database request turns into separate requests for every row in a table.
⚡ Performance
ORMs are abstractions over several concepts, so they inherently do more work than handcrafted data access. However, modern EF Core uses many database tricks and crafts clean SQL queries, often outperforming handcrafted ADO.NET queries!
✅
Best of Both Worlds
Use an ORM for quick development of the bulk of your application, and fall back to lower-level APIs like ADO.NET for those few areas that prove to be bottlenecks. This way, you get good-enough performance with EF Core while optimizing only the areas that need it.

How EF Core Builds the Internal Model

When your application first uses EF Core, it creates an internal representation of the database based on the DbSet<T> properties on your application's DbContext and the entity classes themselves.

ProcessEF Core Model Building
1. EF Core looks for DbSet<T> properties on your DbContext
   Example: DbSet<Recipe> Recipes

2. It scans all properties on known entities for linked types
   Example: Recipe has ICollection<Ingredient>

3. It uses relationships between .NET classes to model 
   relationships between database tables (foreign keys)

Result: Complete internal model of your data structure

For the recipe app, EF Core builds a model of the Recipe class because it's exposed on the AppDbContext as a DbSet<Recipe>. It then loops through all properties of Recipe, looking for types it doesn't know about, and adds them to its internal model.

EF Core Model Discovery
1
EF Core looks at DbSet<Recipe> Recipes on your AppDbContext — this tells it Recipe is an entity to map.
2
It scans all properties on Recipe. It finds ICollection<Ingredient> — a type it doesn't know yet — so it adds Ingredient to the model too.
3
EF Core uses RecipeId on Ingredient and the navigation property on Recipe to model a one-to-many foreign key relationship — one Recipe has many Ingredients.
✓
Result: a complete internal model describing 2 tables (Recipes, Ingredients) and their relationship — ready for migrations and queries.

Building a Data Model

EF Core is flexible in letting you define your entities the way you want to, as POCO (Plain Old CLR Object) classes. EF Core heavily favors a convention over configuration approach.

Entity Classes Example

Here are the Recipe and Ingredient entity classes for the recipe app:

C#Recipe and Ingredient Entities
public class Recipe
{
    public int RecipeId { get; set; }
    public required string Name { get; set; }
    public TimeSpan TimeToCook { get; set; }
    public bool IsDeleted { get; set; }
    public required string Method { get; set; }
    public required ICollection<Ingredient> Ingredients { get; set; }
}

public class Ingredient
{
    public int IngredientId { get; set; }
    public int RecipeId { get; set; }
    public required string Name { get; set; }
    public decimal Quantity { get; set; }
    public required string Unit { get; set; }
}

EF Core Conventions

These classes conform to certain default conventions that EF Core uses:

Primary Key Convention
Properties with an Id suffix (e.g., RecipeId, IngredientId) are identified as the primary key of the table.
Foreign Key Convention
The RecipeId property on Ingredient is interpreted as a foreign key pointing to the Recipe class. Combined with ICollection<Ingredient> on Recipe, this represents a many-to-one relationship.
Primary Key
A value that uniquely identifies the row among all the others in the table. It's often an int or a Guid.
Foreign Key
A foreign key on a table points to the primary key of a different table, forming a link between the two rows.

Code-First Approach

This code-first approach is typical: you define your entities in code, and EF Core creates the database schema from them. If you have an existing database, you can also automatically generate the EF entities and DbContext instead (called "reverse engineering").

🎨
Customization with DataAnnotations
You can use DataAnnotations attributes to decorate your entity classes, controlling things like column naming and string length. EF Core will use these attributes to override the default conventions.

public class AppDbContext : DbContext

        : base(options) { }

The AppDbContext for your app is simple, containing a list of your root entities. You'll use the Recipes property to query the database. Note that you didn't list Ingredient on AppDbContext, but EF Core models it correctly as it's exposed on the Recipe.

Registering a Data Context

As with any other service in ASP.NET Core, you should register your AppDbContext with the dependency injection (DI) container. When registering your context, you also configure the database provider and set the connection string.

C#Program.cs — Registering DbContext
using Microsoft.EntityFrameworkCore;

WebApplicationBuilder builder = WebApplication.CreateBuilder(args);

// ① Load connection string from appsettings.json
var connString = builder.Configuration
    .GetConnectionString("DefaultConnection");

// ② Register AppDbContext with DI (scoped lifetime)
builder.Services.AddDbContext<AppDbContext>(
    // ③ Choose database provider — swap for UseSqlServer, UseNpgsql…
    options => options.UseSqlite(connString));

WebApplication app = builder.Build();
app.Run();
① Connection string from config — loaded from appsettings.json (or env vars / secrets in production). The key "DefaultConnection" must match the name in your config file.
② AddDbContext<T> registers your context as a Scoped service — one instance per HTTP request. You can then inject AppDbContext into services and controllers.
③ UseSqlite() sets the database provider. To switch to SQL Server or PostgreSQL later, just swap this one method — all your entity and query code stays the same.
🔒
Connection String as Secret
The connection string is a typical secret, so loading it from configuration makes sense. At runtime, the correct configuration string for your current environment is used, so you can use different databases when developing locally and in production.
📌 What AddDbContext does under the hood
🔌
Registers as Scoped
One DbContext per HTTP request — avoids concurrency issues and matches request lifetime
⚙️
Configures Provider
The options lambda sets which database engine to use — swap UseSqlite for UseSqlServer to change DB
🔑
Injects Connection
Connection string from config — different values in dev, staging, production without code changes

Now you have a DbContext registered as a scoped service with the DI container (typical for database-related services), and a data model corresponding to your database. The next step is to create the database itself using migrations.

Two Approaches: Code-First vs Database-First
✅ Code-First (Typical)
C# Classes → Migrations → Database
You write entities in C# → EF Core creates the DB schema for you. Best for new projects.
↩️ Database-First (Reverse Engineering)
Database → Scaffold → C# Classes
Existing database? Run dotnet ef dbcontext scaffold to auto-generate entity classes. Best for legacy DBs.

Managing Changes with Migrations

Managing schema changes for databases is notoriously difficult. Your application code is explicitly tied to a particular version of a database, and you need to make sure that the two are always in sync.

Schema
How the data is organized in a database, including the tables, columns, and relationships among them.

The Database Deployment Problem

When you deploy an app:

  • You can delete old code and replace it with new code
  • If you need to roll back, delete the new code and deploy an old version
  • But databases contain data! You can't just blow it away and create a new one

What are Migrations?

EF Core provides its own version of schema management called migrations. Migrations provide a way to manage changes to a database schema when your EF Core data model changes.

Migration
A C# code file in your application that defines how the data model changed—which columns were added, new entities, and so on. Migrations provide a record over time of how your database schema evolved as part of your application, so the schema is always in sync with your app's data model.

What You Can Do with Migrations

✅
Create a new database
Use command-line tools to create a database from migrations
✅
Update an existing database
Apply new migrations to an existing database
⏮️
Roll back a migration
Update a database to a previous schema
⚠️
Data Loss Warning
Applying migrations modifies the database, so you must always be aware of data loss. If you remove a table from the database using a migration and then roll back the migration, the table will be re-created, but the data it previously contained will be gone forever!

Applying the Migration

Adding a migration doesn't update anything in the database itself. You need to apply the migration to your database. EF Core provides four different ways to apply migrations:

1 .NET CLI Tool

2 Visual Studio Package Manager Console (PMC)
PowerShellPMC Cmdlet
Update-Database

3 In Code — Program.cs Startup

{
    var dbContext = scope.ServiceProvider
        .GetRequiredService<AppDbContext>();

Useful for applications that need to apply migrations automatically on startup. Great for desktop apps or containers, but be careful with multiple web server instances.
4 Migration Bundle (EF Core 6+)

💡
Which Method Should You Use?
Development: Use the .NET CLI tool or PMC cmdlets.
CI/CD Pipelines: Use the .NET CLI tool or migration bundles.
Production (no SDK): Use migration bundles — they're self-contained.
Desktop/Container Apps: Use context.Database.Migrate() in startup code.

Files Created

This command creates three files in the Migrations folder in your project:

Timestamp_MigrationName.cs
Migration file describing the actions to take on the database (creating tables, adding columns, etc.). Commands are database-provider-specific.
Timestamp_MigrationName.Designer.cs
Describes EF Core's internal model of your data model at the point in time when the migration was generated.
AppDbContextModelSnapshot.cs
Describes EF Core's current internal model. Updated when you add another migration. Should always match the current (latest) migration.
📁 Files Created in Your Project
MyApp/
├── Program.cs
└── Migrations/
├── 20240101_InitialCreate.cs Up() and Down() methods — the actual schema changes
├── 20240101_InitialCreate.Designer.cs Snapshot of EF Core's model at creation time (auto-generated)
└── AppDbContextModelSnapshot.cs Current model state — always updated to match latest migration
💡
Up() and Down()
Every migration has two methods: Up() applies the change (e.g., creates a table), and Down() reverses it (e.g., drops the table). This is what makes rollbacks possible.

Applying the Migration

Adding a migration doesn't update anything in the database itself. You need to apply the migration:

BashApplying Migration
dotnet ef database update

What Happens During Update

1
Builds your application
2
Loads the services configured in Program.cs, including AppDbContext
3
Checks whether the database exists; if not, creates it
4
Updates the database by applying any unapplied migrations

When you apply the migrations, EF Core creates the necessary tables in the database and adds the appropriate columns and keys. It also creates the __EFMigrationsHistory table, which stores the names of migrations that have been applied to the database.

🗃️ The __EFMigrationsHistory Table

EF Core automatically creates and manages this special table. Each time you run database update, it checks this table to know which migrations are already applied and skips them.

MigrationId ProductVersion
20240101120000_InitialCreate 8.0.0
20240115093000_ExtraRecipeFields 8.0.0
When you run dotnet ef database update, EF Core compares your migration files against this table and only applies new ones — so it's safe to run repeatedly.

Creating a Record

All interactions with EF Core and the database start with an instance of AppDbContext, which is typically DI-injected via the constructor. Creating a new entity requires three steps:

1
Create the entities
Create the Recipe and Ingredient entities with their properties
2
Add to tracked entities
Use _context.Add(entity) to tell EF Core to track the new entities
3
Save to database
Call _context.SaveChangesAsync() to execute SQL INSERT statements

C#RecipeService.CreateRecipe()
readonly AppDbContext _context;  // ① injected by DI
    
readonly  ILogger  _logger;
    public RecipeService(AppDbContext context, ILoggerFactory factory)
    {
        _context = context;
        _logger = factory.CreateLogger();
    }

public async Task<int> CreateRecipe(CreateRecipeCommand cmd)
{
    // ② Build entity from the command/DTO — NOT from raw input (avoids overposting)
    var recipe = new Recipe
    {
        Name = cmd.Name,
        TimeToCook = new TimeSpan(cmd.TimeToCookHrs, cmd.TimeToCookMins, 0),
        Method = cmd.Method,
        IsVegetarian = cmd.IsVegetarian,
        IsVegan = cmd.IsVegan,
        // ③ Map child collection inline — EF Core handles INSERT for both tables
        Ingredients = cmd.Ingredients.Select(i => new Ingredient
        {
            Name = i.Name,
            Quantity = i.Quantity,
            Unit = i.Unit,
        }).ToList()
    };

    _context.Add(recipe);              // ④ tells EF Core to track this as "Added"
    await _context.SaveChangesAsync();  // ⑤ executes INSERT for Recipe + all Ingredients

    return recipe.RecipeId;             // ⑥ ID was 0 before save, now populated by DB
}
①–② The service receives a CreateRecipeCommand DTO, not a raw Recipe entity. This prevents overposting — a user can't set IsDeleted=true or other fields they shouldn't control.
③ Setting Ingredients on the Recipe is enough — EF Core detects the child collection and generates INSERT statements for both the Recipes and Ingredients tables automatically.
④–⑥ Add() marks the entity as "Added" in the change tracker. SaveChangesAsync() executes the INSERT and the database returns the generated ID, which EF Core writes back into recipe.RecipeId — no extra SELECT needed.
🔑
EF Core Auto-Populates IDs After Save
Before SaveChangesAsync(), recipe.RecipeId is 0. After the call, the database assigns a real ID (e.g., 42), and EF Core writes it back into your in-memory object automatically. That's why you can return recipe.RecipeId after saving — without a separate SELECT query.
🛡️
Avoid Overposting
Many simple sample applications allow you to bind directly to the Recipe entity in your endpoint. This approach exposes a security vulnerability known as overposting. Always use command/DTO objects for binding and map to your entities manually or using a library like AutoMapper.

In professional applications, we don't expose our Database Entities (like Ingredient) directly to the API users. Instead, we use a Command or DTO (Data Transfer Object) to filter and validate input.

C#Models/CreateIngredientCommand.cs

using System.ComponentModel.DataAnnotations;
using RecipeApp.Models;

public class CreateIngredientCommand
{
    [Required, StringLength(100)]
    public required string Name { get; set; }

    [Range(0, int.MaxValue)]
    public decimal Quantity { get; set; }

    [Required, StringLength(20)]
    public required string Unit { get; set; }

    // Logic to convert DTO back to a Database Entity
    public Ingredient ToIngredient()
    {
        return new Ingredient
        {
            Name = this.Name,
            Quantity = this.Quantity,
            Unit = this.Unit
            // ID is handled by Database (Auto-increment)
        };
    }
}

If a problem occurs when EF Core tries to interact with your database, this code throws an exception. It's important to handle these exceptions in your application so you don't present an ugly error message to users.

Assuming all goes well, EF Core updates all the autogenerated IDs of your entities (RecipeId on Recipe, and both RecipeId and IngredientId on Ingredient).

Loading a List of Records

Loading data is simple in EF Core, relying heavily on LINQ methods to control the fields you need. EF Core uses a fluent chain of LINQ commands to define the query.

Three Parts of an EF Core Query

1
DbSet Property Access
_context.Recipes
2
LINQ Commands to Modify Data
.Where(), .Select(), .OrderBy(), etc.
3
Execute Query Command
.ToListAsync(), .SingleAsync(), .FirstAsync(), etc.
🔗 LINQ Query Pipeline → SQL
_context.Recipes → Start: the full Recipes table
.Where(r => !r.IsDeleted) → Filter: only non-deleted recipes
.Select(r => new RecipeSummaryViewModel…) → Project: pick only needed columns (efficient!)
.ToListAsync() → Execute: sends SQL to DB, materialises results
-- Generated SQL (roughly):
SELECT RecipeId, Name, TimeToCook
FROM Recipes
WHERE IsDeleted = 0
C#RecipeService.GetRecipes()
public async Task<ICollection<RecipeSummaryViewModel>> GetRecipes()
{
    return await _context.Recipes           // ① start: full Recipes table
        .Where(r => !r.IsDeleted)           // ② filter: skip soft-deleted rows
        .Select(r => new RecipeSummaryViewModel
        {
            Id = r.RecipeId,
            Name = r.Name,
            TimeToCook = $"{r.TimeToCook.TotalMinutes}mins"  // ③ project: only needed columns + C# formatting
        })
        .ToListAsync();                     // ④ execute: send SQL → get results
}
① _context.Recipes — accesses the DbSet<Recipe>. No SQL runs yet; LINQ builds up a query description in memory.
② .Where(r => !r.IsDeleted) — translated to WHERE IsDeleted = 0 in SQL. Filtering happens on the database server, not in C# memory.
③ .Select() maps each row to a RecipeSummaryViewModel. EF Core generates a SELECT with only the columns referenced here — not SELECT *. The string interpolation ($"...") runs client-side.
④ .ToListAsync() is the trigger — this is when EF Core finalises the SQL, sends it to the database, awaits the response, and materialises each row into a view model object.
🔄
Client-Side Evaluation
Notice the string interpolation in the Select method. EF Core can't convert some expressions to SQL, so it evaluates them on the client side. It finds the fields it needs from the database, selects them, and then runs the expression in C#. This powerful feature combines database performance with C# functionality, but use it carefully to avoid performance issues.
⚡ EF Core Execution Split
🗄️ Executed in SQL (server-side)
.Where(), .OrderBy(), .Skip(), .Take() — EF Core translates these to SQL before fetching data. Only matching rows travel over the network.
🖥️ Executed in C# (client-side)
String interpolation and custom C# logic inside .Select() that can't be translated to SQL — EF Core fetches required columns and runs these expressions in memory.

Loading a Single Record

Loading a single record follows the same pattern as loading a list, but typically uses a Where clause that restricts the data to a single entity and an execute command like SingleOrDefaultAsync().

🔍 Ways to Load a Single Record
Method Finds by Checks cache? Best for
FindAsync(id) Primary key only ✅ Yes — may skip DB Loading by PK, updates
SingleOrDefaultAsync() Any .Where() clause ❌ Always hits DB Complex filters, projections
FirstOrDefaultAsync() Any .Where() clause ❌ Always hits DB When >1 result is OK
💡 SingleOrDefaultAsync throws if more than one match is found — useful as a safety net. FindAsync is faster when you already have a tracked entity in the same request.
C#RecipeService.GetRecipeDetail()
public async Task<RecipeDetailViewModel> GetRecipeDetail(int id)
{
    return await _context.Recipes
        .Where(x => x.RecipeId == id)
        .Select(x => new RecipeDetailViewModel
        {
            Id = x.RecipeId,
            Name = x.Name,
            Method = x.Method,
            Ingredients = x.Ingredients
                .Select(item => new RecipeDetailViewModel.Item
                {
                    Name = item.Name,
                    Quantity = $"{item.Quantity} {item.Unit}"
                })
        })
        .SingleOrDefaultAsync();
}
⚠️
SingleOrDefaultAsync Behavior
SingleOrDefaultAsync() throws an exception if the previous Where clause returns more than one record. It returns null if no records are found.

Notice that you can map the related Ingredients for a Recipe as though you're working with the objects directly in memory. One advantage of using an ORM is that you can easily map child objects and let EF Core decide how best to build the underlying queries to fetch the data.

📊
Viewing Generated SQL
EF Core logs all the SQL statements it runs as LogLevel.Information events by default, so you can easily see what queries are running against the database. This is invaluable for understanding and optimizing your database access patterns.

C#RecipeService.UpdateRecipe()
public async Task UpdateRecipe(UpdateRecipeCommand cmd)
{
    // ① Load from DB — EF Core starts tracking this entity and takes a snapshot
    var recipe = await _context.Recipes.FindAsync(cmd.Id);

    // ② Guard: null check before modifying (404 if not found)
    if (recipe is null) {
        throw new Exception("Unable to find the recipe");
    }

    UpdateRecipe(recipe, cmd);           // ③ mutate properties via helper method
    await _context.SaveChangesAsync();  // ④ EF Core diffs snapshot vs current, runs UPDATE
}

// Private helper keeps the mapping logic separate and easy to test
static void UpdateRecipe(Recipe recipe, UpdateRecipeCommand cmd)
{
    recipe.Name = cmd.Name;
    recipe.TimeToCook = new TimeSpan(cmd.TimeToCookHrs, cmd.TimeToCookMins, 0);
    recipe.Method = cmd.Method;
    recipe.IsVegetarian = cmd.IsVegetarian;
    recipe.IsVegan = cmd.IsVegan;
}
① FindAsync(id) checks EF Core's in-memory cache first — if the same entity was already loaded in this request, no database round-trip occurs. Otherwise, it runs SELECT … WHERE RecipeId = @id.
③–④ Simply assigning new values to the loaded entity's properties is enough. EF Core compares the new values against its snapshot and generates a targeted UPDATE that only touches the columns that actually changed.
🔍
The FindAsync Method
FindAsync(id) is a simple helper method exposed by DbSet that loads an entity by its ID. It first checks to see whether the entity is already being tracked in EF Core's DbContext. If so, the entity is returned immediately without calling the database, which can be faster.

IsVegetarian: true
vs
Current Values (after edit)
Name: "Pasta"

-- EF Core generates only what changed:

How Change Tracking Works

EF Core internally tracks the state of any entities it loads from the database and creates a snapshot of all the entity's property values. When you call SaveChanges(), EF Core:

  • Compares the state of any tracked entities with the tracking snapshot
  • Identifies which properties have changed
  • Includes only changed properties in the UPDATE statement
  • Ignores unchanged properties
🔍
Viewing Change Tracker
You can view which details the DbContext is tracking by accessing DbContext.ChangeTracker.DebugView. This is useful for debugging and understanding what EF Core is doing behind the scenes.

Deleting Records

When it comes to deleting data, you have two options: hard delete (permanently remove from database) or soft delete (mark as deleted but keep in database). In most cases, soft delete is preferred.

Why Soft Delete?

🔄
Reversible
Can undo accidental deletions
🔗
No referential integrity issues
Avoids foreign key constraint problems
📊
Audit trail
Maintains historical data for analysis

Implementing Soft Delete

A common approach is to include an IsDeleted flag on your entity:

🗑️ Soft Delete vs Hard Delete
✅ Soft Delete (recommended)
-- Not an actual DELETE
UPDATE Recipes
SET IsDeleted = 1
WHERE RecipeId = 42
✔ Data is recoverable
✔ No FK violations
✔ Audit trail preserved
⚠ Must filter IsDeleted everywhere
⚠️ Hard Delete (permanent)
-- Permanent removal
DELETE FROM Recipes
WHERE RecipeId = 42
✔ Clean, simple
✔ Required for GDPR compliance
✗ Unrecoverable
✗ May break FK references
C#RecipeService.DeleteRecipe() — Soft Delete
public async Task DeleteRecipe(int recipeId)
{
    var recipe = await _context.Recipes.FindAsync(recipeId);

    if (recipe is null) {
        throw new Exception("Unable to find the recipe");
    }

    recipe.IsDeleted = true;             // ① just flip the flag — no context.Remove()
    await _context.SaveChangesAsync();  // ② EF Core runs UPDATE, not DELETE
}

// Every query must exclude deleted records:
_context.Recipes.Where(r => !r.IsDeleted)…   // ③ or use a global query filter

With this approach, you need to make sure that you have Where() clauses in all your other methods to ensure you can't return a deleted Recipe:

🔧
Global Query Filters — The Clean Solution
Instead of adding .Where(r => !r.IsDeleted) to every query, EF Core's global query filters let you define this once on the model. EF Core automatically appends the filter to every query against that entity — you can't accidentally forget it. Also useful for multi-tenant apps.

Hard Delete (When Necessary)

For cases where you truly need to delete data (such as complying with data protection regulations), EF Core provides the Remove method:

C#Hard Delete — _context.Remove()
var recipe = await _context.Recipes.FindAsync(recipeId);
if (recipe is not null)
{
    _context.Remove(recipe);            // marks entity as "Deleted" in change tracker
    await _context.SaveChangesAsync();  // runs: DELETE FROM Recipes WHERE RecipeId = @id
}

Using EF Core in Production Applications

Before using EF Core in a production application, you need to consider several important topics beyond the basics covered in this chapter.

Data Model Considerations

Scaffolding of columns
EF Core uses conservative values for string columns (large or unlimited length). In practice, you may want to restrict these to sensible values.
Validation
You can decorate your entities with DataAnnotations validation attributes, but EF Core won't validate values automatically before saving (unlike EF 6.x).

Runtime Considerations

Handling concurrency
EF Core provides ways to handle when multiple users attempt to update an entity at the same time. One solution is to use Timestamp columns.
Handling errors
Databases and networks are inherently flaky. EF Core includes features to maintain connection resiliency by retrying on network failures.
Synchronous vs. asynchronous
EF Core provides both sync and async commands. Often, async is better for web apps, but this has nuances that make it impossible to recommend one approach universally.

Deployment Considerations

Automatic migrations
For automated deployment pipelines, you need a way to apply migrations automatically. Options include scripting the .NET tool, applying migrations in startup code, using EF Core bundles, or custom tools.
Multiple web hosts
If you have multiple web servers pointing to the same database, ensure that only one app can migrate the database at a time.
Backward-compatible schema changes
When deploying with multiple web hosts, you'll often have apps accessing a database with a newer schema. Endeavor to make schema changes backward-compatible wherever possible.
Storing migrations in different assembly
In larger apps, data access is often in a different project from the web app. You must use slightly different commands when using .NET CLI or PowerShell cmdlets.
Seeding data
When you first create a database, you often want initial seed data (like a default user). EF Core requires you to seed your database explicitly yourself.

🎯 Chapter Summary
EF Core is an ORM that maps .NET classes to database tables, reducing SQL and database knowledge needed to be productive.

Database providers let you change the underlying database without changing your code. Support for SQL Server, SQLite, PostgreSQL, MySQL, and more.

Code-first approach is typical: define entities in code, and EF Core creates the database schema via migrations.

Migrations track changes to your data model and update the database schema accordingly. Use dotnet ef migrations add to create and dotnet ef database update to apply.

CRUD operations use LINQ queries: Add entities with Add(), query with Where() and Select(), update by modifying tracked entities, and delete with Remove() or soft delete.

Production considerations include validation, concurrency, error handling, deployment strategies, and migration management for multiple servers.
