Chapter 16

Binding and Validating Requests with Razor Pages

How ASP.NET Core extracts data from HTTP requests, maps it to C# objects, and validates it — all before your page handler ever executes.

1

Understanding the Models in Razor Pages and MVC

The classic MVC design pattern defines three independent components: Model, View, and Controller. ASP.NET Core Razor Pages refines this further by splitting the "Model" into multiple focused models, each with a single responsibility.

📦
Model
The data to display and methods for updating it
🖼️
View
Displays a representation of the data from the model
🎮
Controller
Calls methods on the model and selects a view

ASP.NET Core Razor Pages goes further — it uses multiple specialized models built from plain C# objects (POCOs), applying the Single Responsibility Principle. Each model handles one specific aspect of the overall request.

ArchitectureASP.NET Core models — one responsibility each
① Binding Model
Data extracted from the HTTP request (route, query, form, body)
↓ feeds into
② PageModel (Controller + ViewModel)
Exposes page handlers + all data the view needs
↓ calls
③ Application Model
Services, business logic, databases — independent of ASP.NET Core
↓ result displayed via
④ Razor View (HTML generation)
Renders HTML using data already prepared by the PageModel
2

The Binding Model

The binding model is all the information provided by the user when making a request, plus any additional contextual data. It is the bridge between the raw HTTP request and your strongly-typed C# objects.

It includes data from:

Route Parsed URL segments — e.g., /product/42 → id = 42
Query Values at the end of the URL — e.g., ?page=2&sort=desc
Form / Body Data POSTed in the request body — form fields or JSON payload

Binding models in Razor Pages are typically defined by creating a public property on the page's PageModel decorated with [BindProperty], or passed directly as handler parameters.

C#Defining a binding model with [BindProperty]
public class CheckoutModel : PageModel { // ❶ This property is auto-populated from the request before OnPost() runs [BindProperty] public CheckoutInput Input { get; set; } public IActionResult OnPost() { // Input is already populated — no manual parsing needed var name = Input.FullName; return Page(); } } // Handler parameter binding — alternative approach public IActionResult OnPost(ProductModel product) { // product is automatically bound from the request }
⚠️
Silent Failure — Setter Required
Properties decorated with [BindProperty] must have a public setter. If the setter is missing or private, binding will silently fail — no exception, the value will just be null or default.

3

PageModel and Application Model

📋 PageModel — Two Jobs
1. Acts as Controller — exposes page handler methods (OnGet, OnPost, etc.) that respond to HTTP verbs.

2. Acts as ViewModel — all data required for the Razor view to render is exposed as public properties on the PageModel. The view accesses it via Model.PropertyName.
🧠 Application Model — Business Logic
Not really a single ASP.NET Core "model" — it's a collection of services and classes that implement your business rules.

Anything needed to perform a business action: database queries, email services, payment processing, external APIs, etc.

Crucially independent of ASP.NET Core itself.

From Request to Binding Model: Model Binding

The process of extracting values from the HTTP request and creating C# objects from them is called model binding. ASP.NET Core does this automatically for you — you never manually read Request.Form["name"].

📋
Note — How it works
ASP.NET Core automatically populates your binding models using properties of the request, such as the request URL, any headers sent in the HTTP request, any data explicitly POSTed in the request body, and so on.

5

Default Binding Sources (Order of Precedence)

By default, ASP.NET Core uses three binding sources when creating binding models in Razor Pages. It takes the first value it finds (if any) that matches the name of the binding model property:

PrioritySourceDescriptionExample
1st Form values Sent in the body of a POST request when a form is submitted name=Alice&age=25 in body
2nd Route values Obtained from URL segments or default route values /product/42
3rd Query string Passed at the end of the URL — not used during routing ?page=2&sort=asc
💡
Key Note: Model binder is NOT case-sensitive
The default model binder isn't case-sensitive, so a binding value of QTY=50 will happily bind to the qty parameter. You don't need to match casing exactly.
⚠️
Razor Pages vs Minimal APIs — Important Difference
In Razor Pages, different properties of a complex model can be bound to different sources — one property from route, another from form. This is called "partial" binding.

In Minimal APIs, the whole object must be bound from a single source — partial binding is not possible. Razor Pages also bind to form bodies by default, while Minimal APIs cannot.

These differences are partly for historical reasons and partly because Minimal APIs opts for performance over convenience.
6

Binding Simple Types

Simple types (int, string, bool, decimal, etc.) are bound directly from the matching source by name. You don't write any extraction code — just declare a parameter or property with the right name.

RazorRoute parameter → handler parameter — automatic binding
@* @page "{number}" — declares "number" as a route segment *@ @page "{number}" @model ProductModel // In the PageModel: public IActionResult OnGet(int number) { // GET /product/42 → number = 42 // No code needed to extract it from the URL! var product = _db.Find(number); return Page(); } // Multiple route parameters: @page "/{currencyIn}/{currencyOut}" public IActionResult OnGet(string currencyIn, string currencyOut) { // GET /USD/GBP → currencyIn="USD", currencyOut="GBP" }
📋
Note — The key insight
The key thing to appreciate is that you didn't have to write any extra code to extract the number from the URL when the method executed. All you needed to do was create a method parameter (or public property) with the right name and let model binding do its magic.

If none of the binding sources contains the required value, the binding model is set to a default:

TypeDefault when not foundExample
Value types (int, bool)default(T)int → 0, bool → false
Reference types (custom classes)New instance via default constructornew ProductModel()
Nullable types (int?, bool?)nullint? page = null
stringnullstring name = null

The model binder converts string values from binding sources to the target .NET type. It handles any primitive type (int, float, decimal, bool, string), any type with a TryParse method, and anything with a TypeConverter.

Binding Complex Types

When a handler has many parameters, it becomes messy fast. The solution is to bind to a complex type — a class that groups multiple properties. The binder creates an instance of the class and populates each property individually.

❌ Too many parameters — hard to maintain
public IActionResult OnPost( string firstName, string lastName, string email, int age) { // What if we add 5 more fields tomorrow? }
✅ Complex type — clean and extensible
public IActionResult OnPost(UserBindingModel user) { // user.FirstName, user.LastName, etc. // Adding new fields = just add to the class } public class UserBindingModel { public string FirstName { get; set; } public string LastName { get; set; } public string Email { get; set; } public int Age { get; set; } }
📋
Note — How the binder handles complex types
Rather than look for a parameter matching the whole type name, the model binder creates a new instance of the model using new UserBindingModel(). For a class to be model-bound, it must have a default (parameterless) public constructor. You can bind only properties that are public and settable.

You can also bind complex hierarchical models whose properties are themselves complex models. As long as each property exposes a type that can be model-bound, the binder can traverse it with ease.

Binding Collections and Dictionaries

ASP.NET Core can bind to List<T>, arrays, and dictionaries. The key values can come from route or query values, but it's far more common to POST them in a form.

C#Binding a collection of strings
public IActionResult OnPost(List<string> currencies) { // Can POST in three different formats: }
FormatPOST body exampleNotes
name[index] currencies[0]=GBP&currencies[1]=USD Explicit name + index
[index] [0]=GBP&[1]=USD Omit name when binding a single list
name repeated currencies=GBP&currencies=USD Omit index — same key multiple times
💡
Collections of complex types
You can also bind collections of complex types, such as List<UserBindingModel>. The binder will create a new instance of the complex type for each element in the collection.

Binding File Uploads with IFormFile

Razor Pages supports users uploading files via IFormFile and IFormFileCollection. You use these just like any other binding model — as a [BindProperty] or a handler parameter.

C#IFormFile — single file upload
public class UploadModel : PageModel { [BindProperty] public IFormFile Upload { get; set; } // single file [BindProperty] public IFormFileCollection Uploads { get; set; } // multiple files public async Task<IActionResult> OnPostAsync() { if (Upload != null && Upload.Length > 0) { var path = Path.Combine("uploads", Upload.FileName); using var stream = System.IO.File.Create(path); await Upload.CopyToAsync(stream); } return Page(); } } // In the Razor view, the form must use enctype="multipart/form-data": <form method="post" enctype="multipart/form-data"> <input type="file" asp-for="Upload" /> <button type="submit">Upload</button> </form>

Choosing a Binding Source with Attributes

You can override the default binding source for each property or parameter using attributes:

AttributeBinds fromNotes
[FromHeader]HTTP request headerse.g., Authorization, X-Api-Key
[FromQuery]Query string?page=2
[FromRoute]Route parameters/product/{id}
[FromForm]Form data in request bodyNot available in Minimal APIs
[FromBody]Request body (JSON)Only one parameter can use this — body can only be read once
[FromServices]Dependency injection containerInjects a service; usually not needed in .NET 7+ (auto-detected)
[BindNever]—Prevents binding entirely — useful to prevent mass assignment
[BindRequired]Same as normalAdds a validation error if the value is missing or empty
⚠️
[FromBody] and [FromForm] are mutually exclusive
Both attributes read from the request body. Since an HTTP request body can only be safely read once, only one parameter may use [FromBody], and [FromBody] and [FromForm] cannot be used together on the same request.
📋
Note — JSON from body in Razor Pages
Developers coming from .NET Framework and the legacy ASP.NET should note that the [FromBody] attribute is explicitly required when binding to JSON requests in Razor Pages. It is not automatic. Automatic binding of multiple formats from the request body is one of the features specific to Razor Pages and MVC controllers — this is missing from Minimal APIs.
C#Overriding default binding sources
public IActionResult OnPost( [FromRoute] int id, // from URL: /product/42 [FromQuery] string sort, // from ?sort=asc [FromHeader(Name = "X-Api-Key")] string apiKey, // from header [FromBody] OrderModel order) // from JSON request body { // Each parameter comes from its declared source }

Validating Binding Models — DataAnnotations

In Razor Pages, validation is built in. Validation occurs automatically after model binding but before the page handler executes. Minimal APIs have no direct validation support — you have to add it yourself with filters and packages.

🔒
Always validate on the server
It's always necessary to validate data as it arrives at your web application using server-side validation. Client-side validation can be bypassed with browser developer tools or direct HTTP requests. Never trust user input without server-side checks.
C#DataAnnotations validation attributes on a binding model
public class UserBindingModel { // ❶ Values marked Required must be provided [Required] [Display(Name = "Your name")] // ❸ Customizes the property display name public string FirstName { get; set; } // ❹ Validates that Email is a valid email address [Required] [EmailAddress] public string Email { get; set; } // ❷ StringLength sets the maximum (and optionally minimum) length [StringLength(200, MinimumLength = 10)] public string Bio { get; set; } // ❺ Validates that PhoneNumber has a valid telephone format [Phone] public string PhoneNumber { get; set; } [Range(18, 120)] public int Age { get; set; } } // For cross-property validation, implement IValidatableObject: public class DateRangeModel : IValidatableObject { public DateTime StartDate { get; set; } public DateTime EndDate { get; set; } public IEnumerable<ValidationResult> Validate(ValidationContext ctx) { if (EndDate <= StartDate) yield return new ValidationResult( "End date must be after start date", new[] { nameof(EndDate) }); } }
12

Validating on the Server — ModelState

Validation occurs before the page handler executes, but the handler always executes regardless — whether validation succeeded or failed. It is the handler's responsibility to check the result of the validation via ModelState.IsValid.

C#Checking ModelState in the page handler
public IActionResult OnPost() { // ❶ Always check ModelState — handler runs even if invalid! if (!ModelState.IsValid) { // Validation failed — re-render the form with error messages return Page(); } // ❷ Add custom business rule validation errors if (Input.Email.EndsWith("@spam.com")) { ModelState.AddModelError(nameof(Input.Email), "Spam email addresses are not allowed."); return Page(); } // ❸ All valid — proceed with business logic _userService.Register(Input); return RedirectToPage("Success"); }
📋
Note — Customizing error messages
The error messages displayed on the form are the default values for each validation attribute. You can customize them by setting the ErrorMessage property on any validation attribute. For example: [Required(ErrorMessage="This field is required")].

POST-REDIRECT-GET Pattern

The POST-REDIRECT-GET (PRG) pattern is a best practice for handling form submissions. Without it, refreshing after a POST re-submits the form — which can create duplicate records.

FlowPOST-REDIRECT-GET — prevents duplicate form submissions
1 User submits form → POST /checkout
2 Server processes the form, then returns 302 Redirect → /order-confirmed
3 Browser follows redirect → GET /order-confirmed
✓ User refreshes → GET /order-confirmed only — no duplicate POST!
C#PRG in a Razor Page handler
public IActionResult OnPost() { if (!ModelState.IsValid) return Page(); // Re-render form with errors _orderService.PlaceOrder(Input); // PRG: redirect after successful POST return RedirectToPage("OrderConfirmed"); // Browser gets 302, follows it with GET /OrderConfirmed // Now refreshing is safe — it just re-GETs the confirmation page }

Organizing Your Binding Models — Best Practices

The model binding framework gives you many equivalent approaches. The book recommends a set of consistent patterns so your Razor Pages are easy to understand at a glance.

✅ Use a Single Nested InputModel Class

Instead of decorating multiple individual properties with [BindProperty], define a single nested class called InputModel inside your Razor Page and bind just that one property. This keeps everything together and makes all your pages consistent.

❌ Multiple [BindProperty] properties — inconsistent
public class IndexModel : PageModel { [BindProperty] public string Category { get; set; } [BindProperty(SupportsGet = true)] public string Username { get; set; } // Easy to forget [BindProperty] on a new property }
✅ Single InputModel — clean and consistent
public class IndexModel : PageModel { [BindProperty] public InputModel Input { get; set; } public class InputModel { public string Category { get; set; } public string Username { get; set; } } }

Full Recommended Pattern

Here's the complete recommended structure for a typical "edit form" Razor Page:

C#Recommended Razor Page structure (edit form)
public class EditProductModel : PageModel { private readonly ProductService _productService; // ❶ Injected service public EditProductModel(ProductService productService) => _productService = productService; [BindProperty] // ❷ Single [BindProperty] for all input data public InputModel Input { get; set; } public IActionResult OnGet(int id) // ❸ Route param as handler parameter { var product = _productService.GetProduct(id); Input = new InputModel { Name = product.ProductName, Price = product.SellPrice, }; return Page(); } public IActionResult OnPost(int id) { if (!ModelState.IsValid) // ❹ Always validate first return Page(); _productService.UpdateProduct(id, Input.Name, Input.Price); return RedirectToPage("Index"); // ❺ POST-REDIRECT-GET } public class InputModel // ❻ Nested class, always called InputModel { [Required] public string Name { get; set; } [Range(0, int.MaxValue)] public decimal Price { get; set; } } }
The Five Recommended Patterns
❶ Bind only a single property decorated with [BindProperty]. When more than one value needs to be bound, put them in a nested InputModel class.
❷ Define your binding model as a nested class called InputModel inside the Razor Page. Binding models are highly specific to one page, so keeping them together is cleaner.
❸ Don't use [BindProperties] (the plural attribute). It binds every property on the PageModel, which leaves you open to overposting attacks. Stick to a single [BindProperty].
❹ Accept simple route parameters as handler method parameters (e.g., OnGet(int id)). This avoids the clunky SupportsGet = true syntax for GET requests.
❺ Always validate before using data. Check ModelState.IsValid at the start of every handler that processes user input.
📝
class vs record for Binding Models
The book primarily uses class instead of record for binding models. While record is supported, the terse positional syntax it offers is largely lost when you add validation attributes like [Required] to properties — the syntax becomes as verbose as a regular class anyway.

Client-Side Validation

Client-side validation improves the user experience by giving immediate feedback in the browser before the form is submitted. It does not replace server-side validation — it complements it.

You can add client-side validation in two ways:

🌐 HTML5 built-in
HTML5 has built-in validation behaviors that many browsers enforce automatically — required, type="email", min, max, etc. Tag Helpers generate these attributes automatically from your DataAnnotations.
📜 jQuery Validation (most common)
The jquery.validate and jquery.validate.unobtrusive libraries read data-val-* attributes generated by Tag Helpers and validate the form in JavaScript before submission.
💡
How it ties together
When you use DataAnnotations ([Required], [EmailAddress], etc.) on your view model and Tag Helpers in your Razor view, the framework automatically generates data-val-* attributes on HTML inputs. The jQuery validation libraries then pick up these attributes and enforce the same rules in the browser — no extra code needed.
